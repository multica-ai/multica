# Issue Energy Loop

## Summary

Issue Energy Loop 是把 issue 管理闭环和个人精力管理打通的一组产品能力。用户可以从 issue 捕获和整理开始，把重要工作放入 Plan，在执行时进入专注/计时，并在复盘中把实际进展、阻塞、低精力和恢复需求反馈到下一轮计划。

## Problem

当前产品已经有 issue、计划、日复盘、focus/flowtime、time tracking 和精力记录，但用户体验上仍容易断开：

- issue 可以被创建和更新，但不一定自然进入“今天/明天要做什么”的计划。
- 计划可以生成 Markdown，但用户不一定能把计划项推进到实际执行。
- Focus 可以记录低精力、放弃、跳过休息等信号，但这些信号对 issue 优先级、明日负载和恢复安排的影响还不够明确。
- 用户完成一天后，需要知道哪些 issue 推进了、哪些没有推进、为什么没有推进，以及明天应该如何调整。

## Goals / Non-goals

1. V1 的目标是形成个人执行闭环：issue 捕获 → 整理 → 计划 → 执行 → 复盘 → 下一轮计划。
2. V1 的精力管理是 issue 执行的约束和反馈信号，不是独立健康管理系统。
3. V1 优先服务个人工作管理；团队报表、团队负载均衡和管理者视图后置。
4. V1 不引入 Knowledge 功能；Knowledge 已挂起，后续单独恢复。
5. V1 不做医学、诊断、健康建议或医疗提醒。
6. V1 不要求复杂预测模型；系统可以基于用户主动记录和当天行为给出轻量建议。
7. V1 不要求自动替用户改 issue 状态或计划，除非用户明确触发确认。
8. V1 不要求实时协作编辑计划；同一用户多窗口冲突后续单独设计。
9. V1 不引入 Sprint/Cycle 作为独立概念；所有已决定要推进的工作统一进入 Plan。
10. V1 的 Plan 可以用 Today、Tomorrow、Later 等时间范围表达，但用户不需要理解多套计划模型。

## Behavior

1. 用户可以把 Multica 当作一个 issue 闭环系统使用，而不只是 issue 列表。

2. 系统中的核心闭环是：

   ```text
   Capture issue
        ↓
   Clarify / triage
        ↓
   Plan
        ↓
   Execute with Focus / timer
        ↓
   Record outcome and energy signals
        ↓
   Daily review
        ↓
   Adjust next plan
   ```

3. 用户可以从任何现有 issue 进入闭环，而不需要重新创建任务。

4. 用户可以创建新 issue，并在创建后继续补充标题、描述、优先级、状态、负责人、日期、标签和关联上下文。

5. 新创建的 issue 默认不要求立即进入计划；用户可以稍后再整理。

6. 用户可以区分“还没整理的 issue”和“已经准备执行的 issue”。

7. 用户可以通过状态、优先级、日期、负责人、标签和搜索找到需要整理或执行的 issue。

8. 用户可以在 issue 详情中看到当前 issue 的状态、优先级、负责人、日期、标签、子 issue、依赖、评论、活动历史和相关执行记录。

9. 用户修改 issue 状态后，issue 仍是同一条 issue；列表、详情、看板、计划和复盘中都应显示最新状态。

10. 用户修改 issue 标题后，计划、执行和复盘中应尽可能显示最新标题；若某些历史记录使用旧标题快照，界面应避免让用户误以为生成了另一条 issue。

11. 用户可以把 issue 放入个人 Plan。

12. Plan 的核心不是只生成一段 Markdown，而是让用户明确“接下来要推进哪些 issue”。

13. Plan 中的每个计划项可以关联一条 issue。

14. 计划项可以没有关联 issue，但 V1 应鼓励用户优先使用 issue 作为计划载体。

15. 用户可以在 Plan 中看到计划项标题、关联 issue、计划时间范围、计划顺序、预估工作量或时间、当前完成状态。

16. 用户可以调整计划项的顺序。

17. 用户可以把计划项标记为：
    - planned：计划中，尚未执行完成。
    - done：已完成或今日目标已达成。
    - skipped：今天决定跳过或延期。

18. 标记计划项状态不应自动把关联 issue 改为 done，除非用户明确选择同步 issue 状态。

19. 如果计划项关联的 issue 已经是 done 或 cancelled，计划页应清楚显示它已经处于终态。

20. 如果用户把一个已完成或已取消的 issue 加入计划，系统可以允许，但应给出明确状态提示，避免用户误以为它仍待执行。

21. 用户可以从 issue 详情把该 issue 加入 Plan。

22. 用户可以从 issue 列表、看板或全局搜索进入 issue 详情，再加入计划。

23. 用户可以从 Plan 候选区把未计划 issue 加入计划。

24. Plan 候选 issue 应优先展示与用户相关、未完成、未取消的 issue。

25. Plan 候选 issue 应考虑优先级、日期和当前状态排序，让最值得处理的 issue 更容易被看到。

26. Plan 是唯一的计划概念；用户不需要在多套计划模型之间切换心智。

27. Plan 可以按时间范围查看：
    - Today：今天准备推进的计划项。
    - Tomorrow：下一天准备推进的计划项。
    - Later：后续可能推进但尚未进入 Today/Tomorrow 的计划项。

28. V1 默认优先支持 Today 和 Tomorrow；Later 可以作为后续扩展或轻量候选区存在。

29. “还没整理的 issue”指至少满足以下一种情况的 issue：
    - 标题或描述还不足以让用户知道下一步行动。
    - 没有明确优先级。
    - 没有明确状态或仍停留在 backlog。
    - 没有负责人，或负责人不明确。
    - 没有日期、计划项或执行入口。

30. “已经准备执行的 issue”指用户能直接开始行动的 issue，至少应具备：
    - 用户能理解的标题。
    - 当前状态不是 done 或 cancelled。
    - 明确 next step，或详情中已有足够上下文让用户可以开始。
    - 用户知道它是否应进入 Plan，以及进入哪个时间范围。

31. 系统可以通过视图、筛选、候选区或提示帮助用户发现“还没整理的 issue”，但不应要求用户一次性补齐所有字段才能保存 issue。

32. 用户可以把未整理 issue 加入计划，但计划页应提示它可能缺少 next step 或关键上下文。

33. 用户可以从计划项直接打开关联 issue。

34. 用户可以从计划项启动执行。

35. 启动执行时，用户可以选择 Focus/Flowtime/Pomodoro/快速开始中的一种模式。

36. 执行开始前，用户可以看到或填写当前要做的 next step。

37. 执行开始前，如果用户觉得难以开始，可以记录开始阻力原因。

38. 开始阻力原因至少包括：
    - 下一步不清楚。
    - 任务太大。
    - 精力不足。
    - 回避。
    - 被打断。
    - 被阻塞。
    - 其他。

39. 如果用户选择“精力不足”作为开始阻力，系统应把它视为精力信号，而不仅是一条普通备注。

40. 用户可以从 issue 详情直接启动 Focus/计时，而不必先进入 Plan。

41. 用户从 issue 详情启动 Focus/计时时，该执行记录应与 issue 关联。

42. 用户从计划项启动 Focus/计时时，该执行记录应与计划项和关联 issue 建立可见关系。

43. 如果用户从无关联 issue 的计划项启动 Focus，该执行记录应与计划项关联，并在复盘中显示为“无关联 issue 的计划执行”。

44. 无关联 issue 的计划项可以后续转为 issue，转换后历史执行记录应能让用户理解它曾来自该计划项。

45. 无关联 issue 的计划项不应出现在 issue 列表或 issue 看板中，除非用户明确把它转换为 issue。

46. 用户启动 Focus 后，应看到当前正在执行的 issue 或计划项、计时状态、next step、已用时间和可用操作。

47. Focus 执行中，用户可以暂停、恢复、完成、放弃。

48. 用户暂停或放弃 Focus 时，可以记录原因。

49. 暂停或放弃原因中的“精力不足”应进入精力反馈。

50. 用户完成 Focus 时，可以记录完成备注。

51. 用户完成 Focus 后，系统应形成一条实际执行记录，用户可以在 issue、Plan、time tracking 和日复盘中回看到它。

52. Focus 完成后，系统可以建议休息。

53. 用户可以开始建议休息、跳过休息或完成休息。

54. 跳过休息是一个精力风险信号，系统应在复盘或下一轮计划中体现。

55. 完成休息是一个恢复信号，系统应在复盘或下一轮计划中体现。

56. 用户从计划项启动执行后，计划项不应在执行刚开始时自动变成 done。

57. 用户完成执行后，系统可以提示用户是否把计划项标记为 done。

58. 如果用户的 Focus 完成但 issue 仍未完成，用户应能保留 issue 当前状态，并只把本次执行记录为一次推进。

59. 如果用户认为 issue 已完成，用户可以在完成执行后把关联 issue 更新为 done。

60. 如果用户放弃执行，系统不应自动把计划项标记为 skipped；用户应能明确选择是否跳过、延期或保留 planned。

61. 用户可以为计划项设置预估时间或预估工作量。

62. 用户执行后可以看到计划预估与实际执行时间的差异。

63. 计划预估与实际执行差异用于帮助用户调整后续计划，不用于评价用户表现。

64. 用户可以在 Plan 中看到当前时间范围的计划总量。

65. 系统应根据用户精力信号提示计划负载是否偏高。

66. 用户可以在每日开始或计划前记录当前精力水平。

67. 每日开始或计划前记录的精力是“计划前容量信号”，用于帮助用户判断当天或明天能承担多少工作。

68. 精力水平使用轻量输入，范围为 1 到 5：
    - 1：很低。
    - 2：偏低。
    - 3：正常。
    - 4：良好。
    - 5：很好。

69. 记录精力水平是可选的；用户不记录时，计划和执行仍可继续。

70. 用户可以记录精力备注。

71. 用户可以标记是否需要降低负载或安排恢复。

72. 如果用户记录低精力或需要恢复，系统应在计划中降低大块深度工作建议，或提示用户减少计划项。

73. 如果用户记录高精力，系统可以提示把最重要或最难的 issue 放在高精力窗口。

74. 系统不应把精力建议表现为医疗建议。

75. 系统不应因为用户低精力而阻止用户创建、计划或执行 issue。

76. 系统可以提示“计划可能过载”，但最终是否继续由用户决定。

77. 当用户的计划总量明显超过默认可用工作容量时，系统应显示负载提示。

78. 用户可以忽略负载提示继续计划。

79. 用户可以根据负载提示减少计划项、降低预估时间、标记恢复或把计划项延期。

80. Plan 应展示一个可理解的精力/容量摘要，而不是只展示原始数字。

81. 精力/容量摘要至少能表达：
    - 当前计划总量。
    - 低精力或恢复需求是否存在。
    - 是否建议降低负载。

82. 用户完成一天后，可以生成或查看日复盘。

83. 日复盘应汇总当天推进过的 issue。

84. 日复盘应汇总当天完成的 issue。

85. 日复盘应汇总计划但未完成或跳过的计划项。

86. 日复盘应汇总 Focus 执行情况，包括完成次数、放弃次数、专注总时长、休息跳过/完成情况。

87. 日复盘应包含精力相关信号，包括用户主动记录的精力水平、精力备注、恢复需求、执行中记录的低精力原因。

88. 日复盘应帮助用户回答：
    - 今天推进了什么？
    - 哪些 issue 没推进？
    - 为什么没推进？
    - 明天应该继续、降低负载还是调整顺序？

89. 用户确认日复盘时，可以补充或修改当天精力水平。

90. 日复盘时记录的精力是“回顾信号”，用于描述当天实际消耗、恢复情况和明天负载建议。

91. 如果用户同一天既记录了计划前容量信号，又在复盘中记录了回顾信号，系统应让用户理解两者分别代表“开始时的预期/状态”和“结束后的实际感受”。

92. 如果界面只显示一个当日精力值，应优先说明它来自哪个阶段，避免用户误以为系统自动推断了全天精力。

93. 用户确认日复盘时，可以补充精力备注。

94. 用户确认日复盘时，可以标记明天需要降低负载或安排恢复。

95. 用户不填写精力字段也可以确认复盘。

96. 确认复盘后，系统应把复盘信号用于下一轮 Plan 建议。

97. 下一轮 Plan 应参考前一轮复盘中的 issue 进展、未完成计划项、低精力信号和恢复需求。

98. 下一轮 Plan 应优先把未完成但仍重要的 issue 带回用户视野。

99. 下一轮 Plan 不应无提示地丢弃上一轮 skipped 或未完成的计划项。

100. 上一轮 planned 但没有完成、没有跳过、也没有执行记录的计划项，应在下一轮 Plan 中作为“未处理”或“未完成”回流，而不是静默消失。

101. 上一轮 skipped 的计划项应以“已跳过/待重新决定”的方式回流，用户可以选择重新加入、延期或放弃。

102. 上一轮有执行记录但未标记 done 的计划项，应以“已推进但未完成”的方式回流。

103. 用户主动放弃的计划项和因为时间不够未处理的计划项应尽量区分，避免下一轮计划误判原因。

104. 如果上一轮低精力、放弃 Focus 多、跳过休息多或用户标记需要恢复，下一轮 Plan 应提示降低负载或安排恢复。

105. 如果上一轮执行顺畅且精力良好，下一轮 Plan 可以建议更高挑战度或更多深度工作。

106. Plan 建议应可被用户编辑，不能作为不可修改的自动决策。

107. 用户可以确认 Plan。

108. 确认 Plan 表示用户认可该计划作为下一轮执行参考。

109. 确认 Plan 不应自动开始任何 Focus 或 timer。

110. 如果用户重新生成 Plan，系统应提示可能覆盖当前草稿内容或计划项调整。

111. 用户重新生成 Plan 后，应能看见新的建议内容。

112. 如果已有用户手动调整过的计划项，重新生成不应静默丢弃这些调整。

113. 系统生成、重新生成、carry over、降低负载和排序建议都属于建议；在用户确认前，不应自动改变 issue 状态、计划项状态或删除用户手动调整。

114. 如果系统建议把 issue 加入计划，用户应能接受、忽略或稍后处理。

115. 如果系统建议降低负载，用户应能保留原计划继续执行。

116. 当计划生成失败时，用户应看到错误状态，并能重试。

117. 当复盘生成失败时，用户应看到错误状态，并能重试。

118. 当计划或复盘为空时，页面应显示空状态，引导用户生成计划、创建 issue 或开始记录执行。

119. 当 issue 没有关联执行记录时，issue 详情应清楚显示暂无执行记录，而不是隐藏该区域导致用户误解。

120. 当 issue 有执行记录时，issue 详情应显示近期执行记录和总计信息。

121. 用户应能从 issue 详情看到该 issue 是否出现在 Plan 中，以及对应时间范围。

122. 如果同一 issue 同时出现在多个日期计划中，用户应能理解它们是不同日期的计划安排，而不是重复 issue。

123. 如果 issue 被归档，普通 Plan 候选区默认不应推荐它。

124. 如果已在计划中的 issue 后来被归档，计划项应保留可见历史，但应提示该 issue 已归档。

125. 如果 issue 被删除或变得不可见，计划项和执行历史应尽量保留可读标题或不可用状态，不能让页面崩溃。

126. 用户只能看到自己有权限访问的 issue 和自己的个人执行/精力信号。

127. 个人精力记录默认不应展示给其他 workspace 成员。

128. 团队成员可以看到 issue 本身的协作信息，但不应默认看到用户私人的精力备注。

129. 个人 Plan 默认只对计划创建者本人可见。

130. 计划项默认只对计划创建者本人可见，即使计划项关联的是团队 issue。

131. 从个人计划或个人 Focus 产生的私密备注默认只对本人可见。

132. 与 issue 关联的实际执行时间是否在 issue 详情中对团队可见，应有明确边界：
    - issue 的协作状态、评论和公开活动仍按团队协作规则可见。
    - 用户私人精力备注、恢复需求、低精力原因默认不可见。
    - 用户主动公开的工作备注可以出现在 issue 协作上下文。

133. Time tracking 中用于个人复盘的记录默认对本人可见；团队统计视图只能显示必要的汇总，不应暴露个人精力备注。

134. Agent 可以根据明确分配的 issue 工作，但 V1 不应默认把用户私人精力备注注入 agent 上下文。

135. 如果未来让 agent 使用精力信号，必须明确告知用户哪些信号会被使用。

136. 系统不应用羞辱、评价或绩效化语言描述低精力、跳过休息或未完成计划。

137. 系统文案应把精力信号表达为计划输入和恢复提示。

138. 桌面端应提供一个清晰的个人执行主视图，能同时看到计划、候选 issue、当前 Focus/计时状态和精力摘要。

139. 移动端可以使用单栏布局，但用户应能完成创建/查看 issue、查看计划、启动执行、完成执行和记录精力。

140. 页面交互关系如下：

   ```text
   Issues / Board / Search
        ├── open issue detail
        │      ├── add to plan
        │      └── start focus
        │
        └── selected issue candidates
               ↓
   Plan
        ├── reorder plan items
        ├── set estimate
        ├── mark done / skipped
        └── start focus
               ↓
   Focus / Timer
        ├── pause / resume
        ├── abandon with reason
        ├── complete with note
        └── break start / skip / complete
               ↓
   Daily Review
        ├── review issue progress
        ├── record energy
        └── confirm
               ↓
   Next Plan
   ```

141. 页面级交互改动如下：

   ```text
   +----------------+        +------------------+        +----------------+
   | Issues / Board | -----> | Issue Detail     | -----> | Focus / Timer  |
   | Search / New   |        | Add to plan      |        | Current work   |
   |                |        | Start focus      |        | Complete/Stop  |
   +-------+--------+        +--------+---------+        +--------+-------+
           |                          |                           |
           | add selected issue       | add to plan               | outcome
           v                          v                           v
   +----------------+        +------------------+        +----------------+
   | Plan           | -----> | Plan Item Detail | -----> | Daily Review   |
   | Candidates     |        | Estimate/status  |        | Energy/progress|
   | Capacity       |        | Start focus      |        | Confirm        |
   +-------+--------+        +------------------+        +--------+-------+
           ^                                                       |
           | carry over / suggestions                              |
           +---------------------- Next Plan <----------------------+
   ```

142. 计划项状态机如下：

   ```text
   planned
      | start focus
      v
   in_progress
      | complete focus
      v
   progressed
      | user marks done              user skips/defer
      v                              v
   done                           skipped
      |                              |
      +---------- review ------------+
                     |
                     v
              next plan decision

   planned -- no action by day end --> unhandled --> next plan decision
   ```

143. Issue 状态和计划项状态关系如下：

   ```text
   Issue status:      backlog / todo / in_progress / in_review / done / cancelled
                             ^
                             |
                  user explicitly changes issue
                             |
   Plan item status: planned / in_progress / progressed / done / skipped / unhandled

   Rule:
   Plan item done does not imply issue done.
   Issue done does not rewrite old plan item history.
   ```

144. 信号流向如下：

   ```text
   Issue fields + plan item estimates
              |
              v
   Plan capacity summary <---- morning energy check-in
              |
              v
   Focus / timer execution
      |          |            |
      |          |            +--> low energy / interruption / blocked reasons
      |          +--------------> actual duration
      +-------------------------> completion / abandon / break skipped
              |
              v
   Daily review
      | issue progress
      | unfinished / skipped / unhandled items
      | evening energy check-in
      v
   Next Plan suggestions
   ```

145. 用户在 Plan 中操作计划项时，应尽量不离开当前页面。

146. 用户打开 issue 详情、Focus、复盘或计划时，应能返回上一层工作上下文。

147. 用户通过全局搜索打开 issue 后，应能继续把它加入计划或启动执行。

148. 用户通过全局快捷入口创建 issue 后，应能继续进入整理或计划。

149. 键盘和可访问性：
    - 计划项、issue 候选、开始执行、完成执行、跳过、确认复盘等主要操作应可通过键盘访问。
    - 精力水平输入应有可识别 label。
    - 负载提示和保存/生成状态不应只依赖颜色表达。
    - Focus 当前状态应有可读文本，而不是只靠图标。

150. 加载状态：
    - 加载 issue 列表时显示列表加载状态。
    - 加载计划时显示计划加载状态。
    - 加载复盘时显示复盘加载状态。
    - 加载 Focus 当前状态时显示当前状态加载或 idle 状态，不应出现空白页面。

151. 错误状态：
    - issue 加入计划失败时，用户应看到错误提示，并且当前 issue 不应被表现为已加入。
    - 启动 Focus 失败时，用户应看到错误提示，并留在可重试状态。
    - 完成 Focus 失败时，用户应看到错误提示，当前计时或本地状态不能被静默丢失。
    - 更新计划项失败时，用户应看到错误提示，并能重试或刷新。

152. 并发与重复操作：
    - 用户快速点击“开始执行”多次时，不应创建多个同时运行的执行状态。
    - 同一用户同一时间只能有一个当前 Focus/计时主状态。
    - 如果已有运行中的 Focus 或 timer，用户启动新的执行时应看到明确处理方式，例如停止现有执行、切换执行或取消新启动。

153. 计划项状态和 issue 状态可以不同步，但界面必须让用户理解差异。

154. 一个计划项 done 表示“该计划项今天达成”，不一定表示整个 issue 完成。

155. 一个 issue done 表示 issue 本身完成，即使它过去的某个计划项不是 done。

156. 用户应能从复盘中看到这种差异，而不是只看到最终 issue 状态。

157. 系统应避免制造重复工作：
    - 同一 issue 已在当天计划中时，再次加入应提示已存在或聚焦到现有计划项。
    - 同一 issue 可以出现在不同时间范围的 Plan 中。
    - 同一 issue 可以有多次执行记录。

158. V1 的闭环完成标准是：用户可以从 issue 出发进入计划，从计划进入执行，从执行形成记录，从记录进入复盘，从复盘影响下一轮计划。

159. Issue 可以有类型，用来表达它是什么性质的工作，而不是它现在处于什么状态。

160. Issue type 应支持 workspace 级扩展；系统可以提供内置默认类型，但不应把 issue type 永久限制为代码中的固定枚举。

161. V1 默认 seed issue type 至少包括：
    - task：普通执行任务。
    - feature：功能或交付型工作。
    - bug：缺陷修复。
    - chore：维护、整理、事务性工作。
    - research：调研、澄清或方案探索。
    - recovery：恢复、降负载或精力恢复安排。

162. Issue type 不替代 status、priority、label 或 assignee：
    - status 表达进度。
    - priority 表达重要性或紧急程度。
    - label 表达团队自定义分类。
    - issue type 表达工作形态。

163. Plan 中应显示关联 issue 的 issue type，让用户能判断当前计划是否都是同一种工作形态。

164. Plan 候选区可以按 issue type 分组、筛选或排序，帮助用户在深度工作、事务性工作、调研和恢复安排之间取得平衡。

165. Issue type 应包含负载画像，用于帮助 Plan 判断它更接近深度工作、轻量工作、恢复安排还是中性工作。

166. 当用户记录低精力或需要恢复时，Plan 可以优先提示负载画像为轻量工作或恢复安排的候选项，或提醒减少深度工作型计划项。

167. 用户可以新增自定义 issue type，并设置名称、颜色、图标、排序和负载画像。

168. 已被 issue 使用的 issue type 不应被硬删除，只能归档或隐藏，避免历史 issue、Plan、Focus 和 Review 失去解释。

169. 内置 issue type 可以调整展示名、颜色、图标和排序，但稳定标识应保留，方便导入导出、AI prompt 和后续迁移。

170. 无关联 issue 的计划项后续转为 issue 时，用户应能选择或确认 issue type；默认可以使用 workspace 的默认 type。

171. 修改 issue type 后，Plan、Focus 和 Review 中展示的关联 issue 应尽量反映最新 issue type；历史记录可以保留创建时上下文，但不应让用户误以为生成了另一条 issue。
