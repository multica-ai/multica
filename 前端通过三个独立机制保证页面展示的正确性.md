前端通过三个独立机制保证页面展示的正确性：

---

## 1. 刷新页面后如何恢复

**计时器状态**：`useCurrentTimeEntryQuery` 在组件挂载时立即发起请求，获取后端的运行中条目： [1](#7-0) 

`LiveDuration` 的初始 state 用 lazy initializer 直接从 `entry.start` 计算当前已过时间，**不会从 0 开始跳**： [2](#7-1) 

所以刷新后的流程是：
1. 页面加载 → `useCurrentTimeEntryQuery` 请求后端
2. 后端返回 `{start: "2024-01-01T10:00:00Z", duration: -负数}`
3. `LiveDuration` 初始化时计算 `Date.now() - start`，直接显示正确的已过时间（如 `0:05:23`），不会闪 `0:00:00`

**视图选择（Calendar/List/Timesheet）**：`useTimerViewStore` 用 Zustand `persist` 将 `view`、`calendarSubview`、`timerInputMode` 持久化到 localStorage（key: `opentoggl:timer-view`），刷新后自动恢复： [3](#7-2) [4](#7-3) 

---

## 2. 停止计时器后 UI 如何更新

停止时用了**乐观更新（Optimistic Update）**，分三步：

**`onMutate`（请求发出前立即执行）**：
1. 把 `currentTimeEntry` 缓存设为 `null` → `TimerElapsedDisplay` 立即显示 `0:00:00`
2. 把列表缓存中该条目的 `stop` 设为当前时间、`duration` 设为正数 → 列表行立即变为已停止状态
3. 取消正在进行的后台轮询，避免覆盖乐观状态 [5](#7-4) 

**`onError`（请求失败时）**：回滚到操作前的快照，UI 恢复运行中状态： [6](#7-5) 

**`onSuccess`（请求成功后）**：`invalidateQueries` 触发后台重新拉取，用服务端真实数据覆盖乐观状态： [7](#7-6) 

---

## 3. 开始计时的乐观更新

开始计时同样有乐观更新，`onMutate` 立即构造一个临时 entry（`id: -Date.now()`）写入缓存，让 UI 立即响应，不等后端返回： [8](#7-7) 

`onSuccess` 后用服务端返回的真实 entry 替换临时 entry，并把它插入列表缓存（防止 invalidate 期间短暂消失）： [9](#7-8) 

---

## 整体保障机制总结

| 场景 | 机制 |
|---|---|
| 刷新页面后计时器时间正确 | `LiveDuration` 用 `Date.now() - entry.start` 初始化，不依赖本地状态 |
| 刷新页面后视图选择保留 | Zustand `persist` 写入 localStorage |
| 停止计时器立即响应 | 乐观更新：`onMutate` 先改缓存，`onError` 回滚 |
| 多端/多标签页同步 | `refetchInterval: 30_000` 每 30 秒轮询后端，`refetchOnWindowFocus: true` 切换标签页时立即同步 |
| 网络失败不乱显示 | `onError` 中检查 `navigator.onLine`，离线时不回滚（避免误判） |
