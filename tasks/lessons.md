# Lessons

- 部署类需求如果用户只说“部署到 VPS”，不要默认把所有前端都纳入范围；在输出方案前先明确是否包含 marketing、workspace、daemon。
- 当用户明确表示会自行处理 OpenResty 配置时，不要继续把交付重点放在反向代理配置和 `openresty -t`；应转而聚焦应用打包方式、运行产物和必需配置文件。
- 当用户明确说明会在 VPS 上直接 clone 源码并编译时，不要继续把“发布包上传”当成主路径；应切换为源码构建路径，并把重点放到环境变量、构建命令和启动顺序。
- 做中大型设计文档时，除了文字决策和风险表，还要提供 ASCII 架构图、数据流图或状态图；否则 review 成本高，关键边界不够直观。
- 做产品闭环设计时，必须同时提供页面交互图；仅有数据流或状态机不够，用户无法评审入口、跳转、弹窗和回流路径。
- 修改依赖数据库迁移的功能后，`make check` 这类总验证必须先确保迁移已跑到当前代码版本；只检查 PostgreSQL 可连接会漏掉 schema 漂移。
- 当用户明确将某条产品线或功能挂起时，停止沿旧规格推进实现；保留已产出的 spec 作为后续参考，当前工作切换到用户重新声明的核心闭环。
- 用户输入中出现可能由语音转写导致的同音歧义时，不要直接扩展成新领域；先结合项目上下文判断，必要时确认。例如本项目里“医学闭环”更可能是“issue 闭环”。
- 设计 Plan/执行闭环时，优先复用和增强 issue 的核心语义字段（例如 issue type），不要把 Plan 设计成脱离 issue 模型的平行系统。
- 用户明确要求“按照设计完成所有开发工作”这类跨前后端功能时，最终验证必须跑完整 `make check`；分段跑 typecheck/unit/backend tests 不能替代 E2E。
