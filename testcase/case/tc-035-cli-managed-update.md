Purpose: Verify that the Fork CLI managed update flow uses the configured channel/manifest as the primary source for version discovery, binary download, daemon auto-update, and platform hardening, while retaining GitHub Release only as a fallback path.

Preconditions: The Multica CLI is installed locally. `MULTICA_UPDATE_MANIFEST_URL` or the CLI config points to a test manifest endpoint that contains a newer version and an asset for the current OS/arch. The manifest asset includes `download_url`, `checksum`, `channel`, and, when needed, `archive_name`. A separate test fixture can make the manifest download fail to verify fallback behavior.

User flow: Run `multica --version` to note current version. Run `multica update --channel test` (or equivalent channel-specific install/update path), or trigger daemon auto-update, with the test manifest endpoint configured. Observe the update logs and network requests. The CLI should read the latest version from the manifest, select the asset matching the current OS/arch and channel, download the archive from the asset `download_url`, verify the archive with the asset `checksum`, extract the `multica` binary, apply macOS ad-hoc codesign when relevant, replace the current binary through the existing backup/restore path, and then report update completion. Run `multica --version` again to confirm the updated version. Restart the daemon with login-shell environment and verify update configuration persists. In a separate fallback fixture, make the manifest asset download fail and verify that the GitHub Release download path is used only after the manifest path reports failure.

Expected results: The CLI compares its current version against the channel-aware manifest-reported latest version. The primary download request goes to the manifest asset `download_url`, not to the GitHub Release asset `browser_download_url`. Checksum verification uses the manifest asset `checksum`, not GitHub `checksums.txt`. The existing binary is backed up before replacement. macOS downloads are ad-hoc codesigned after extraction. Daemon auto-update is enabled by default when `update_manifest_url` is configured. After update, `multica --version` shows the new version number. If replacement fails mid-way, the backup is restored and the CLI remains functional. If the manifest asset download fails, the CLI logs the manifest failure and then falls back to GitHub Release download without removing the GitHub code path.

Notes for automation: This test modifies the CLI binary; run it in an isolated temp install path or verify rollback works. Use a local HTTP fixture for the manifest and archive so requests can be asserted precisely. This case guards against the merge regression from OPE-366/OPE-1005 where version lookup stayed manifest-based but the binary download was accidentally reverted to GitHub Release.
