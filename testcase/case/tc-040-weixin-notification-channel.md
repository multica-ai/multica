Purpose: Verify that the OpenClaw WeChat notification channel can be bound in settings, per-event toggles work, reply notifications are delivered, and notifications are deduplicated at comment level.

Preconditions: The Multica web app is reachable. The user is signed in. An OpenClaw runtime is online and connected to the workspace (the WeChat notification section only appears when an OpenClaw runtime is detected). The user has a valid WeChat ID for binding.

User flow: Navigate to Settings → Notifications (or the equivalent notification preferences page). Locate the "WeChat Notification" or "微信通知" section. Verify it is visible (if no OpenClaw runtime is online, verify the section is hidden and skip remaining steps). Enter a WeChat ID in the binding input field and confirm binding. After binding succeeds, observe per-event toggle switches for: mentioned, task_completed, task_failed, replied. Toggle each event on and off individually. Trigger enabled events: have another user mention you in a comment, reply to one of your comments, and complete/fail an agent task on an issue you subscribe to. Verify that notifications are dispatched (check delivery status or WeChat receipt).

Expected results: The WeChat notification section appears only when an OpenClaw runtime is online in the workspace. The binding flow accepts a WeChat ID and stores the binding (shows "已绑定" / bound status). Per-event toggle switches allow granular control of which events trigger WeChat notifications. Reply notifications go to the parent comment author when `replied` is enabled. Task and mention events create at most one delivery per recipient/comment/channel even when multiple listeners or subscriber paths match. Notification content uses emoji/render-mode preferences consistently with DingTalk where applicable. If the OpenClaw runtime goes offline after binding, the section should indicate the runtime is unavailable but the binding is preserved.

Notes for automation: The WeChat notification section visibility depends on runtime status. Check for the section heading or a feature-flag indicator. Binding success can be verified by the UI state change (showing bound status and toggle switches). Actual delivery verification may require checking backend logs or a mock WeChat endpoint.
