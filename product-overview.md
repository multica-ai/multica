## 1. Multica 是什么

### 一句话定位

**Multica 是一个以任务管理为核心，串联时间记录、项目协作与 AI 执行助手的工作系统，目标是在同一套对象上同时支撑个人成长复盘与团队协作执行。**

它帮助个人与团队回答四件事：

1. 现在最重要的工作是什么
2. 谁在做，做到哪一步
3. 时间花在了什么工作上
4. 今天如何复盘，明天如何更低成本地开始

AI 仍然存在，但不再是产品定义本身。它除了作为执行增强层帮助完成任务，也会逐步承担资料查找、调研总结、任务分发建议、夜间复盘和次日计划等助手角色。

### 它解决的问题

很多团队的问题不是“没有任务系统”，而是这些信息分散在不同地方：

- 任务在看板里
- 时间在另一个工具里
- 项目状态在单独的文档里
- AI 工作流又在终端或外部平台里

结果是：

- 任务和时间脱节，知道做了什么，不知道花了多少投入
- 项目和执行脱节，知道项目目标，不知道具体 issue 如何推进
- AI 和团队工作流脱节，agent 能做事，但不在同一套协作坐标里

Multica 现在的产品目标，是把这些能力收拢到同一个工作区里，让团队围绕统一的工作对象来推进执行。

在当前阶段，这个目标还有一个更具体的个人来源：现有工具往往能管理任务，却很难同时满足更强提醒、完整时间段记录、AI 辅助复盘，以及后续长期沉淀这几类需求。Multica 的演进方向，是在不丢掉团队协作和 agent 资产的前提下，把这些需求逐步收拢到同一套产品骨架里。

### 新版本的产品主轴

当前版本的 Multica，核心不再是“如何管理很多 coding agent”，而是“如何让团队围绕任务稳定推进执行”。因此文档和产品叙事都围绕三条主线展开：

- **任务管理**，定义工作的最小执行单元
- **时间记录**，记录工作投入和执行轨迹
- **项目协作**，把多个任务组织成更高层目标

AI 相关能力被保留，但位置后移，作为增强层出现。

### 当前阶段的项目目标与展望

当前这条产品线的直接目标，不是做一个泛化平台，而是先围绕真实日常工作流程补齐几个高价值环节：

- **把任务管理做深**，让任务、项目、时间安排和协作关系足够可靠，成为后续所有自动化和复盘能力的事实底座
- **保留并强化 agent**，让它从“可执行任务的 AI 成员”逐步扩展为资料查找、调研总结、任务分发、夜间复盘和次日计划助手
- **让时间记录不只服务记账**，还要成为夜间复盘、次日准备和专注支持的输入层
- **补齐更强的提醒能力**，支持邮件、ntfy 或多通道组合，避免执行闭环只依赖浏览器或移动端在线状态
- **把知识管理继续后置**，先让任务、时间和 AI 辅助的主链路稳定，再定义长期沉淀的对象模型

从这个角度看，Multica 的中近期展望不是回到旧的 AI-first 产品壳，也不是转成纯个人习惯工具，而是演进成一套同时服务个人执行闭环与团队协作的任务系统。

### 产品形态

从系统形态看，Multica 目前由几部分组成：

- **Workspace Web App**，团队主要使用界面
- **Go Backend + PostgreSQL**，承载业务、实时同步和数据存储
- **CLI / Daemon**，用于接入本地 agent runtime

对多数用户来说，首先接触的是 Web 侧的任务、项目、时间记录和协作能力。CLI 与 daemon 主要服务于 AI 相关流程。

## 2. 为什么产品主线发生了调整

旧版产品全景文档建立在一个明确前提上：**Agent 是产品世界观的中心角色。**

这个前提在早期是合理的，因为当时产品最有辨识度的部分，确实是：

- agent 可被指派任务
- daemon 可接入本地 runtime
- skills 可以沉淀经验
- issue、comment、task run 都围绕 AI 执行展开

但在新的产品方向里，这套结构有两个问题：

### 2.1 它解释不了日常执行工作的主体

对团队来说，最常发生的动作不是“配置 agent”，而是：

- 创建和拆分任务
- 安排今天与未来的工作
- 跟进项目进度
- 记录时间投入
- 在同一条任务线上协作

如果文档仍把 AI 放在最前面，就会让真正高频的产品主线被埋掉。

### 2.2 它容易把 AI 误写成产品本体

AI 是能力，但不是全部。  
当产品开始承接任务管理、时间记录、项目协作这些更基础也更稳定的需求时，AI 应该被放回合适位置：

- 它是执行的一种方式
- 是 assignee 的一种形态
- 是团队能力的放大器
- 不是所有页面、所有概念、所有模块的起点

### 2.3 新文档的目标

因此，这份新文档要完成三件事：

1. 重新定义产品主线
2. 解释旧的 AI 资产在新结构中的位置
3. 给未来继续补功能时，留下更稳定的产品骨架

### 2.4 暂不纳入主结构的能力

“知识管理”在这次讨论里被认为重要，但定义还不稳定。当前文档不把它写成一级模块，避免把尚未定形的能力写成已定事实。

这不代表它不重要，只代表本版产品全景先不把它作为主结构的一部分。

## 3. 核心概念词典

理解下面这些概念，是理解新产品结构的前提。

| 概念 | 定义 | 在新结构中的位置 |
|------|------|------------------|
| **Workspace 工作区** | 一切数据和协作的边界。成员、任务、项目、时间记录、AI 配置都隔离在 workspace 内 | 产品的容器 |
| **Issue 任务** | 最核心的执行单元。一个任务可以有标题、描述、状态、优先级、负责人、时间安排、评论、依赖关系 | 第一核心对象 |
| **Project 项目** | 一组相关任务的聚合体，用于表达更高层目标、范围和状态 | 第二层组织对象 |
| **Time Entry 时间记录** | 一段实际投入的工作时间，可运行中、可停止、可编辑、可继续 | 执行投入的事实记录 |
| **Member 成员** | 用户在某个 workspace 下的身份 | 人类协作者 |
| **Assignee 负责人** | 某个任务当前由谁处理，可以是 member，也可以是 agent | 执行责任归属 |
| **Agent 智能体** | 可被指派任务的 AI 行动者 | AI 增强层中的参与者 |
| **Runtime 运行时** | agent 实际运行的执行环境 | AI 执行基础设施 |
| **Daemon 守护进程** | 连接本地机器与平台的后台进程，用于发现 CLI、注册 runtime、认领任务 | AI 执行基础设施 |
| **Skill 技能** | 可复用的操作说明或经验沉淀，供 agent 运行时读取 | AI 辅助知识层 |
| **Inbox / Notifications 通知** | 个人的提醒与更新入口 | 协作辅助层 |
| **Board / Backlog / Today / Upcoming** | 不同的任务视图，用不同方式组织 issue | 任务管理的不同观察面 |
| **My Work** | 以“我”为中心看的任务列表 | 个人执行视角 |
| **My Time** | 以“我”为中心看的时间记录界面 | 个人投入视角 |
| **Calendar 日历** | 时间维度上的工作视图，既服务任务排期，也服务时间复盘 | 时间与计划的交汇面 |
| **Comment / Timeline** | 任务里的讨论和关键动作记录 | 协作轨迹 |
| **Workspace Context** | 工作区级的 AI 上下文说明 | AI 增强层中的共享背景 |

新结构里的一个关键变化是：

> **Issue、Project、Time Entry** 是产品主干，  
> **Agent、Runtime、Skill** 是增强能力，  
> **Inbox、Calendar、Settings** 是支撑整体体验的基础层。

## 4. 功能全景

### 4.1 任务管理

> **角色**：整个产品的核心执行层。

任务管理是 Multica 的第一主轴。无论团队是在做产品迭代、缺陷修复、运营排期，还是让 AI 参与执行，最后都要落到一个具体任务上。

#### 4.1.1 任务对象本身

一个 issue 是一条完整的工作记录，通常包含：

- 标题
- 描述
- 状态
- 优先级
- 负责人
- 开始时间 / 截止时间 / 结束时间
- 项目归属
- 标签
- 依赖关系
- 评论、反应、活动时间线

这意味着 issue 不只是“卡片”，而是同时承担：

- 执行单元
- 协作上下文
- 审计轨迹

#### 4.1.2 任务的多种视图

当前产品把同一批 issue 以不同方式组织出来：

- **Issues**，最标准的任务列表视图
- **Board**，按状态组织的看板视图
- **Backlog**，待排期、未进入近期执行窗口的任务
- **Today**，今天应处理的任务
- **Upcoming**，未来计划处理的任务
- **My Work / My Issues**，当前用户负责的任务
- **Calendar**，把任务放到时间坐标里看

这些视图不是独立模块，而是同一套任务系统的不同观察面。

#### 4.1.3 任务里的协作能力

任务详情页承载了大量协作动作：

- 评论与回复
- 时间线，记录状态变更、负责人变化等关键动作
- 反应与轻量互动
- 标签和依赖关系管理
- 文件上传
- 任务链接与层级关系

因此，issue 详情页本质上是一个执行工作台，而不是一个静态详情页。

#### 4.1.4 任务和时间、项目的关系

在新结构里，任务不是孤立存在的：

- 它可以归属于某个 **project**
- 它可以带有自己的时间安排
- 它可以直接启动或承接 **time entry**
- 它也可以被指派给 **agent**

换句话说，任务是整个产品里最重要的连接点。

#### 4.1.5 任务管理在整体中的位置

如果只保留一个产品主轴，那就是任务管理。  
时间记录、项目协作、AI 辅助，都是围绕任务展开的不同层面：

- 时间记录解决“投入”
- 项目协作解决“聚合”
- AI 辅助解决“执行方式”

### 4.2 时间记录

> **角色**：把“工作做了什么”补全成“工作花了多少投入，并把这些投入转化为复盘与专注输入”。

时间记录是这次产品转移里被显著提升的一条主轴。它不再只是附带能力，而是执行系统的第二核心面。

#### 4.2.1 时间记录解决什么问题

很多任务工具能表达状态，但不能表达投入。  
团队会知道一个任务“已完成”或“进行中”，却不知道：

- 花了多少时间
- 时间被哪些工作切走
- 某个项目实际消耗了多少投入
- 一天或一周的工作分布长什么样

时间记录补上的是这层事实。

在当前阶段的产品目标里，这层事实还会进一步成为夜间复盘、次日计划和专注支持的输入，而不是只停留在统计意义上。

#### 4.2.2 当前的时间记录形态

当前产品已经具备完整的个人时间记录界面：

- **My Time**，按天聚合查看时间记录
- **My Time Calendar**，用日历方式查看和编辑时间块
- **Global Timer Widget**，在全局侧边栏持续显示运行中计时器
- **Issue Timer Section**，在任务详情内直接开启或关联时间记录

用户可以：

- 开始计时
- 停止计时
- 编辑时间条目
- 删除时间条目
- 继续一条已停止的时间记录

这说明时间记录已经不是“统计字段”，而是一套独立的使用体验。

#### 4.2.3 时间记录的数据语义

一条 time entry 通常表达：

- 何时开始
- 何时停止
- 是否正在运行
- 描述是什么
- 归属于哪个任务或项目

因此它既可以作为个人工作日志，也可以作为项目投入的客观依据。

#### 4.2.4 时间记录在整体中的位置

在新产品里，时间记录承担四种价值：

1. **个人视角**，帮助一个人看到自己的工作流和投入分布
2. **任务视角**，帮助团队知道某项工作消耗了多少时间
3. **项目视角**，帮助项目协作具备更真实的推进背景
4. **复盘与专注视角**，为夜间回顾、次日计划和番茄/专注支持提供输入

任务系统定义“做什么”，时间系统定义“花了多少”，而下一阶段的个人执行闭环会进一步利用这些时间事实降低每日决策成本。

### 4.3 项目协作

> **角色**：把分散任务组织成更高层目标。

项目协作是第三条主轴。它的职责不是替代任务，而是为任务提供一个更大的组织边界。

#### 4.3.1 项目对象解决什么问题

如果只有任务列表，团队很快会遇到两个问题：

- 很难看出一批任务共同服务于什么目标
- 很难表达一个较大范围工作的整体状态

Project 正是用来解决这两个问题。

#### 4.3.2 当前的项目能力

当前产品已经有一套完整的项目界面：

- **Projects** 列表
- **Project Detail**，查看和编辑项目基础信息
- **Project Board**，在项目范围内看任务推进

项目本身通常包含：

- 标题
- 描述
- 图标
- 状态
- 归属的任务集合

#### 4.3.3 项目和任务的关系

在新结构里，project 不是一个脱离任务的高层空壳。它更像：

- 范围定义器
- 进度聚合器
- 协作上下文容器

可以用一句话理解：

> **Issue 是执行单元，Project 是协作单元。**

任务负责推动具体工作，项目负责让团队看清这批工作是不是朝着同一目标前进。

#### 4.3.4 项目协作在整体中的位置

项目层让产品从“任务工具”变成“团队执行工具”。  
因为它把单点任务、跨人协作、状态管理和时间投入，开始放进同一张更大的图里。

---

### 4.4 AI 辅助能力

> **角色**：增强执行，不主导产品定义。

AI 相关能力没有被删除，但它在产品叙事中的位置发生了变化。  
现在更准确的说法是：

**Multica 支持把 AI agent 纳入任务执行体系，但产品本身不再围绕 AI 展开定义。**

#### 4.4.1 Agent

Agent 仍然是一类重要行动者。它可以：

- 出现在 assignee 中
- 被分配任务
- 参与任务线程中的协作
- 以任务运行的方式完成一段工作

但 agent 现在只是 assignee 的一种，不再是文档里的总纲。

#### 4.4.2 Runtime 与 Daemon

AI 执行仍依赖 runtime 和 daemon：

- **Runtime** 表示某个可执行 agent 的环境
- **Daemon** 连接本地机器，发现可用 CLI，并认领分配给 agent 的任务

这层能力的价值，在于把外部 AI CLI 接入统一任务系统，而不是让团队离开平台去单独管理 AI。

#### 4.4.3 Skills

Skills 仍然重要，但它的意义也发生了变化。  
它更接近：

- 团队经验的复用载体
- agent 执行时可挂载的说明材料
- 某类工作方法的标准化表达

在旧叙事里，skill 更像世界观的一部分。  
在新叙事里，skill 是一项很强的增强能力，但不是入口概念。

#### 4.4.4 AI 辅助能力在整体中的位置

AI 相关模块现在主要服务三个场景：

1. **任务执行增强**，让部分工作可以交给 agent
2. **团队经验复用**，让重复性工作不必每次从零开始
3. **个人执行辅助**，让资料查找、调研总结、任务分发建议、夜间复盘和次日计划逐步进入同一套任务与时间坐标

因此，AI 不再是第一层目录的逻辑起点，而是围绕任务系统工作的加速层。

在当前项目展望里，AI 的价值不只是“替人执行一段工作”，也包括“帮助人更快进入工作”。它应优先挂在现有任务、时间和项目对象之上，而不是另起一套脱离主对象的泛化 AI 产品壳。

---

### 4.5 Workspace 工作区

> **角色**：整个产品的协作边界，也是所有核心能力的容器。

如果说 issue 是最核心的工作对象，那么 workspace 就是这些工作对象存在的坐标系。  
任务、项目、时间记录、成员、agent、skills，都不是直接散落在系统里，而是先属于某个 workspace。

#### 4.5.1 Workspace 解决什么问题

团队执行系统要长期成立，必须先回答一个问题：**谁和谁在同一个协作空间里工作。**

Workspace 解决的是这层边界问题：

- 哪些成员属于同一个团队空间
- 哪些任务、项目和时间记录彼此可见
- 哪些 AI 配置和辅助能力属于这个团队
- 哪些设置在团队范围内生效

没有 workspace，Multica 就只是一组页面集合。  
有了 workspace，它才变成一个可长期使用的多团队系统。

#### 4.5.2 当前的 Workspace 体验

当前产品已经把 workspace 放进日常导航与使用入口里：

- 侧边栏顶部可以查看当前 workspace
- 用户可以在 workspace 菜单里切换已有 workspace
- 可以直接创建新的 workspace
- workspace 切换后，任务、项目、通知和其它数据上下文一起切换

这意味着 workspace 不是后台设置项，而是用户日常会反复触达的顶层上下文。

#### 4.5.3 Workspace 承接哪些能力

在当前结构里，workspace 至少承接下面几类东西：

- **成员关系**，谁属于这个团队空间
- **工作对象**，这个空间里的 issue、project、time entry
- **协作体验**，通知、个人工作台、日历等视图的上下文边界
- **AI 增强层**，agent、runtime、skill、workspace context 等配置归属

可以把它理解成一层共享容器：  
主轴能力在里面运行，增强能力在里面挂载，基础体验也在里面成立。

#### 4.5.4 Workspace 在整体中的位置

在新文档里，workspace 不能被写丢，因为它不是附属设置，而是整个产品的组织基础。

可以用一句话概括它的角色：

> **Workspace 定义团队边界，Issue 定义执行单元，Time Entry 记录投入，Project 聚合目标。**

少了 workspace，后三者都失去协作语境。

---

### 4.6 协作与基础体验

> **角色**：把核心能力串成一个日常可用的产品。

除了任务、时间、项目和 AI 辅助层，当前产品还有一组不可忽略的基础体验模块。

#### 4.6.1 Inbox / Notifications

通知系统承担个人提醒入口的职责。  
它让用户知道：

- 有哪些任务更新
- 有哪些需要回看的协作动作
- 哪些变化与自己相关

它不是核心对象，但决定了用户是否能及时回到正确的任务上下文。

#### 4.6.2 Calendar

Calendar 是任务与时间两个主轴的交汇面：

- 一方面，它承接 issue 的时间安排
- 另一方面，它也与时间记录形成更强的时间视角

它的重要性不在于“又多一个视图”，而在于让工作从列表坐标进入时间坐标。

#### 4.6.3 Settings、Auth

这些模块定义产品的运行边界和使用环境：

- 用户身份与登录
- 设置页面

它们不定义产品主线，但决定多团队、多成员、长期使用是否成立。

#### 4.6.4 Sidebar 与个人工作台体验

当前导航结构已经体现出新产品取向：

- Projects
- Issues
- Board
- Backlog
- Today
- Upcoming
- My Work
- My Time
- Calendar
- Notifications

这套导航本身就在说明，产品中心已经从“Agent 管理台”转向“个人与团队执行系统”。

## 5. 典型工作流

下面用一个典型流程，说明新结构下各模块如何连起来。

### 5.1 接收和整理工作

团队成员先进入任务系统：

- 在 Issues 中查看全部任务
- 在 Backlog 中放置未进入执行窗口的工作
- 在 Today / Upcoming 中安排时间
- 在 My Work 中聚焦自己负责的部分

这个阶段的关键词是：**明确要做什么。**

### 5.2 进入具体执行

当某个任务开始推进时，用户会进入 issue 详情页：

- 阅读描述
- 查看依赖与链接
- 讨论、评论、补充信息
- 必要时直接开始计时

这个阶段的关键词是：**围绕一条任务线形成上下文。**

### 5.3 记录投入

执行过程中，时间记录开始发挥作用：

- 在 issue 内开启时间记录
- 在 My Time 查看今日和本月投入
- 在日历页看时间块分布

这个阶段的关键词是：**把执行轨迹变成可回看的事实。**

### 5.4 聚合到项目层

多个相关任务会被归入同一个 project：

- 让团队知道这批任务服务于哪个目标
- 让项目状态不必靠外部文档手工维护
- 让项目看板承接范围内的任务推进

这个阶段的关键词是：**从单个任务上升到协作目标。**

### 5.5 AI 作为可选执行方式介入

当某类任务适合交给 agent 处理时：

- 团队可以把 issue 指派给 agent
- agent 通过 runtime / daemon 执行
- skill 提供可复用的操作说明

此时 AI 参与的是执行方式，而不是另起一套工作流。

### 5.6 最终形成闭环

这套流程最终形成的闭环是：

**任务定义工作，时间记录投入，项目组织目标，AI 在需要时加速执行。**

## 6. 系统视角

### 6.1 前端视角

当前前端是一个以 workspace 为边界的 SPA，主要页面已经清晰分成三类：

- **任务类**，Issues、Board、Backlog、Today、Upcoming、My Work、Calendar
- **时间类**，My Time、My Time Calendar
- **项目类**，Projects、Project Board

AI 相关页面仍存在：

- Agents
- Runtimes
- Skills

但它们不再构成主要导航的唯一解释中心。

### 6.2 后端视角

后端继续承担：

- 任务与项目的数据持久化
- 时间记录 API
- 实时同步
- AI 运行调度相关接口

也就是说，产品主线换了，但后端并没有变成另一套系统，而是在原有能力上重新排序。

### 6.3 实时与状态同步

当前产品仍保留实时同步能力。  
这对于下面几类场景仍然重要：

- 任务状态变更
- 评论与协作更新
- AI 执行进度反馈
- 时间相关视图刷新

实时能力现在不是 AI 专属，它服务的是整个执行系统。

### 6.4 CLI / Daemon 在新结构里的角色

CLI 与 daemon 仍然有价值，但它们主要属于 AI 接入层：

- 接入本地机器作为 runtime
- 管理 agent 执行环境
- 连接平台与本地 CLI

对不使用 AI 执行的团队来说，核心产品价值仍然成立。  
这正是新结构相对于旧结构的一个重要变化。

## 8. 迁移说明：旧产品全景到新产品全景

这一章专门说明旧文档里的逻辑，怎样映射到新文档。

### 8.1 被提升为主轴的能力

以下能力在新文档里被提升：

- **Issue / 任务管理**
- **Time Tracking / 时间记录**
- **Project / 项目协作**

原因是它们更直接定义日常使用路径，也是产品稳定性的来源。

### 8.2 被降级为增强层的能力

以下能力仍重要，但从产品总纲降为增强层：

- **Agent**
- **Runtime / Daemon**
- **Skill**

这不是否定它们的价值，而是重新定义它们的角色：

- 从“产品本体”改为“执行增强”
- 从“世界观入口”改为“可选加速器”

### 8.3 被延后处理的能力

以下方向本次不纳入一级结构：

- **知识管理**

原因不是无关，而是当前定义尚不稳定。  
在产品还没有形成清晰边界之前，把它写成一级能力，只会让文档提前失真。

### 8.4 文档结构本身的变化

旧文档更像“功能百科”，它对每个概念、每个表、每个系统角色都做了细分。  
新文档更像“产品地图”，它更强调：

- 哪些能力是主干
- 哪些能力是支撑
- 这些能力怎样串成日常工作流

这也是这次重写真正要完成的事。

## 9. 附录：关键对象速查

### 9.1 产品主干对象

| 对象 | 关注的问题 | 对应价值 |
|------|------------|----------|
| **Issue** | 做什么 | 定义执行单元 |
| **Time Entry** | 花了多少时间 | 记录投入 |
| **Project** | 这些任务服务于什么目标 | 聚合协作范围 |

### 9.2 执行参与者

| 对象 | 含义 |
|------|------|
| **Member** | 人类成员 |
| **Agent** | AI 成员，属于增强执行层 |
| **Assignee** | 任务当前负责人，可以是人，也可以是 agent |

### 9.3 AI 增强层对象

| 对象 | 含义 |
|------|------|
| **Runtime** | agent 的执行环境 |
| **Daemon** | 本地接入 runtime 的后台进程 |
| **Skill** | 可复用的 agent 工作说明 |
| **Workspace Context** | 工作区级共享 AI 上下文 |

### 9.4 一句话总结产品结构

如果要用一句最短的话解释现在的 Multica，可以这样说：

> **Multica 先是一个任务管理、时间记录、项目协作一体化的团队执行系统，然后才是一个可把 AI 纳入执行流程的平台。**
