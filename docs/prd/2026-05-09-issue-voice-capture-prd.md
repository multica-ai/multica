# Issue Voice Capture PRD

## 1. 文档目标

这份 PRD 用来定义“在创建 issue 时支持语音输入与语音转录”的产品目标、用户价值、范围边界、阶段计划和验收标准，为后续实现与评审提供统一依据。

该功能的目标不是把 Multica 改造成语音助手，而是把 **创建 issue 的记录成本进一步压低**，让用户能更快把想法、待办、结论和阻塞沉淀进现有 issue 模型。

## 2. 背景与问题

### 2.1 当前问题

当前的 issue 创建体验已经具备较完整的文本录入能力，但对“快速捕获”仍不够友好。

现状证据：

- `apps/workspace/src/features/modals/create-issue.tsx#CreateIssueModal`：当前创建入口是标题 + 描述 + 属性 + 文件上传，没有语音入口。
- `apps/workspace/src/shared/hooks/use-file-upload.ts#useFileUpload`：系统已具备上传基础设施，意味着“保留原始录音”并非从零开始。
- `server/internal/handler/file.go#UploadFile`：上传接口已接受音频文件。
- `server/internal/handler/ai.go#buildLLMClient`：当前 AI 配置只覆盖文本类能力，不足以直接承接语音转录。

这带来几个典型痛点：

1. 临时想法出现时，手动输入太慢
2. 开会、走路、切换上下文时，打字不是最低成本的 capture 方式
3. 语音材料如果保存在外部工具，就会脱离 issue 主对象
4. 纯文本录入对“先记下来再整理”的场景不够友好

### 2.2 为什么现在做

这个需求和当前产品方向高度一致：

- 它强化了 **任务优先**，因为最终仍然落在 issue 上
- 它强化了 **个人执行闭环**，因为降低了记录成本
- 它保留了 **AI / agent 的正确位置**，因为第一版只解决 capture，不把问题泛化成聊天产品

## 3. 产品目标

### 3.1 目标

1. 让用户可以通过语音更快创建 issue
2. 让语音内容可以直接转成可编辑文本，而不是仅保存一段音频
3. 让用户在需要时保留原始录音，作为 issue 的可追溯材料
4. 让该能力自然挂在现有 create issue 流程中，而不是新增平行对象模型

### 3.2 非目标

1. 不做独立的语音 inbox 或语音任务系统
2. 不在第一版支持 comment、reply、project、bulk import 的语音录入
3. 不在第一版做语音摘要、语义拆分、自动打标签、自动排期
4. 不在第一版承诺音频播放器体验
5. 不把现有文本 AI 设置直接视为语音转录能力

## 4. 目标用户与典型场景

### 4.1 目标用户

- 正在工作流中，需要快速记一条任务的人
- 在会议、走动、切换窗口时，不方便打字的人
- 希望先把内容记录下来，再稍后编辑 title / description 的人

### 4.2 核心场景

#### 场景 A：临时待办快速捕获

用户突然想到一件要跟进的事情，打开新建 issue，点麦克风，说出标题和几句说明，完成后稍微检查一下文本就创建。

#### 场景 B：会议后快速落任务

用户在会议结束时，用语音快速说出结论和动作项，让系统直接变成 issue 草稿，而不需要先记在外部语音工具里。

#### 场景 C：不方便打字但需要完整上下文

用户已经输入了 issue 标题，但描述较长，适合直接说出来，让 transcript 追加到 description 中。

#### 场景 D：需要保留原始表达

用户希望保留录音本身，后续可以回听原始语境，因此在创建 issue 时一并保存音频附件。

## 5. 解决方案概览

### 5.1 核心方案

在 `CreateIssueModal` 中新增一个语音输入入口，支持：

- 录音
- 转录
- 把 transcript 写入当前 issue 草稿
- 可选保留原始录音

### 5.2 设计原则

1. **Issue-first**：最终仍然创建和编辑 issue，不新建平行 capture 对象
2. **Human-in-the-loop**：语音只是输入方式，用户仍要确认最终文本
3. **Capability-aware**：根据浏览器能力决定体验，不承诺所有环境一模一样
4. **Phased delivery**：先做浏览器原生能力，再做服务端转录增强

## 6. 功能范围

### 6.1 Phase 1：浏览器原生语音录入 MVP

包含：

- Create Issue 弹窗中的麦克风入口
- 浏览器原生录音
- 浏览器原生语音识别 / 转录（若浏览器支持）
- transcript 写入 title / description 草稿
- 用户确认 transcript 后再写回表单
- 可选保留原始录音；issue 创建成功后上传为 issue attachment
- 明确的错误提示、权限提示和浏览器能力提示

不包含：

- 服务端转录
- 跨浏览器统一质量保证
- 音频播放器
- 其他输入面的复用

### 6.2 Phase 2：服务端转录增强

包含：

- 在浏览器原生转录不可用时，支持音频上传后服务端转录
- 新增独立 transcription provider 配置
- 更稳定的跨浏览器 fallback

不包含：

- 自动摘要成结构化 issue
- 自动拆解 subtasks
- 和 schedule / label suggestion 的自动联动

## 7. 详细需求

### 7.1 入口与交互

1. 在 create issue 弹窗中提供明显但不过度抢占的麦克风按钮
2. 点击后触发麦克风权限请求
3. 录音中要有清晰状态反馈：正在录音、已停止、正在整理文本、可确认插入
4. 用户必须能在文本写回前先看到 transcript 结果

### 7.2 文本写回规则

#### 规则 A：标题为空

- transcript 的第一句或第一行作为标题候选
- transcript 全量进入 description
- 用户可手动修改

#### 规则 B：标题已有内容

- 不覆盖标题
- transcript 全量追加到 description

#### 规则 C：标题候选无效

- 若 transcript 太短或不适合作为标题，不自动强行覆盖标题
- 保持用户手动确认空间

### 7.3 原始录音保留

1. 用户可选择是否保留原始录音
2. 若选择保留，录音在 issue 创建成功后上传为 issue attachment
3. 若录音上传失败，不回滚已创建的 issue
4. 需要明确向用户说明“文本已保存，但录音未保存”

### 7.4 兼容与降级

1. 若浏览器支持录音和转录，提供完整体验
2. 若仅支持录音，不隐瞒限制，要明确告知当前无法实时转录
3. 若完全不支持，禁用语音入口但不影响手动创建 issue

### 7.5 权限与隐私

1. 麦克风权限必须由显式点击触发
2. 在用户未确认提交前，不自动上传录音到服务器
3. 若未来启用服务端转录，要明确说明音频会发送到外部 provider

## 8. UX 细节要求

### 8.1 状态机

- Idle
- Requesting permission
- Listening
- Transcribing
- Review
- Uploading audio
- Error
- Unsupported

### 8.2 文案要求

需要有清晰文案表达：

- 正在录音
- 正在转录
- 当前浏览器不支持实时转录
- 录音将作为附件保存
- issue 已创建，但录音上传失败

### 8.3 失败恢复

1. 权限拒绝后允许用户继续手动输入
2. transcript 失败不清空已经录到的文本或草稿
3. 创建 issue 失败时不应丢失当前 transcript 草稿

## 9. 技术策略

### 9.1 Phase 1

- 前端基于浏览器媒体与语音识别能力实现
- 后端仅复用现有 issue create 和 file upload API
- 不新增数据库字段

### 9.2 Phase 2

- 新增服务端转录接口
- 配置模型独立于现有 chat completion 设置
- 不把现有 DeepSeek / chat 模型直接视为转录能力

## 10. 成功指标

### 10.1 产品指标

1. 使用语音入口创建 issue 的用户占比
2. 语音创建 issue 的平均耗时，相比纯手动输入是否下降
3. 语音创建后被立即编辑的比例
4. 录音保留功能的开启比例

### 10.2 质量指标

1. 语音入口权限失败率
2. 转录失败率
3. issue 创建成功但录音上传失败率
4. 浏览器不支持导致功能不可用的占比

## 11. 风险与约束

1. 浏览器原生转录兼容性不一致，MVP 的稳定性依赖用户环境
2. 没有音频播放器时，保留录音更像“附件留存”而不是“即时回放体验”
3. 标题映射规则是确定性规则，可能仍需用户手动修正
4. 若把文本 AI 配置和语音转录配置混用，后续 provider 能力解释会混乱

## 12. 发布建议

### 12.1 发布顺序

1. 先上线 Phase 1 到 create issue 弹窗
2. 收集浏览器兼容性与真实使用反馈
3. 再决定是否推进 Phase 2 服务端转录

### 12.2 验收标准

MVP 上线前至少满足：

1. 用户可在 create issue 中完成一次完整的语音录入
2. transcript 能写回 title / description 草稿
3. 用户可选保留原始录音
4. issue 创建成功后录音可作为 attachment 保存
5. 浏览器不支持时有明确降级提示

## 13. 后续扩展

后续可以评估但不应阻塞当前版本的方向：

1. 把相同能力扩展到 comment composer
2. 音频附件内联播放器
3. transcript 的 AI 标题优化
4. 语音转录后自动建议标签 / 排期
5. 个人 nightly review capture 的语音入口
