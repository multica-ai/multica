1.3 接口层错误码定义

错误码 HTTP 说明 处理建议
INVALID_REQUEST400请求体非法、字段缺失、路径参数或 UUID 格式错误。修正入参与请求格式后重试。
UNAUTHORIZED401登录态、PAT、Daemon Token 或 Task Token 无效或过期。重新登录或重新签发令牌。
FORBIDDEN403工作区成员身份不足、角色不匹配、调用主体受限。检查角色、工作区归属和调用来源。
NOT_FOUND404工作区、议题、Agent、Runtime、Task 等目标不存在。刷新上下文并重新获取对象。
CONFLICT409资源冲突、重复操作或当前状态不允许执行目标动作。读取最新状态后再次提交。
VALIDATION_FAILED422业务校验失败，例如模板导入失败、组合参数不合法。根据返回 message 修正业务输入。
DEPENDENCY_UNAVAILABLE503Cloud Runtime、Composio、Lark、Slack 等依赖未配置或不可用。补齐依赖配置或走降级方案。

关系分组 关系说明
租户与身份 workspace 是多租户边界；user 通过 member 加入工作区并获得角色。
Agent 执行域 agent_runtime 是 Agent 的实际执行环境；agent 绑定运行时并通过 agent_skill 关联工作区技能。

4状态机

4.1 议题状态机

议题状态机表达工作项从创建、处理中、受阻到完成/取消的生命周期。Agent 执行结果不会直接替代议题状态，而是通过评论、任务结果和人工操作间接推动状态变化。

stateDiagram-v2 [*] --> todo todo --> in_progress: 开始处理 in_progress --> blocked: 遇到阻塞 blocked --> in_progress: 阻塞解除 in_progress --> completed: 完成处理 todo --> cancelled: 主动取消 in_progress --> cancelled: 主动取消 blocked --> cancelled: 主动取消 completed --> [*] cancelled --> [*]

4.2 Agent 任务状态机

任务状态机覆盖 issue、chat、autopilot 三类来源的统一执行生命周期，是 daemon 调度与运行结果回传的核心状态机。

stateDiagram-v2 [*] --> queued queued --> dispatched: runtime 原子认领 dispatched --> waiting_local_directory: 本地目录冲突 waiting_local_directory --> dispatched: 资源释放后重新调度 dispatched --> running: daemon 上报 start running --> completed: 上报 complete（一锁二判三更新） running --> failed: 上报 fail（一锁二判三更新） queued --> cancelled: 用户/系统取消 dispatched --> cancelled: 用户/系统取消 running --> cancelled: 用户/系统取消 completed --> [*] failed --> [*] cancelled --> [*]

4.3 Runtime 在线状态机

Runtime 在线状态机由 daemon 注册、heartbeat、Redis/DB liveness 联合驱动，用于表示运行时是否可接单以及是否稳定。

stateDiagram-v2 [*] --> offline offline --> idle: register + heartbeat 成功 idle --> working: 认领并启动任务 working --> idle: 任务全部完成 idle --> unstable: 心跳/探测异常 working --> unstable: 运行抖动 unstable --> idle: 恢复正常 unstable --> working: 恢复但仍有执行任务 idle --> offline: 长时间未心跳 working --> offline: 长时间未心跳 unstable --> offline: 长时间未心跳 offline --> archived: 删除/归档 archived --> [*]

4.4 Autopilot 运行状态机

Autopilot run 状态机记录每次自动化触发的执行结果。run_only 与 create_issue 两种模式共用同一状态流转。

7.1 系统级风险判断

改动对象 修改前 修改后 影响范围
系统设计文档对现网能力的覆盖视角 文档中尚未定义 系统是“多租户协作 + 本地 daemon 执行 + 外部集成扩展”的复合架构；最高风险区集中在工作区隔离、任务状态一致性、runtime 在线判断、聊天与任务联动、自动化触发链路；第三方集成通常可降级但会影响扩展体验。 回归测试优先级排序、发布核查重点、线上告警分层

7.2 建议回归优先级

优先级 回归范围 说明
P0登录、工作区隔离、issue 主链路、daemon 执行、chat 主链路、autopilot 触发系统核心生存链路，任何一项异常都会直接影响主业务可用性。
P1skill/agent 配置、realtime、inbox、notification、附件影响日常协作效率和感知，但通常不阻断所有主流程。
P2GitHub/Lark/Slack/Composio、cloud runtime、billing、dashboard扩展能力或统计能力，失败需可观测、可降级，但不应影响核心 issue/agent/chat 运转。

文档生成方式本文档不是基于单一 PRD，而是基于现有仓库代码、数据库迁移、路由定义、共享类型和产品全景文档逆向整理生成。
