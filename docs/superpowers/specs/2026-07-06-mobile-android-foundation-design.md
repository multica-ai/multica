# Mobile Android Foundation — Design

Date: 2026-07-06
Status: Approved (sub-project 1 of 2)

## Context

`apps/mobile` is an Expo/React Native app that currently targets iOS only:
no `android` block in `app.config.ts`, no `android` scripts in
`package.json`, no `android/` native directory, no `eas.json`. Several UI
call sites use iOS-only APIs.

This is sub-project 1 of a two-part effort to add Android support:

1. **This spec** — get the app building and running on a connected Android
   device/emulator with no crashing interactions.
2. **Follow-up spec (separate brainstorm)** — migrate the ~24 `formSheet`
   routes to `@gorhom/bottom-sheet` for iOS/Android visual parity. Depends
   on this spec (needs a working Android build to verify against).

## Goal

`expo run:android` builds and launches the app on a physically connected
Android device (three are currently attached and visible via `adb
devices`). Every existing interaction — including the 8 screens that call
`ActionSheetIOS` — works without crashing or silently no-opping on
Android. Visual parity with iOS (bottom-sheet look, adaptive icon
artwork, status bar polish) is explicitly out of scope for this spec.

## Non-goals

- Play Store submission (signing, store metadata, EAS Android build
  profile).
- Bottom-sheet / formSheet visual parity (~24 routes) — sub-project 2.
- Custom adaptive icon artwork (foreground/background layers) — Expo's
  default single-icon fallback is accepted for this phase.
- Status bar / edge-to-edge / navigation-bar theming polish.

## Changes

### 1. `apps/mobile/app.config.ts` — add `android` block

```ts
android: {
  package: isProd
    ? "ai.multica.mobile"
    : isStaging
      ? "ai.multica.mobile.staging"
      : "ai.multica.mobile.dev",
},
```

Unlike iOS's `bundleIdentifier`, Android's `package` needs no
`EXPO_*_DEV/PROD` env-var escape hatch: Apple requires the local signing
identity to own the bundle prefix even for on-device installs, which is
why iOS carries that override. Android has no equivalent restriction —
any package name can be installed on any device via `expo run:android`
with no account ownership check. Hardcoding the three package names is
the correct simplification, not a gap.

No `android.adaptiveIcon` entry is added — Expo falls back to the
top-level `icon` field, matching current iOS behavior. The system will
apply its own mask (circle/squircle/rounded square depending on OEM
launcher), which is visually imperfect but not a blocker.

### 2. `apps/mobile/package.json` — add android scripts

Mirrors `ios` / `ios:staging` / `ios:prod`. No `ios:device`-style variant
is needed: which physical device or emulator receives the build is
purely a function of what's connected when the script runs, not a
separate code path.

```json
"android": "expo run:android",
"android:staging": "dotenv -e .env.staging -- cross-env APP_ENV=staging expo run:android",
"android:prod": "dotenv -e .env.production -- cross-env APP_ENV=production expo run:android",
```

### 3. Root `.gitignore` — ignore Expo-generated native dirs

Neither `apps/mobile/ios/` nor `apps/mobile/android/` currently exist on
disk in this checkout, and the root `.gitignore` has no rule for either.
Running `expo run:android` generates a full native Gradle project (~hundreds
of files) under `apps/mobile/android/`; without an ignore rule these show
as untracked and risk being swept into a commit by an incautious `git add
-A`. Add, with a comment explaining why:

```
# Expo continuous native generation (regenerated by `expo prebuild` / `expo run:*`)
apps/mobile/android/
apps/mobile/ios/
```

### 4. `ActionSheetIOS` → `@expo/react-native-action-sheet` (8 call sites)

New dependency: `@expo/react-native-action-sheet` (Expo-maintained,
purpose-built for this exact cross-platform gap — native-feeling sheet on
both platforms, not a generic third-party UI kit).

- Wrap the app root (`apps/mobile/app/_layout.tsx`) in
  `<ActionSheetProvider>`.
- Replace `ActionSheetIOS.showActionSheetWithOptions(...)` with
  `useActionSheet().showActionSheetWithOptions(...)` in all 8 call sites
  (all confirmed to be React function components, so the hook is legal
  there):
  - `app/(app)/[workspace]/(tabs)/inbox.tsx`
  - `app/(app)/[workspace]/issue/[id].tsx`
  - `app/(app)/[workspace]/project/[id].tsx`
  - `app/(app)/[workspace]/more/settings/profile.tsx`
  - `components/chat/message-long-press.tsx`
  - `components/chat/chat-message-list.tsx`
  - `components/issue/comment-card.tsx`
  - `components/issue/comment-context-menu.tsx`
- Parameter shape (`options`, `destructiveButtonIndex`,
  `cancelButtonIndex`, callback signature) is compatible with the
  existing `ActionSheetIOS` call shape at each site — this is a
  mechanical swap, not a logic change.
- Add `@expo/react-native-action-sheet` to the "Tech-stack baseline" list
  in `apps/mobile/CLAUDE.md` per that file's own rule ("Add to this list
  when actually adopted").

### 5. Local machine setup (not committed — one-time, user-confirmed)

`ANDROID_HOME` is currently unset in this shell even though the SDK is
installed at `~/Library/Android/sdk` (confirmed via `adb`, which only
works today because `platform-tools` happens to be on `PATH`). `expo run:android`
needs `ANDROID_HOME` to locate the SDK. Proposed addition to
`~/.zshrc` (or `~/.zprofile`), executed only after explicit user
confirmation since it changes global shell config outside the repo:

```bash
export ANDROID_HOME="$HOME/Library/Android/sdk"
export PATH="$PATH:$ANDROID_HOME/platform-tools:$ANDROID_HOME/emulator"
```

## Follow-up (sub-project 2, separate spec)

Migrate the ~24 `formSheet`-presentation routes (registered via the
shared `SHEET_OPTIONS` constant in
`app/(app)/[workspace]/_layout.tsx`) to `@gorhom/bottom-sheet` so Android
gets the same rounded-corner / grabber / spring-physics sheet behavior
iOS gets natively from `UISheetPresentationController`. This is a real
architectural change to the sheet system (route-based → component-based)
and depends on this spec's Android build being in place to verify
against. Brainstorm separately.
