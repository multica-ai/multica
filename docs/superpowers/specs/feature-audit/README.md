# 功能审计工作区

## 目标

这个目录用于持久化 `docs/功能列表清单.md` 与当前仓库实现之间的对齐结果。

它的设计目标是支持多个 Copilot CLI 会话或 smux 窗口并行审计，并且在 session 中断后仍然可以从仓库文档继续接力。

## 审计规则

### 完成状态口径

- `已完成`：该能力已端到端实现，并且有代码证据支撑
- `部分完成`：已有部分实现，但范围偏窄、缺关键流程，或者只覆盖部分条目
- `缺失`：当前仓库中没有找到实现证据
- `blocked`：在缺少产品决策或底层能力前，当前无法合理判断或推进

### 完成度计算

- 完成度按三级条目等权统计
- 三级条目汇总到二级能力
- 二级能力再汇总到一级模块

### 证据要求

每个结论都必须包含：

1. 文件路径
2. 组件、hook、handler、query 或函数名
3. 该证据为什么能支撑该结论的简短说明

如果搜索为空，必须记录搜索过的关键词，并明确写出“未找到匹配”，不能凭印象推断。

## 接力规则

每个一级模块概览和二级能力 spec 都应该持续维护这些字段：

- 当前状态
- 完成度
- 证据列表
- 已确认缺口
- 下一步实现切片
- handoff 说明

后续会话继续推进时，应该优先从对应模块的 `overview.md` 和二级 `spec.md` 继续，而不是依赖聊天上下文。
