# 单能力 Design

## 目标

- 对 7.1 给出明确产品边界判断。
- 若未来重启，提前定义正确的能力落点与进入条件。

## 非目标

- 不在当前阶段实现屏蔽列表。
- 不承诺纯前端即可完成网站/应用阻断。
- 不让执行 Agent 自行补 system daemon 设计。

## 当前架构基线

- 当前入口：无。
- 当前核心逻辑：无。
- 当前存储或状态：无。
- 当前 UI 或接口：无。

### 代码证据

- `apps/workspace/src/router.tsx` `routeTree`：当前没有 blocklist 入口。
- `apps/workspace/src`、`server` `rg(blocklist|website block|app block|custom block rule)`：代码无实现。
- `product-overview.md` `当前阶段的项目目标与展望`：当前产品主轴不包含该能力。

## 缺口定义

- 7.1 缺的不只是页面或表结构，而是完整的运行时能力、权限模型与产品确认。
- 在没有 daemon / OS 权限模型时，屏蔽列表无法闭环。

## 方案与权衡

### 方案 A：浏览器内做一个“网站清单”页面

- 做法：只在前端保存黑名单，不真正拦截。
- 优点：实现快。
- 风险：无法兑现“屏蔽列表”承诺，属于伪能力。

### 方案 B：设计 daemon 管理规则并执行系统级阻断

- 做法：新增规则模型、后台守护进程、平台适配与权限申请。
- 优点：能力闭环。
- 风险：复杂度、隐私和平台兼容成本极高。

### 方案 C：当前阶段挂起，仅保留进入条件，推荐

- 做法：把 7.1 明确标为低优先级、非当前阶段目标；若未来重启，再以 daemon 方案为唯一可行方向。
- 优点：与仓库现状一致，避免伪实现。
- 风险：短期内不会交付此能力，但这是真实边界。

## 推荐方案

选择方案 C。

7.1 当前应被明确视为 **P3 / 非当前阶段目标**。如果未来需要重启，不应以浏览器假动作实现，而应直接进入“规则模型 + daemon 执行 + 权限申请”的系统能力设计。

## 数据模型或状态模型

```text
future blockRule
├─ id
├─ target_type(domain | application)
├─ target_value
├─ schedule
├─ enabled
└─ source(workspace | user)
```

- 当前阶段不创建该模型。
- 若未来重启，必须先决定规则 owner 是 workspace 共享还是用户个人。

## 接口契约

### 输入

- 当前阶段无输入接口。
- 未来若重启，输入至少包括阻断目标、时段、启停状态与作用域。

### 输出

- 当前阶段输出仅为“挂起决策”。
- 未来输出应由 daemon 执行结果、失败原因与状态同步组成。

## UI 或交互流程

### 页面交互流

```text
当前阶段：无页面

未来若重启：
Settings / Utilities
  -> Blocklist
  -> 编辑规则
  -> 提交给 daemon
  -> daemon 应用系统级阻断
```

### 状态机

```text
[hypothesis]
   -> [product-approved]
   -> [runtime-designed]
   -> [implementable]
```

### 数据变化流

```text
future UI
   -> blockRule API
   -> daemon runtime
   -> OS / browser enforcement
   -> status callback
```

## 权限、边界条件、异常路径

- 谁可以使用：当前阶段无人可用。
- 哪些输入非法：任何企图只用前端假列表声称“已支持屏蔽”的方案都视为越界。
- 失败时如何处理：若后续发现平台权限不足，应在设计层先做 capability downgrade，而不是静默失效。

## 实现约束

- 不要在没有运行时的情况下创建伪 blocklist UI。
- 不要跳过产品确认门。
- 若未来重启，必须先明确 daemon 与权限策略。

## 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 伪实现 | 用户得到错误预期 | 当前阶段明确挂起 |
| 运行时权限不足 | 跨平台不可用 | 未来设计时先验证平台能力 |
| workspace / user 归属不清 | 规则共享边界混乱 | 未来重启前先定义 owner |

1. 7.1 在文档中被明确标成 P3 / 非当前阶段目标。
2. 文档说明未来若重启，唯一可行方向是引入运行时能力。
3. 不出现“先做页面以后再接线”的伪计划。
