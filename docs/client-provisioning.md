# Client Provisioning Guide

This document describes how to spin up a new isolated 3J Tracker instance for a client.

## Quick start

```bash
make new-client
```

The interactive script will:

1. Ask for a **client slug** (short name, e.g. `acme`)
2. Ask for **allowed emails** — the initial list of people who can log in
3. Generate strong random secrets (`JWT_SECRET`, `POSTGRES_PASSWORD`)
4. Write a ready-to-deploy `deploy/<slug>.env` file

The generated config locks down sign-ups and workspace creation so only the people you list can access the instance.

## Prerequisites

- Docker + Docker Compose v2
- `openssl` (available on macOS, Linux, and WSL2)
- A [Resend](https://resend.com) account for outbound email (OTP login + invitations)

## Step-by-step

### 1. Generate the env file

```bash
make new-client
# Prompts:
#   Client slug: acme
#   Allowed emails: alice@acme.com,bob@acme.com
#   Resend API key: re_xxxx
#   From email: [noreply@tracker.3jtech.app]
#   Public frontend URL: https://tracker.acme.com
```

This writes `deploy/acme.env` (mode 600 — secrets only readable by owner).

### 2. Transfer to server

```bash
scp deploy/acme.env your-server:/opt/3j-tracker/acme/.env
```

### 3. Start the stack

```bash
ssh your-server
cd /opt/3j-tracker/acme
docker compose -f /path/to/docker-compose.selfhost.yml --env-file .env up -d
```

Check backend logs to confirm email is working:

```bash
docker compose logs -f backend | grep EmailService
# Should print: EmailService: Resend API from=noreply@tracker.3jtech.app
```

### 4. First login

Log in with one of the `ALLOWED_EMAILS` addresses. An OTP will be sent by email. On first login you will be prompted to create a workspace — this is the **only** workspace creation that can happen (subsequent creation is blocked by `DISABLE_WORKSPACE_CREATION=true`).

### 5. Invite more team members

From **Settings → Members → Invite**, invite additional users by email. They do **not** need to be in `ALLOWED_EMAILS` once already invited — the invite flow bypasses the signup guard.

To also allow them to log in before accepting an invite, add their email to `ALLOWED_EMAILS` in `.env` and restart the backend:

```bash
docker compose restart backend
```

## Access control variables

| Variable | Default | Description |
|---|---|---|
| `ALLOW_SIGNUP` | `false` | When `false`, only emails matching `ALLOWED_EMAILS` or `ALLOWED_EMAIL_DOMAINS` can receive an OTP |
| `ALLOWED_EMAILS` | `""` | Comma-separated list of allowed email addresses |
| `ALLOWED_EMAIL_DOMAINS` | `""` | Comma-separated list of allowed domains (e.g. `acme.com`) |
| `DISABLE_WORKSPACE_CREATION` | `true` | When `true`, only the first workspace (created on first login) is allowed |

## Email configuration

Set **one** of the following:

### Option A — Resend (recommended)

```env
RESEND_API_KEY=re_xxxx
RESEND_FROM_EMAIL=noreply@tracker.yourdomain.com
```

You must verify your sending domain in the [Resend dashboard](https://resend.com/domains) before emails will deliver.

### Option B — SMTP

```env
SMTP_HOST=smtp.yourdomain.com
SMTP_PORT=587
SMTP_USERNAME=user@yourdomain.com
SMTP_PASSWORD=password
SMTP_TLS=starttls
```

For implicit TLS (port 465) set `SMTP_TLS=implicit`.

## Backup

To enable daily automated backups, add the backup sidecar:

```bash
# Add a cron job on the host (example: 2am UTC daily)
0 2 * * * docker compose \
    -f /opt/3j-tracker/docker-compose.selfhost.yml \
    -f /opt/3j-tracker/docker-compose.backup.yml \
    --env-file /opt/3j-tracker/acme/.env \
    run --rm backup
```

Backups are stored in the `multica-v1-backups` Docker volume as gzip-compressed SQL dumps, retained for 7 days (configurable via `BACKUP_RETAIN_DAYS`).

To restore:

```bash
# Copy the dump out of the volume
docker run --rm \
    -v multica-v1-backups:/backups \
    -v $(pwd):/out \
    alpine cp /backups/tracker_20260603_020000.sql.gz /out/

# Restore to a running instance
gunzip -c tracker_20260603_020000.sql.gz \
    | docker exec -i multica-v1-db \
        psql -U multica -d multica
```

## Security hardening checklist

- [ ] `JWT_SECRET` is a 64-character hex string (generated by `make new-client`)
- [ ] `POSTGRES_PASSWORD` is random and not shared with other instances
- [ ] `APP_ENV=production` (enforced — server refuses to start without email if set)
- [ ] `ALLOW_SIGNUP=false`
- [ ] `DISABLE_WORKSPACE_CREATION=true`
- [ ] Ports bound to `127.0.0.1` only — reverse proxy (Caddy / nginx / Cloudflare Tunnel) handles TLS
- [ ] `RESEND_API_KEY` or `SMTP_HOST` set — server refuses to start in production without one
- [ ] Daily backup cron installed and tested

## Locking signups on an existing instance

If an instance was started with `ALLOW_SIGNUP=true` and you want to lock it:

1. Edit `.env`:
   ```env
   ALLOW_SIGNUP=false
   ALLOWED_EMAILS=alice@acme.com,bob@acme.com
   ```
2. Restart the backend:
   ```bash
   docker compose restart backend
   ```
