Grounding Context

Multica 中"task run"是 agent_task_queue 的一行；它与 issue comment 之间存在两条方向相反的单向链接，但都不是"一条 comment 对应一个 run"的简单关系。本次接地调查（一次 codebase 扫描 + 5 个 evidence scout + 一次外部研究）揭示了几个关键事实，回答用户的前提：

  • run 并不天然对应"一条"comment。存在两条链：agent_task_queue.trigger_comment_id（带 FK，ON DELETE SET NULL）指向"触发该 run 的 comment"；comment.source_task_id（无 FK，应用层解析）指向"产出该 comment 的 run"。
  • 最常见的产物恰好没盖戳。task.go:1827 的合成成功回退评论传的是空 UUID，而 :1983 的失败系统评论才传 task.ID。issue_child_done.go:303 同样未盖。→ 任何"从 comment 回到 run"的 UI 在最常见的成功评论上失明。
  • 三种 run 根本不产生 issue comment：chat（写 chat_message）、quick_create（仅 inbox）、squad-leader no_action（仅 activity_log）。→ 用户提问的"每次 run"对这些 kind 不成立。
  • 当前唯一的评论侧 run 入口是失败按钮。TaskCommentRetryButton 的守卫 retryableAgentFailureComment 要求 actor_type==="agent" AND comment_type==="system" AND source_task_id 非空——只有失败评论能进入。
  • TranscriptButton 与 AgentTranscriptDialog 已与侧边栏解耦。自门控、纯展示、自管 dialog 状态——可在评论内部直接复用，零后端改动。
  • 时间线本就异构。useIssueTimeline 把 activity 和 comment 交错到 TimelineEntry[]，TimelineItem 联合类型当前三种 kind；新增 kind 在结构上受支持。
  • 外部行业共识：人类/对话式 run 进线程（Copilot @mention、Devin Slack、Linear @Linear），计划/事件驱动自动化进独立流（Jira rules、Linear triage、GitHub Actions）。GitHub Checks 是最强先例——侧栏保全量历史 + 评论放 details_url 风格的 provenance 链接。

Topic Axes

  1. linkage-data-model — run↔comment 的身份接缝（trigger_comment_id / source_task_id），覆盖缺口，kind 派生而非存储，哪些 run 真的产生 comment。
  2. comment-placement — 入口在评论 UI 哪个位置（正文页脚槽位、头部元数据区、作为新的 typed timeline 条目）。
  3. run-detail-reveal — 详情如何展示（复用 TranscriptButton/AgentTranscriptDialog、内联 Collapsible、hover card、Drawer、新建 run 路由）。
  4. run-scope-conditioning — 哪些 run 值得挂内联链接（issue run vs chat/quick_create/no_action；失败 vs 合成成功 vs CLI 执行评论）。
  5. sidebar-inline-bridge — 侧边栏与评论内联的关系（hybrid provenance、双向 coverage、镜像 TaskCommentCoverage、删除或降级侧边栏）。

4. GitHub Checks 混合设计方向 — 行业验证的双轨架构

Downsides

  • 这是方向性决策而非具体功能，需团队对"不要把 run 全搬进评论"达成共识。
  • 要求所有具体实施都尊重双轨边界——未来若有人想"全进时间线"，会被方向性拒绝。
  • 侧边栏的次要地位意味着其 UI 质量不能被忽视，否则仍会失修。

6. 双向 TaskCommentCoverage — 桥的薄变厚

sidebar-inline-bridge85%Medium

Description：侧边栏的 TaskCommentCoverage（execution-log-section.tsx:482-508）计算 run 覆盖了多少评论（"Includes N comments"），但评论侧没有任何反向信号。添加一个镜像 chip："属于一次包含 N 条评论的 run"（当 run 涵盖 >1 条评论时），点击滚到侧边栏中的该 run 并高亮。关闭循环：侧边栏和评论时间线成为同一个 run↔comment 图的两个视图，每个都可导航到另一个。

双向桥：实线 = TaskCommentCoverage（sidebar 已存在）；虚线 = 镜像 chip + 跨导航（新增）。

Downsides

  • 跨表面导航（评论 → 侧栏滚动/高亮）是新的交互模式，需用户体验测试。
  • 评论侧 chip 的视觉密度需控制，避免在长线程评论中视觉噪音。
  • 反向 chip 仅在 run 涵盖 >1 条评论时显示（与现有 chip 隐藏条件对称）—— 信息密度反而低于 sidebar。

这是 run ↔ comment 关系的根本反转：不再是"给评论加 run 入口"，而是"run 本就在时间线里"。一旦 run 在时间线中内联，侧边栏的作用减弱，覆盖率 chip 变成连接相邻时间线条目的桥梁，"查看详情"的受众范围也扩大了——同时为三种无评论 run 提供叙事位置（之前在时间线完全隐形）。侧栏不必删除——降级为摘要即契合 GitHub Checks 混合模式（Idea 4）。这是一个杠杆性的结构改动：一种新 TimelineItem kind 同时解决 comment-attached run 嵌套、no-comment run 显形、kind 条件化三个问题。

Downsides

  • 这是最大的结构改动之一——信息架构层级反转。需团队对"时间线是单一事实源"达成共识。
  • 三类 run（chat/quick_create/no_action）的时间线条目会让部分 issue 的时间线显得密集——需视觉密度控制。
  • depends on Phase 3 of C1（kind 持久化）才能稳定 gate；未持久化前 kind 仍需派生计算。
  • 需重新审视 flattenGroups 与 renderItem 派发逻辑——可能影响性能与现有测试。
