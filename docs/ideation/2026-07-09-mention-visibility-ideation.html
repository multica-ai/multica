Grounding Context

当前状态

成员/智能体/小队 mention 渲染为纯文本 .mention 类（color: var(--primary); font-weight: 600）。而 Issue 和 Project mention 已有丰富的 chip 组件（IssueChip、ProjectChip），带边框、图标、标识符。

核心矛盾

autocomplete 有头像 + 类型 badge，hover card 有完整信息——但内联渲染时全部被丢弃。用户在选中时看到丰富预览，渲染后却变成普通加粗文字。

1. Mention Avatar Chip + 类型着色
visual-form + type High Confidence Large Scope
描述

将成员/智能体/小队 mention 从纯文本升级为与 IssueChip/ProjectChip 同级的 chip 组件：圆角 pill 容器 + 14-16px ActorAvatar + 名称。按 data-mention-type 分配不同背景着色：成员用中性色（bg-muted），智能体用品牌色（bg-brand/10），小队用信息色（bg-info/10），@all 用警告色（bg-warning/10）。

视觉示意
成员：张@张三   智能体：⚙@ReviewerBot   小队：◆@设计组   @all：⊕@all
与 IssueChip 对比： ⚙@ReviewerBot #456 实现缓存 #123 修复登录
Basis

direct: IssueChip 使用 rounded-md border mx-0.5 px-2 py-0.5 text-xs + hover:bg-accent transition-colors；ActorAvatar 已按类型区分（squad=方形 Users 图标、agent=圆形 Bot 图标、member=首字母）；data-mention-type 已在 DOM 上；移动端 ActorAvatar 已使用 bg-brand/15 标识智能体。

external: Slack、Discord、Jira、Confluence、Notion 全部使用背景 pill 渲染 mention；Jira/Confluence 嵌入头像缩略图。

Rationale

当前 mention 系统的核心矛盾是：Issues 和 Projects（相对少用）有丰富的视觉表现，而成员/智能体（最常用）只有纯文本。这种倒置让最重要的实体看起来最不重要。Avatar chip 将 mention 升级到与 Issue/Project chip 同级，同时复用已有的 ActorAvatar 类型区分能力——这是最高杠杆的改动。

Downsides

改动范围较大：需要新建 ActorMentionChip 组件，修改 MentionView 和 ReadonlyContent 的渲染逻辑，更新 prose.css。14px avatar 在密集文本中可能显得拥挤。需要同时在 web 和 desktop 验证。但 IssueChip 已证明 22px 高度在 1.625 行高内可正常工作。

2. 插入闪光动画
creation-feedback High Confidence Small Scope
描述

当 mention 从 autocomplete 插入后，对新渲染的 mention 节点施加一个 150-200ms 的背景色过渡动画：从 bg-brand/20 渐变到最终背景色。这个短暂的"闪光"直接桥接了 autocomplete 的丰富预览和渲染后的稳定态，为用户提供"系统识别成功了"的即时确认。

动画序列
⚙@Bot
0ms · 插入瞬间
高亮 + 发光
→
⚙@Bot
~100ms · 过渡中
渐变消退
→
⚙@Bot
200ms · 稳定态
最终 chip
Basis

direct: Tiptap mention command 插入节点 + 尾部空格后 collapseToEnd()，无任何视觉确认；autocomplete 有完整的头像+badge 预览，插入后瞬间降级为纯文本；transition-colors 已是 chip 的标准交互模式（150ms ease-out 符合 DESIGN.md §8.2）。

external: 游戏 UX 中的"拾取确认"反馈——在复杂视觉环境中通过短暂动画确认状态变化。

Rationale

直接回答用户的核心诉求："能不能更显式地让我知道正确使用了 mention 语法"。闪光动画在不改变最终渲染形态的前提下，创造了一个可感知的 before/after 时刻。它是所有方案中成本最低但最直接的"确认信号"。

Downsides

需要检测"新插入"与"已有"mention 的区别（可通过 Tiptap 事务或 CSS animation on first render 实现）。在长评论中频繁 mention 时，多个闪光可能显得杂乱。但动画持续时间短（150ms），且只在创建瞬间触发，不影响已有内容的阅读体验。

3. 通用 Hover 交互
interaction-feedback High Confidence Small Scope
描述

为所有成员/智能体/小队 mention 接入 hover 交互反馈：cursor-pointer + hover:bg-muted（符合 DESIGN.md §5.2 的 hover 模式）。同时接入已有的 MentionHoverCard——hover 时显示 ActorAvatar + 全名 + 角色，与 Issue/Project mention 的交互体验对齐。可选增加 active:scale-95 的 pressed 反馈（但 DESIGN.md 反对 scale，建议用 active:translate-y-px 代替）。

Basis

direct: 当前成员 mention 的 hover 反馈为零：cursor-default、无 hover 背景变化、无点击动作。而 IssueChip/ProjectChip 有完整的 hover:bg-accent transition-colors cursor-pointer。MentionHoverCard 组件已存在但未绑定到内联 mention。

Rationale

用户"不确定 mention 是否正确"的另一个维度是交互层面：如果 hover 一个 mention 没有任何反应，它在感知上就是"死的"文字。接入 hover 交互后，用户 hover 时看到头像+角色信息弹出——这就是"mention 是活的"的直接证明。改动量极小：CSS 一行 + MentionHoverCard 包裹。

Downsides

MentionHoverCard 可能需要适配 chip 形态的 mention（当前设计针对文本 mention）。在密集 mention 场景下频繁弹出 hover card 可能干扰阅读。但 hover card 本身有延迟（dwell time），且只在鼠标停留时触发。

4. 上下文信息 Pill
actor-type-distinction Medium Confidence Medium Scope
描述

在 mention chip 中嵌入实时状态信息：成员 mention 显示 6px 在线状态点（绿色=在线、灰色=离线，复用 ActorAvatar 的 showStatusDot 模式）；智能体 mention 显示运行状态图标（active/idle）；小队 mention 显示成员计数后缀（如 ·5）；@all 显示当前工作空间成员数。Mention 从静态名称标签变为一个迷你仪表盘。

视觉示意
张@张三   李@李四   ⚙@Bot   ◆@设计组·5
Basis

direct: ActorAvatar 在 suggestion list 中已使用 showStatusDot 属性；IssueChip 已建立了 inline chip 反映实体当前状态的模式（StatusIcon 显示 open/closed）。

Rationale

这让 mention 从"引用"变为"感知"——看到 @张三 旁边有绿点，你知道他在线、会很快看到消息。对于智能体，运行状态指示器让用户知道"这个 AI 正在工作"。这直接强化了 Multica 的核心差异化：agents as first-class assignees。

Downsides

状态数据可能过时（autocomplete 查询和渲染之间的时间差）。多个 mention 各自查询状态可能增加 React Query 订阅开销。状态点在高密度文本中可能显得杂乱。建议将状态点设为可选增强（通过 prop 控制），默认关闭，在关键场景（如 issue sidebar 的 assignee 列表）开启。

5. 未识别 Mention 错误态
creation-feedback + visual-form Medium Confidence Medium Scope
描述

反转思路：不在"正确"时做文章，而在"错误"时给出明确信号。当用户输入了 @某某 但 autocomplete 未匹配到任何结果时，该文本自动变为红色下划线 + 感叹号图标（类似拼写检查错误），aria-label="未识别的 mention"。匹配成功的 mention 正常渲染。用户的不确定性被转移到"什么情况下是错的"——没有红色标记就是正确的。

视觉示意
成功：请 张@张三 审核这个 PR
失败：请 @张小五⚠ 审核这个 PR
Basis

direct: Autocomplete 已有 "No matches" 空状态（$.mention.no_results）；Badge 组件已有 destructive variant（bg-destructive/10 text-destructive）；DESIGN.md §5.7 定义了 aria-invalid:border-destructive 错误模式。

reasoned: 拼写检查是用户已有的心智模型——红色下划线 = 有问题。利用已有心智模型比创造新视觉语言更有效。

Rationale

这是对"怎么让 mention 更显眼"问题的反向解答。与其让正确 mention 更显眼，不如让错误 mention 更显眼——用户看到没有红色标记就知道"系统识别了"。这个思路成本低、效果明确，且与 DESIGN.md 的错误状态规范完全对齐。

Downsides

需要区分"用户正在输入 @query"（autocomplete 还在搜索）和"输入完毕后未匹配"（确认错误）。Tiptap 的 mention 逻辑只在选中 suggestion 时创建 mention node，纯文本 @某某 不会自动变成 mention——需要额外的 plugin 检测未匹配模式。

6. 渐进式展开 Mention
inline-text-flow + visual-form Medium Confidence Medium Scope
描述

保持默认渲染态轻量（带背景色的 @name 文本，不加 avatar），通过 CSS :hover 过渡在 hover 时展开显示 avatar + 类型图标——全部在同一个 DOM 元素内用 CSS transitions 实现。mention 在"阅读模式"下保持紧凑不干扰文本流，在"探索模式"下展开为完整 chip。这解决了"chip 太拥挤"和"纯文本不明显"之间的矛盾。

视觉示意
默认态（紧凑）
@张三 @Bot
Hover 态（展开）
张@张三 ⚙@Bot
Basis

direct: DESIGN.md §5.2 定义了 hover 状态为"我注意到你了"，视觉变化应该轻微、即时；data-mention-type 已在 DOM 上，CSS 可以针对不同类型设置不同 hover 颜色；transition-colors 是 DESIGN.md 推荐的首选 transition（150ms）。

Rationale

这个方案在"mention 需要更显眼"和"mention 不应该占太多空间"之间找到了平衡点。默认态通过背景色提供基础的"这不是普通文字"信号，hover 态通过展开 avatar 提供完整的身份确认。适合 comment 密集的场景（如 issue 讨论串），其中阅读效率优先。

Downsides

hover 展开在触摸设备上不可用（移动端需要 tap）。CSS-only 展开需要 avatar 在默认态也存在于 DOM 但宽度为 0（overflow: hidden; width: 0），过渡到 width: 16px——但 width transition 性能不如 transform。可以用 max-width 或 margin 替代。

7. 纯头像圆形 Mention
visual-form Low Confidence Small Scope
描述

极端约束翻转：完全移除名称文本。Mention 变为一个 16px 的 ActorAvatar 圆形（成员=首字母、智能体=Bot 图标、小队=Users 图标方形）。全名仅在 hover card 中显示。这让 mention 成为文本中最小的内联元素——比 IssueChip 还小——但因为形状打破了排版节奏，它反而比任何文字处理都更显眼。

视觉示意
任务分配给了 张 并通知了 ⚙ 和 ◆ 。
Basis

reasoned: ActorAvatar 已能在任意尺寸下渲染不同视觉（fontSize 按 size * 0.45 缩放）；MentionHoverCard 已在 hover 时提供全名+角色；mention:// URL 保留完整语义信息。

Rationale

最大胆的方案。通过让 mention "不再是文字"来解决"mention 融入文字流导致不明显"的问题。圆形头像在文字排版中是天然的视觉中断点——眼睛会自动注意到它。对于 agent-heavy 的工作流（Multica 的核心场景），一个 Bot 图标比 "@ReviewerBot" 文字更容易扫视识别。

Downsides

可读性风险：在 5+ mentions 的长评论中，纯头像难以快速扫读（需要逐个 hover 才知道是谁）。不兼容无头像的成员（首字母可能在某些文化中不够直观）。这是一个值得探索的方向但不建议作为默认方案——可以作为"紧凑模式"供用户选择。
