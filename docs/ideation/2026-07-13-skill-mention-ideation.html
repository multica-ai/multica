Grounding Context

Codebase: Multica is a TypeScript + Go monorepo — Next.js (web), Electron (desktop), Expo (mobile). AI-native task management for small teams with agents as first-class assignees.

Current @ mention system: Tiptap v3 editor extension supporting 6 types (member, agent, squad, issue, project, all). Serialized as [@Label](mention://type/uuid) markdown. Backend regex in mention.go:16, frontend type union in mention-suggestion.tsx:51-65. Data sourced from Query cache (workspaceKeys.members/agents/squads, issueKeys.list).

Skill system: Workspace-level reusable instruction packages (SKILL.md + optional files). Full CRUD API exists. React Query hooks: workspaceKeys.skills(wsId), skillListOptions(wsId). Skills are currently only accessible via / slash commands (for agent invocation), not via @ mentions.

Key insight: Skills are the mechanism by which agents gain capabilities. Making them @-mentionable creates a unified entity graph where any first-class object (people, agents, squads, issues, skills) can be referenced inline. The existing architecture has clear extension points but requires coordinated changes across 8+ files — which is why the registry pattern (Survivor #1) is the highest-leverage investment.

External prior art: Notion's unified @ dropdown with grouped entity types is the closest UX analog. GitHub Copilot Chat uses typed @ prefixes (@file, @workspace). Slack separates triggers (@, #, /). The industry trend is toward entity-graph mentions where any first-class object is mentionable.

    • MentionType 注册表 — 一个扩展点统管所有未来类型

    自动补全与发现 Conf High Cx High
    Description

    今天添加一个 mention 类型需要修改至少 8 个硬编码位置：MentionItem.type 联合类型、Go 正则 MentionRe、readonly-content.tsx 正则、MentionView if/else 分发、groupItems() 分组、ActorMentionType 守卫、renderMarkdown 前缀逻辑、defaultRenderMention 分发。建立一个 MentionTypeRegistry — 声明式映射 { type, labelPrefix, chipVariant, groupLabel, renderComponent } — 让每个调用点遍历注册表而不是 switch 字面量联合。Skill mention 成为第一个消费者；后续每个新类型只需一行注册。

    示意：从 8 处硬编码到 1 行注册
    Basis

    direct: 代码确认 8 处独立的类型分发：MentionItem.type 联合类型 (mention-suggestion.tsx:51-65)、ActorMentionType (actor-mention-chip.tsx:30)、MentionView if/else (mention-view.tsx:35-62)、MentionHoverCard 三元链 (mention-hover-card.tsx:39-48)、ReadonlyContent 正则 (readonly-content.tsx:266)、renderMarkdown 前缀 (mention-extension.ts:109-110)、defaultRenderMention (markdown.tsx:63-81)、groupItems 桶逻辑 (mention-suggestion.tsx:87-115)。

    Rationale

    这是最高杠杆的单一投资。添加第 7、第 8 或第 12 个 mention 类型的边际成本从多文件、多包重构降到一行配置。Skill mention 是第一个消费者，验证模式；后续每个新实体类型（文档、模板、里程碑）只需一行注册。

    Downsides

    高前期投入。需要重构现有 6 种类型的分发逻辑，同时保持行为不变。后端正则也需要抽象（Go 没有 TypeScript 那样的注册表模式，可能需要配置驱动的正则构建）。

    @ 弹窗中展示工作区 Skills

    自动补全与发现 Conf High Cx Medium
    Description

    用户目前无法通过 @ 发现或引用 skills。buildSyncItems() 从 4 个缓存读取（成员、代理、小队、issues）但从不读取 skills 查询缓存。MentionItem 类型联合中没有 "skill"。扩展方案：在 MentionItem.type 中加入 "skill"，在 buildSyncItems() 中读取 workspaceKeys.skills(wsId)，在 groupItems() 中添加 "Skills" 分组，实现 MentionRow 的 skill 分支渲染。

    Basis

    direct: mention-suggestion.tsx:538-548 确认只读取 workspaceKeys.members、.agents、.squads 和 issueKeys.list — 从不读取 skills。workspaceKeys.skills(wsId) 在 queries.ts:18 已存在，skillListOptions(wsId) 在 queries.ts:74-79 提供完整的 React Query 配置。

    Rationale

    Skills 是产品的核心可复用指令包，却只能通过 / 斜杠命令路径发现，该路径要求用户先选择一个 agent 然后才知道输入 /。这使 skills 对任何从 @ 开始的用户流程实际上不可见。

    Downsides

    增加下拉菜单的复杂度（从 6 种到 7 种类型）。需要决定 skill 是否参与最近使用排序、如何处理大量 skills 的情况（分页或搜索阈值）。

    Skill 优先的 Mention 反转 — 说「做什么」而非「谁来做」

    语义行为 Conf High Cx Medium
    Description

    反转当前 agent 优先模型。今天用户先 @AgentA，再用 / 附加 skill。翻转：用户直接 @code-review，系统通过 selectSkillAssignments 解析哪个 agent 执行。Agent 变成实现细节而非主要选择面。结合自动路由（优先当前 issue 指派人 > 最近使用 > 在线状态）和 hover card 上显示 "via Agent Alpha"，用户可以点击切换。

    从「告诉系统谁来做」到「告诉系统做什么」
    Basis

    direct: selectSkillAssignments 在 queries.ts:101-115 已构建 Map<skillId, Agent[]> — 这正是从 skill 解析到 agent 所需的数据结构。后端 computeCommentAgentTriggers 在 comment.go 中已有 squad → leader 的解析模式，可类比扩展到 skill → agent。待处理任务去重逻辑 (comment.go:1970-1973) 已处理同一 agent 对同一 issue 重复排队的情况。

    Rationale

    用户以结果为导向思考（「运行代码审查」）而非路由（「调用拥有代码审查 skill 的 Agent Alpha」）。Skill-assignment 数据模型已经支持这种反转；只有 UI 强制了 agent 优先的路径。

    Downsides

    多 agent 拥有同一 skill 时的路由策略需要设计（默认选谁？）。对 agent 拥有大量 skills 的工作区，@ 弹窗可能过于拥挤。需要决定触发语义：纯引用链接 vs 触发 agent 执行。

    Skill Mention Hover Card — 悬停即发现

    行内渲染与视觉识别 Conf High Cx Low
    Description

    扩展 MentionHoverCard 为 type === "skill" 渲染 SkillProfileCard。卡片展示 skill 的名称、描述，以及关键信息 — 当前绑定了哪些 agents（使用 selectSkillAssignments 的 Map<skillId, Agent[]>）。每个行内 skill mention 变成交互式发现表面：悬停即可看到 "绑定于: Agent-Alpha, Agent-Beta" 并可点击跳转。

    Basis

    direct: MentionView 已将 actor 类型分发到 ActorMentionChip + MentionHoverCard (mention-view.tsx:65-71)。MentionHoverCard 使用类型驱动的三元链选择 profile card 组件 (mention-hover-card.tsx:39-48)。selectSkillAssignments (queries.ts:101-115) 已在 Query 缓存中提供 skill→agent 映射。

    Rationale

    Agent-skill 绑定目前在上下文中不可见 — 只能在 agent 设置页或 skill 详情页看到。在上下文中（评论、issues、聊天）暴露它创建被动发现能力：每个 skill mention 教会每个读者哪些 agent 可以执行它。随时间推移，这在无需刻意培训的情况下提高工作区整体的 AI 素养。

    Downsides

    需要设计 SkillProfileCard 组件。当 skill 绑定了很多 agents 时，hover card 需要截断或滚动。

    SkillChip 视觉识别 — 技能一目了然

    行内渲染与视觉识别 Conf High Cx Low
    Description

    创建专属 SkillChip 组件，使用 BookOpenText 图标和紫色调（与 skill-import-search ideation 一致），遵循 TYPE_STYLES 的颜色映射模式。Skill mention 应可从视觉上与人/agent mention 立即区分。在 MentionView 中为 type === "skill" 添加渲染分支：类似 IssueChip 模式（可导航链接到 skill 详情页），而非 actor 模式。

    Basis

    direct: TYPE_STYLES 在 actor-mention-chip.tsx:57-62 已建立每类型颜色映射模式（member=muted, agent=brand, squad=info, all=warning）。IssueChip 和 ProjectChip 已证明「mention chip + 可导航链接」模式可行。MentionView 对未知类型回退到纯文本 <span className="mention">@{name}</span> — skill 会命中此路径。

    Rationale

    如果用户无法在不阅读文本的情况下区分 mention 类型，mention 系统的可信度下降。视觉识别是最低成本、最高感知的改进。

    Downsides

    需要选择一致的视觉语言（颜色、图标）。如果后续有更多实体类型，每种都需要视觉设计。

    修复非编辑器上下文中 Agent/Member Mention 渲染缺失

    跨界面一致性 Conf High Cx Low
    Description

    共享 markdown 组件中的 defaultRenderMention 函数只处理 issue 和 project 类型，对所有其他类型返回 null。这意味着 agent、member、squad、@all 的 mention 在聊天历史、skill 输出、或任何 MemoizedMarkdown 上下文中渲染为空白。数据存在于 markdown 中，但渲染被静默吞没。修复此 bug 同时为 skill mention 的跨界面渲染铺平道路。

    Basis

    direct: packages/views/common/markdown.tsx:63-82：defaultRenderMention 检查 type === "issue" 和 type === "project"，然后 fall through 到 return null。每个 agent、member、squad、@all mention 在此代码路径中渲染为空。

    Rationale

    用户看到 agent mention 消失会认为系统坏了。mention 数据在 markdown 中保留但渲染被静默吞没，造成「写了什么」和「显示了什么」之间的断裂。这也是确保 skill mention 在所有界面一致渲染的前提。

    Downsides

    修复需要为每种类型实现渲染逻辑 — 如果同时做 Survivor #1 的注册表，可一并解决。

    聊天中发现 Skills — 对话内引用

    跨界面一致性 Conf Medium Cx Low
    Description

    聊天输入的 mentionMode: "context" 目前只展示当前/最近的 issues 和 projects。扩展为也展示当前聊天 agent 的绑定 skills。当用户与 Agent-Alpha 聊天时，输入 @ 应在建议列表顶部展示 Agent-Alpha 的绑定 skills，在 "Agent Skills" 标题下分组。在聊天中插入 skill mention 给 agent 提供该 skill 指令的上下文感知，无需切换到斜杠命令或离开对话。

    Basis

    direct: use-chat-context-items.ts:75-116 从 issues 和 projects 构建 MentionItem[] — skills 被排除是构造性的，不是设计性的。workspaceKeys.skills(wsId) 缓存已存在且被 realtime 事件保持新鲜 (use-realtime-sync.ts:343)。斜杠命令系统已在 slash-command-suggestion.tsx:181 读取 activeAgent?.skills — 相同数据源可喂给 @ mention 自动补全。

    Rationale

    聊天是 ad-hoc agent 交互的主要界面。如果用户不能在那里引用 skills，他们必须上下文切换到评论或使用斜杠命令，打断对话流。

    Downsides

    聊天中 @ 弹窗的 items 增加可能影响响应速度。需要决定是否在非 agent 聊天场景也展示 skills。
