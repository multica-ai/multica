Skill Import Search & Discovery

The Multica frontend's "new skill (copy from runtime)" dialog shows a long unsorted list of skills with no search or filter. Five directions — from an adaptive UI that switches on decision complexity, to a structural rethinking of the dialog around intent declaration — address discovery, organization, ranking, and layout.

Grounding Context

Affected component. packages/views/skills/components/runtime-local-skill-import-panel.tsx — lines 1012–1048 render a plain .map() over runtimeSkills with no search, filter, sort, or grouping. The dialog's height is !h-[min(600px,85vh)]; the scroll region is flex-1 overflow-y-auto. Data comes from useQuery(runtimeLocalSkillsOptions(...)) with poll-based discovery (500ms interval, 30s timeout).

Data model. RuntimeLocalSkillSummary exposes key, name, description?, source_path, provider, root? ("provider" | "universal"), and file_count. The root field is a natural facet for grouping but currently unused in the UI.

Reusable patterns. The subscriber picker (issue-detail.tsx:128-178) uses inline Command + shouldFilter={false} + CommandGroup + checkboxes inside CommandItem — the closest analog. The property-picker (property-picker.tsx:111-144) already ships full keyboard navigation with isImeComposing guard and auto-select-when-one. mention-recency.ts:54-80 and recent-issues-store.ts:37-42 provide LRU patterns that would extend naturally to import-recency tracking.

Critical constraint. CommandDialog cannot nest inside the already-open dialog — any search UI must use inline Command. The multi-select checkbox + single-select inline-edit state machine is unique to this panel and must be preserved.

R1

Adaptive Skill Discovery — UI branches on decision complexity

discovery · list-org · interaction Confidence 92% Complexity Med merged from R1+R3

The dialog recognizes the decision's complexity and adapts its UI accordingly, branching on runtimeSkills.length:
· 0 skills → existing empty state.
· 1–5 skills → single "Import" summary card with skill previews. The root facet becomes two compact buttons — "Import N provider skills" / "Import N universal skills" — plus inline descriptions. No search needed; the list is a confirmation tool, not a discrimination tool.
· 6+ skills → full inline Command (shouldFilter={false}) + root grouping via CommandGroup + alphabetical sort + full keyboard navigation (↑↓, Enter, Esc, isImeComposing guard, auto-select-when-one).

Key insight: the root field is the shared skeleton of both branches. In the summary branch, it becomes two bulk-import buttons; in the full-list branch, it becomes two CommandGroups. Same data dimension, different visual density.

Directional overview — three branches on skill count
Basis
direct subscriber picker at issue-detail.tsx:128-178 uses the exact Command + checkbox pattern; agent.ts:867-874 defines root as a natural 2-group facet; property-picker.tsx:111-144 ships the keyboard handler verbatim; the panel already branches on runtimeSkills.length at lines 1000–1010 (existing empty-state pattern).
reasoned The two original ideas (R1: search + grouping + keyboard; R3: progressive disclosure for small runtimes) were not alternatives — they addressed disjoint regions of the same decision surface. Merging them produces a single coherent principle: adapt the UI to the decision's complexity.
Rationale
Most dialog systems treat all list sizes the same way, forcing power users through scrolling AND trivial decisions through unnecessary UI. This merge solves both with a single, small branching rule. The root field unifies both branches — a rare case where two strong ideas turn out to be one idea seen from different angles.
Downsides
Three render paths to maintain (vs one today). Threshold values (5, 6) are judgement calls that may need tuning. Inline-edit state machine must coexist with CommandItem highlight in the 6+ branch.
R2

Recency LRU Pinning — "Recently Imported"

relevance Confidence 88% Complexity Low

Create a small session-scoped Zustand store mirroring mention-recency.ts. On each successful import, record { key, timestamp } to localStorage under a workspace-scoped key (cap 50 entries, head-insert + LRU eviction). Partition the list into "Recently Imported" (up to 5, LRU order) at the top and "Available" (alphabetical) below. The "Recently Imported" group only renders when non-empty. Search filters across both groups but the recency group always pins first.

Basis
direct mention-recency.ts:54-80 implements recordMentionUsage + sortUserItemsByRecency with LRU eviction; recent-issues-store.ts:37-42 shows the per-workspace LRU pattern.
external Raycast "Pinned" section and Linear "Recent" — both reduce list length by surfacing frequently-used items.
Rationale
This is the compounding bet. Once the recency store exists, it later powers default-checked suggestions, a "quick import" mode, and cross-runtime intelligence. The first commit is ~40 lines; every subsequent feature reuses it.
Downsides
Introduces localStorage state for the first time in this panel. Workspace-scoping must be correct or recency leaks across workspaces.
R3

Scoped Select-All + Multi-Field Search + Conflict Preview

interaction · discovery Confidence 75% Complexity Med

Three improvements to the search + selection flow. (a) When a search query is active, "select all" only selects the filtered skills — show "Select all 3 (of 12)" next to the checkbox. (b) Extend search to match on name + description + source_path, with pinyin support. When the match is on path or description but not name, render a small indicator and highlight the matched substring. (c) Preview name conflicts inline with a warning icon on each conflicting row before import — move conflict resolution from a reactive error state to a proactive preview state.

Basis
direct runtime-local-skill-import-panel.tsx:580-589 already computes pendingConflicts and conflictResolutions; the data is available to surface earlier. property-picker.tsx:166-167 ties search state to selection state. issue-detail.tsx:122-123 substring + pinyin pattern.
external Linear Cmd+K scopes "select all" to the visible filtered set.
Rationale
A search bar without selection-scope awareness creates a trap: the user filters to "auth", hits "Select All", and gets 47 skills instead of 3. Multi-field search + conflict preview closes the two most common post-search surprises.
Downsides
Three changes in one idea; the conflict-preview may require backend cooperation to detect collisions cheaply. Scope risk if tackled all at once.
R4

Hit / Non-Hit Partition with Visual Dimming

relevance · interaction Confidence 70% Complexity Low

When the search box is non-empty, matching skills render at full opacity at the top; non-matching skills remain visible below at ~40% opacity, still selectable, rather than disappearing entirely. This preserves spatial memory of the list and avoids the CommandEmpty dead-end — a common failure mode of pure-filter search is the user typing slightly wrong and seeing "no results" with no path back.

Directional overview — matches prominent, non-matches dimmed but present
Basis
direct today's CommandEmpty state at issue-detail.tsx:137-139 is a dead end when spelling is wrong. The .map() is a single render pass, trivially partitionable into [matches, rest].
external gene promoters upregulate target genes without silencing the rest of the genome — a natural analogy.
Rationale
Dim-but-present preserves spatial memory of the list and lets users correct their query by seeing what almost-matched. Mechanically trivial on the existing render.
Downsides
Unconventional UX — users may not expect partially-dimmed results. For very long lists the dimmed tail adds visual noise. Need to tune the opacity threshold.
R5

Smart Defaults Layer — Presets + Two-Zone + Auto-Collapse

relevance · interaction Confidence 60% Complexity High

A three-part composition that reframes the dialog from "pick a skill from a list" to "declare your intent": (a) Above the searchable list, render 3–5 preset rows for common actions — "New blank skill", "Import all provider skills", "Import all universal skills" — each single-click. (b) Split the dialog into two zones: top = "Available skills" (click to add), bottom = "Will import (N)" (selected chips with inline edit when exactly one is focused) — decoupling "browse" from "edit". (c) After bulk import completes, imported skills collapse into a single summary row ("✓ 5 skills imported") with a "Show imported" toggle, keeping the working set focused on what still needs attention.

Directional overview — three zones stacked
Basis
direct toggleAll at 569–574 makes bulk-by-root natural; singleSelectedSkill at 545–548 already gates inline edit on selectedKeys.size === 1; bulkState.phase transitions at 836–840 make post-import collapse a natural extension.
reasoned Many users don't want to pick individual skills — they want "all of them" or "none." Presets compress the multi-click flow into one click. Decoupling browse from edit addresses the current tension between checkbox rows and inline editing.
Rationale
This is the ambitious "phase 2" vision — a full rethinking of the dialog around intent declaration rather than list scanning. It composes naturally with R1 and R2, building toward a coherent product direction.
Downsides
High complexity — three interacting features. The two-zone layout is a structural change that affects the state machine. Should be tackled incrementally: presets first, then tray, then collapse.
