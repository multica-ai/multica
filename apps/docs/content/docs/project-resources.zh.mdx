[项目](/projects)资源告诉智能体这组工作会用到哪些代码，以及应该在哪里执行。资源会持续关联在项目上，不必在每条 issue 中重复粘贴仓库地址或本地路径。

目前支持两种资源：

| 资源 | 适合什么情况 | 执行位置 |
|---|---|---|
| **GitHub 仓库** | 团队共享的代码库，希望由运行时管理 checkout | 运行时管理的工作目录 |
| **本地目录** | 已经存在的 checkout、很大的仓库，或需要直接查看本地改动 | 指定电脑上的原目录 |

## 资源如何进入一次执行

当智能体处理项目内的 issue 时，Multica 会把项目名称、项目描述和资源列表加入执行上下文，并在工作目录中写入 `.multica/project/resources.json`。

如果项目关联了 GitHub 仓库，智能体可以使用对应 URL 和 ref checkout 代码。项目没有 GitHub 仓库时，运行时会回到工作区关联的仓库列表。

本地目录只对绑定它的那台守护进程有效。如果本次执行所在的守护进程有匹配的本地目录，智能体会直接进入该目录工作；其他电脑仍会使用项目中的 GitHub 仓库或工作区仓库。

## 添加 GitHub 仓库

打开项目，在 **资源** 中选择"添加资源"。你可以选择已经关联到工作区的仓库，也可以粘贴 Git URL。

创建项目时，也可以直接在 **Repos** 中选择仓库。一个项目可以关联多个 GitHub 仓库。

资源中的 `ref` 可以指定后续 checkout 默认使用的分支、tag 或 commit。`default_branch_hint` 只向智能体提供默认分支提示，不会强制切换分支。

## 添加本地目录

添加本地目录的界面只在 Desktop 中提供，因为浏览器不能选择电脑上的文件夹。

1. 确认 Desktop 的本地守护进程在线。
2. 打开项目的 **资源**。
3. 选择"添加本地目录"，再选择要使用的文件夹。

目录必须是绝对路径，已经存在，并且当前守护进程可以读取和写入。系统根目录、用户主目录和其他过于宽泛的系统路径会被拒绝。

每个项目在同一台守护进程上最多关联一个本地目录。团队中的不同电脑可以分别为同一个项目关联各自的目录。

<Callout type="warning">
本地目录不是隔离环境。智能体会直接看到并修改当前分支和未提交的文件；Multica 不会自动切换分支、stash、commit、push 或创建 PR。开始前先保存重要改动，并确认当前 checkout 适合交给智能体。
</Callout>

## 同一个本地目录一次只运行一条任务

如果两条执行任务要使用同一个真实目录，后来的任务会进入 `waiting_local_directory`，等前一条任务释放目录后再继续。即使两个路径通过不同的符号链接指向同一目录，也会串行执行。

等待不会修改目录。你可以取消等待中的任务；否则它会一直等到目录可用。

需要同时处理多项改动时，使用 GitHub 仓库资源，让运行时为不同工作准备各自的工作目录。

## 执行期间会写入什么

除了智能体产生的代码改动，运行时还可能在目录中写入当前 AI 编程工具所需的说明文件，以及 `.multica/project/resources.json`。如果不希望这些文件进入版本控制，请按团队约定加入 `.gitignore`。

Multica 清理执行环境时不会删除你关联的本地目录。不过，智能体本身拥有对该目录的读写权限，因此仍应像在终端中直接运行 AI 编程工具一样审查它的改动。

## 使用 CLI 管理资源

```bash
# 创建项目时关联仓库
multica project create \
  --title "Agent UX" \
  --repo https://github.com/multica-ai/multica

# 查看和添加资源
multica project resource list <project-id>
multica project resource add <project-id> \
  --type github_repo \
  --url https://github.com/multica-ai/multica \
  --ref main

# 关联指定守护进程上的本地目录
multica project resource add <project-id> \
  --type local_directory \
  --local-path /absolute/path/to/repo \
  --daemon-id <daemon-id>

# 移除资源
multica project resource remove <project-id> <resource-id>
```

资源变更会影响之后创建的执行任务，不会改写已经结束的执行记录。
