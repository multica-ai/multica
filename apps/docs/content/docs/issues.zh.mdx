在 Multica 中，issue 是安排工作的基本单位。它可以是一项功能、一个 bug、一次调研，也可以是任何需要有人或智能体负责的工作。

Issue 不只记录"要做什么"。围绕这项工作的讨论、状态变化和每次执行也会留在同一处，之后不需要再从聊天记录或终端日志里拼接上下文。

## 一个 issue 包含什么

| 内容 | 用途 |
|---|---|
| **标题和描述** | 说明目标、背景、要求和验收条件。 |
| **状态和优先级** | 表示工作处于哪个阶段，以及应该先处理什么。 |
| **负责人** | 可以是工作区成员、智能体或小队。 |
| **日期、标签和自定义属性** | 补充计划、分类和团队自己的字段。 |
| **项目和父子关系** | 把 issue 放进更大的工作范围，或拆成多个子 issue。 |
| **动态和执行日志** | 保存评论、状态变化、执行任务以及智能体返回的结果。 |

## 创建 issue

在 **Issues** 页面或项目中选择"新建 issue"，填写标题即可创建。描述和其他属性可以在创建时填写，也可以之后补充。

系统会为每个 issue 分配一个编号，例如 `MUL-123`。数字部分在工作区内递增；如果管理员修改了工作区的 issue 前缀，编号会使用新的前缀显示。详见[工作区](/workspaces#创建工作区)。

## 选择负责人

| 负责人 | 会发生什么 |
|---|---|
| **成员** | 表示由这位成员负责跟进，Multica 不会创建执行任务。 |
| **智能体** | 将工作交给该智能体，并为它创建一条执行任务。 |
| **小队** | 先由小队 leader 接收，再根据小队配置判断是否交给其他智能体。 |

将 issue 分配给智能体或小队时，如果它不在 `backlog`，执行任务会立即入队；运行时离线时，任务会留在队列中等待。

能否运行某个智能体由它的 Access 设置决定。把 issue 分配给无权使用的智能体不会绕过这层限制。详见[分配 issue 给智能体](/assigning-issues)。

## 状态

Multica 提供七种 issue 状态：

| 状态 | 适合什么情况 |
|---|---|
| `backlog` | 暂时不启动。即使已经分配给智能体，也要等离开 `backlog` 后才会创建执行任务。 |
| `todo` | 已经明确，等待开始。 |
| `in_progress` | 正在处理。 |
| `in_review` | 已有结果，等待检查或确认。 |
| `done` | 工作已经完成。 |
| `blocked` | 因依赖或其他问题暂时无法继续。 |
| `cancelled` | 不再继续，但保留已有记录。 |

状态之间没有强制的固定流转。成员和智能体可以根据实际进展直接修改状态。

<Callout type="info">
大多数状态变化都由成员或智能体完成。另有两个系统行为需要注意：一次执行失败且没有其他执行或重试时，`in_progress` 可能回到 `todo`；带有关闭意图的关联 GitHub PR 合并后，issue 会变为 `done`。
</Callout>

## Issue 和执行任务不是一回事

Issue 是持续存在的工作记录；[执行任务（task）](/tasks)是智能体的一次具体执行。

一个 issue 可以先后产生多条执行任务，例如第一次实现、补充修改和再次检查。执行任务显示"已完成"，只表示这一次执行结束，不代表 issue 已经完成。是否结束这项工作，以 issue 的实际进展和状态为准。

## 项目和子 issue

一个 issue 最多属于一个[项目](/projects)。项目用于汇总相关工作，并为其中的智能体提供共享说明和资源；把 issue 移到另一个项目不会复制它。

较大的工作还可以拆成子 issue。父 issue 保留整体目标，子 issue 分别记录可以独立推进的部分。父子关系不会自动替你决定状态，团队仍需按实际进展更新它们。

## 切换视图

Issues 页面提供列表、看板、表格、甘特图和泳道等视图，也可以按状态、负责人、项目或其他属性筛选和排序。这些视图展示的是同一批 issue；切换视图不会创建副本。

## 删除 issue

<Callout type="warning">
删除 issue 会永久移除它及其关联记录，并取消尚未结束的执行任务，无法恢复。如果只是决定不再继续，应把状态改为 `cancelled`，这样讨论和结果仍可查阅。
</Callout>

任何工作区成员都可以删除 issue。删除前先确认团队不再需要其中的上下文。
