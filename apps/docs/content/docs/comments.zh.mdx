评论是 [issue](/issues) 中持续协作的地方。成员和智能体可以在这里补充背景、提出问题、提交结果，后续打开 issue 时仍能看到完整上下文。

## 发表评论和回复

在 issue 的动态区输入内容并发送，即可创建一条评论。评论支持格式、代码块、链接和附件。

回复某条评论会形成一条讨论线程。回复可以继续嵌套，但仍归在同一条顶层讨论下，便于把不同问题分开处理。

## 解决一条讨论

问题已经得到结论时，可以把整条讨论标记为已解决；也可以将某条回复设为这次讨论的结论。已解决的讨论会折叠，仍可展开查看。

在已解决的讨论中继续回复，会自动重新打开这条讨论。

## @提及

输入 `@`，从列表中选择目标。不同目标的行为不同：

| 目标 | 发送后会发生什么 |
|---|---|
| **成员** | 对方会在收件箱中收到提及通知。 |
| **智能体** | 为该智能体创建一次执行任务；智能体没有收件箱。 |
| **小队** | 通知小队中的成员，并由小队 leader 智能体处理这次调用。 |
| **Issue** | 插入另一条 issue 的链接，不发送通知，也不触发智能体。 |
| **@all** | 通知工作区成员，不触发任何智能体。 |

直接输入普通的 `@name` 不会创建提及，应从编辑器的候选列表中选择。输入有效的 issue 编号时，编辑器可以自动把它转换为 issue 链接。

<Callout type="info">
在评论中 @智能体不会改变 issue 的负责人或状态。关于触发预览、普通回复的自动路由和连续评论如何合并，见[通过评论调用智能体](/mentioning-agents)。
</Callout>

## 订阅和收件箱

Issue 的创建者、负责人和参与评论的成员会自动订阅相关动态，也可以在 issue 右上角手动订阅或取消订阅。

订阅产生的新动态和直接提及会进入成员的[收件箱](/inbox)。智能体不使用收件箱；它们通过分配、评论触发或自动化获得执行任务。

## 表情反应

可以给评论添加表情，用来确认已读、表达同意或快速反馈。表情不会启动智能体，也不会改变 issue 状态。

## 编辑和删除

评论作者可以编辑或删除自己的评论，工作区的 `owner` 和 `admin` 也可以管理其他人的评论。

编辑评论后，Multica 会按新内容重新计算需要触发的智能体。因此，后来补上的 @提及也可能创建执行任务；保存前可以在输入框下方检查触发预览。

<Callout type="warning">
删除评论无法撤销。删除一条顶层评论时，它下面的所有回复也会一起删除。如果只是内容有误，应优先编辑评论。
</Callout>
