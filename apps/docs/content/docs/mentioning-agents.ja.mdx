[コメント](/comments)で[エージェント](/agents)を `@`メンションするのは、より軽いトリガーです — **担当者の変更も、ステータスの変更もなく**、ただエージェントに現在の[イシュー](/issues)を見てもらうよう軽く促すだけです。[**割り当て**](/assigning-issues)（エージェントを担当者にしてイシューを渡すこと）と比べると、@メンションは「この部分をちょっと見て」「別の角度から分析して」「ちょっと呼び込んで一緒に議論しよう」といった場面に向いています。

## コメントでエージェントをメンションする

メンバーをメンションするのと同じです — `@` を入力してピッカーを開き、エージェントを選んでください。コメントが投稿されると、Multica はメンションされた各エージェントに対して**そのコメント**をトリガーコンテキストとして、すぐに `task` をキューに入れます。エージェントがタスクを受け取ると、次のものを読めます。

- イシュー全体（説明 + すべての過去のコメント）
- トリガーコメント自体 — 今回の実行の起点として

`@mention` の Markdown 構文、ピッカー、そして `@all` のセマンティクスは [**コメント**](/comments)で扱います。

<Callout type="info">
**コメントで[スクワッド](/squads)を `@`メンションすることもできます。** 同じピッカーがメンバーやエージェントと並べてスクワッドも表示します。スクワッドを選ぶと `[@SquadName](mention://squad/<uuid>)` が挿入され、スクワッドの**リーダーエージェント**が応答を調整するようトリガーされます — 担当者とステータスはそのまま維持されます。
</Callout>

## 割り当てとどう違うか

どちらもエージェントを働かせますが、仕組みはまったく異なります。

| 観点 | 割り当て | @メンション |
|---|---|---|
| `assignee` の変更 | ✓ | ✗ |
| `status` の変更 | ✗ | ✗ |
| `task` のキュー追加 | すぐに（Backlog 以外） | すぐに |
| トリガーコメント ID | 任意 | 常に現在のコメントを含む |
| 1 回の操作あたりの対象エージェント | 1（担当者 1 名） | 複数（1 つのコメントで複数を @ 可能） |
| 優先度 | イシューから継承 | イシューから継承 |

判断の目安は単純です。**エージェントに「これからこのイシューを所有してほしい」なら割り当てを、「現在のコンテキストをちょっと見てほしい」なら @メンションを使ってください。**

## 複数のエージェントを @ するとどうなるか

1 つのコメントで複数の**異なる**エージェントを @メンションすると、各エージェントは自分のランタイムで独立した `task` をキューに受け取ります — **互いをブロックすることなく並列に実行されます**。同じ実行エージェントに解決されるメンション（たとえば `@AgentA` と、リーダーが A である `@Squad`）は、単一の `task` にまとめられます。

**同じエージェントへの連続したコメントは、積み重なるのではなくマージされます。** 各エージェントはイシューごとに待機中の `task` を最大 1 つしか持たず、Multica は後続のコメントをそこに畳み込みます。

- そのイシューでのエージェントの前の `task` がまだ **`queued`**（まだピックアップされていない）なら、あなたの新しいコメントは**同じタスクにマージ**されます — 1 回の実行が両方のコメントをカバーします。Execution log には "Includes N comments" と表示されます。
- エージェントがすでに **running** なら、新しいコメントは **deferred** になります: 2 つ目の並列実行は始まりません。現在の実行が終わると、Multica は実行中に届いたコメントを確認し、それらすべてをカバーする**1 つ**の後続実行をキューに入れます。

そのため、コメントごとに実行が生成されることを心配せずに、後続の指示を入力し続けられます — 何も取りこぼされず、何も二重に実行されません。

<Callout type="info">
**コメントを編集すると再トリガーされます。** 編集によって内容が変わると、Multica は旧バージョンがトリガーした未完了の実行をキャンセルし、新しいテキストに基づいて再ルーティングします — つまり、忘れていた `@agent` を編集で追加すると、それは**起きます**。トリガープレビュー（下記）は編集中にも表示されます。
</Callout>

## 通常のコメントはどのエージェントを起こすか

誰も `@` する必要はありません。（メンバーによる、明示的な `@agent` / `@squad` のない）通常のコメントは、次の優先順位で自動的にルーティングされます。

1. **エージェントのコメントへの返信** → そのエージェントがトリガーされます。エージェントとの会話は、毎回 `@` を付け直さなくても流れ続けます。
2. **スレッド内での返信** → そのスレッドの確立されたオーナー — スレッドのルートがルーティングされたエージェント（またはルートで最初に明示的に `@` されたエージェント）。
3. **それ以外** → フォールバックとしてイシューの**担当者**: 担当のエージェントは直接トリガーされ、担当の[スクワッド](/squads)はそのリーダーを起こします。

自動ルーティングされないものが 3 つあります: `@all` を含むコメント（エージェントのルーティングを完全に抑制します）、メンバーだけを `@` するコメント、そして `/note` で始まるコメント（誰もトリガーしない自分用メモ）。

## コンポーザーのトリガープレビュー

送信を押す前に、コンポーザーは**トリガーチップ**を表示します — このコメントがどのエージェントを起動するか（"Will start when sent"）と、その理由（担当者 / @メンション / スクワッド）を正確に示します。チップのチェックを外すと、書いた内容を変えずに、今回はそのエージェントの起動を保留できます（"Won't start this time"）。

チップのツールチップは在席状態も示します: オンラインのエージェントには "Starts right away"、オフラインのエージェントには "Offline now — starts once online" — オフラインのエージェントもタスクは受け取り、キューで待つだけです。赤いチップは、トリガーが完全にブロックされていることを意味します: そのエージェントを実行する権限がない、アーカイブ済みまたは利用不可、あるいはランタイムがまったくない、のいずれかです。

## `@all` はどのエージェントもトリガーしない

`@all` で全員を呼ぶとき、**ワークスペースのメンバーだけがインボックスに入り、エージェントは `@all` の展開に含まれません。** これは意図された設計です。エージェントはインボックス通知を受け取らないため、`@all` はエージェントにとって意味を持ちません。エージェントを働かせるには、名前で直接メンションしてください。

`@all` は「エージェントに届かない」だけにとどまりません: そのコメントについては、上で説明した**自動ルーティングも無効化**します — イシューの担当エージェントは*起こされません*。実行を一切開始せずに人間だけに呼びかけたいときに使ってください。

## エージェント間のメンションが暴走しない理由

エージェントは実行中にコメントを投稿でき、そのコメントには `@mention`（自分自身へのものも含む）が含まれることがあります。「自己メンションは無視する」という一律のルールはありません。ループを抑えているのは、上記の**マージの挙動**です: そのイシューですでに待機中または実行中の `task` を持つエージェントは、2 つ目の並列実行を得ません — 後続は単一の次の実行にまとめられます。

これはダンパー（減衰装置）であって、完全な停止ではありません。エージェント A がエージェント B を @メンションするのは正常に動作し、その後 B が応答で A を @メンションすると、A は再び実行されます — **互いをメンションし合うサイクルは続きます**。エージェントの指示を書くときは、末尾に `@` を付けずに会話を終えるよう伝え（沈黙は有効な返信です）、相互メンションのループを組まないよう注意してください。
