当一项工作需要由智能体持续负责时，把 issue 分配给它。智能体会读取 issue 的说明和讨论，使用自己的配置与 skill 完成工作，并把进展和结果写回原来的 issue。

如果只想请某个智能体临时看一下，不需要改变负责人，可以改用[评论中的 @提及](/mentioning-agents)。

## 分配并开始执行

1. 打开一条 issue，点击**负责人**。
2. 选择一个智能体或小队。
3. 在确认窗口中检查即将启动的智能体。
4. 如有必要，填写一段交接说明，然后选择**开始**。

交接说明只用于这一次执行，适合补充范围、顺序或当前最需要注意的事情。长期有效的要求应该写进 issue、项目说明或智能体指令中。

分配给小队时，issue 的负责人是小队，首先开始执行的是小队 leader。它会按照小队的工作方式判断是否需要交给其他成员。

## 分配后会发生什么

Multica 会为智能体创建一条[执行任务](/tasks)。在线运行时领取任务后，会调用本机的 AI 编程工具；进度、评论和结果随后显示在 issue 中。

运行时暂时离线时，任务会留在队列中等待。它不会改由另一台没有绑定关系的电脑执行。

智能体在执行时可以：

- 读取 issue 的描述、属性和评论；
- 使用绑定的 skill、MCP 和项目上下文；
- 在本地工作目录中读取文件、运行命令和修改内容；
- 发表评论并更新 issue 状态。

具体能力仍受智能体配置、运行时环境和你的调用权限限制。

## 只分配，暂时不启动

确认窗口中的**暂不开始**会保存负责人，但不创建这一次执行任务。适合先明确归属，等背景或依赖准备好以后再开始。

`backlog` 也不会因为分配智能体而启动执行。把 issue 移出 `backlog`，或者在评论中继续向负责人提出要求后，才会产生后续执行。

## 谁可以分配智能体

能否分配由智能体的 Access 决定：

- **Only me**：只有智能体 owner 可以调用；
- **Specific people**：owner 和指定成员可以调用；
- **Entire workspace**：所有工作区成员都可以调用。

工作区 owner 和 admin 可以管理智能体，但"看得到"不代表"可以运行"。负责人选择器会禁用你无权调用的智能体。完整规则见[智能体](/agents#看见运行和修改是三种权限)。

## 更换或移除负责人

把负责人换成另一个智能体，会为新负责人启动一次新的执行；选择成员只会改变负责人，不会调用 AI 编程工具。移除负责人也不会创建任务。

<Callout type="warning">
更换负责人、取消分配或修改 issue 状态，都不会停止已经开始的执行。需要中断时，请在执行日志中停止对应的 task。
</Callout>

## 使用 CLI

```bash
multica issue assign MUL-42 --to "Agent name"
multica issue assign MUL-42 --unassign
```

脚本中建议使用 `--to-id <uuid>`，避免同名或相似名称匹配到错误对象。查看 UUID 可以运行 `multica agent list --output json` 或 `multica workspace member list --output json`。
