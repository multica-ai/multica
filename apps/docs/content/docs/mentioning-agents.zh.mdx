---
title: 通过评论调用智能体
description: 在 issue 的评论中 @智能体或继续回复，让它处理新的信息而不改变负责人。
---

import { Callout } from "fumadocs-ui/components/callout";

在评论中 @智能体，适合让它查看一条新信息、回答问题或协助当前负责人。评论会成为这次执行的起点，issue 的负责人和状态都不会改变。

## 什么时候用分配，什么时候用 @提及

| | 分配给智能体 | 在评论中 @智能体 |
|---|---|---|
| 适合 | 让智能体负责整条 issue | 请智能体处理当前评论 |
| 改变负责人 | 是 | 否 |
| 改变状态 | 否 | 否 |
| 一次选择 | 一个智能体或小队 | 可以 @多个智能体或小队 |
| 执行上下文 | 整条 issue | 整条 issue，加上触发评论 |

需要明确负责人时使用[分配](/assigning-issues)；只需要临时加入讨论时使用 @提及。

## 发送前确认会触发谁

在评论编辑器中选择智能体或小队后，输入框下方会显示触发预览：

- 哪些智能体会在发送后开始工作；
- 目标来自直接 @提及、issue 负责人还是小队 leader；
- 目标是否不可用，或你是否缺少调用权限。

你可以取消选中某个智能体，只跳过这一次触发，不需要删除评论中的名字，也不会修改智能体 Access。

评论发布后，在线运行时通常很快开始；运行时离线时，执行任务会留在队列中等待。

## 不写 @，回复也可以继续对话

Multica 会根据讨论上下文判断普通评论应该交给谁：

1. 回复智能体的评论时，继续交给该智能体。
2. 在已有智能体参与的讨论中回复时，沿用这条讨论的智能体。
3. 没有明确讨论对象时，交给 issue 的智能体负责人；负责人是小队时，由 leader 处理。

如果评论明确 @了其他对象，Multica 不会再额外使用负责人兜底。这样一条评论不会在你不知情时同时启动多个对象。

## 连续评论不会产生并行的重复执行

同一个智能体在同一条 issue 上已有等待中的执行任务时，新的评论会合并进这条任务。

如果它已经在执行，后续评论会等当前任务结束，再合并为一次跟进执行。你可以继续补充信息，不必等智能体回复后再发下一条。

一条评论中 @多个不同智能体时，每个智能体会得到自己的执行任务。多个提及最终指向同一个智能体时，只会保留一次执行。

## 只留言，不启动智能体

有两种明确的方式：

- 以 `/note` 开头：把评论作为成员之间的备注，不触发任何智能体。
- 使用 `@all`：通知工作区成员，同时关闭这条评论对 issue 负责人的自动触发。

如果触发预览中仍有智能体，也可以在发送前取消选中它。

<Callout type="info">
`@all` 不包含智能体。智能体没有收件箱；要让它工作，需要明确 @它、回复相关讨论，或把 issue 分配给它。
</Callout>

## Access 仍然有效

@提及不能绕过智能体的 Access 设置。Only me 智能体只能由其所有者调用；Specific people 只允许名单中的成员；Entire workspace 对所有工作区成员开放。

如果没有权限、智能体已归档或没有可用运行时，触发预览会说明它不会启动。评论本身仍可正常发布。

## 避免智能体互相循环调用

智能体也可以在自己的回复中 @其他智能体。A @ B、B 又 @ A 时，系统会合并同一时刻的重复执行，但不会自动判断这段协作是否应该结束。

为会互相协作的智能体写清交接条件，并要求它在没有新工作时不要继续 @对方。
