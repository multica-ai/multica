智能体是 [Multica 工作区](/workspaces)中的协作者。你可以把 issue 分配给它、在评论中 @它，或直接和它对话；它会通过绑定的运行时调用 AI 编程工具，并把进度和结果写回工作区。

智能体本身不是一个持续运行的进程。它是一套可以反复使用的身份和配置，只有收到工作时才会产生具体的[执行任务](/tasks)。

## 一个智能体记录什么

| 配置 | 作用 |
|---|---|
| **名称、头像和描述** | 让团队知道它是谁、适合做什么。描述只用于展示，不会进入执行提示。 |
| **指令** | 规定它的职责、工作方式、边界和交付要求。每次执行都会使用。 |
| **Skills** | 提供可以复用的方法、参考资料和支持文件。 |
| **运行时、模型和思考强度** | 决定使用哪台运行环境、哪款 AI 编程工具以及具体模型。 |
| **Access** | 决定哪些成员可以运行它。 |
| **执行配置** | 包括并发上限、环境变量、CLI 参数、MCP 和外部集成。 |

这些配置持续存在。更换模型或修改指令不会创建一个新智能体，也不会丢失过去的 issue、评论和执行记录。

## 智能体、运行时和执行任务

| 概念 | 回答的问题 |
|---|---|
| **智能体** | 谁来做，以及它应该如何工作？ |
| **运行时** | 在哪里、通过哪款 AI 编程工具执行？ |
| **执行任务（task）** | 这一次具体执行发生了什么？ |

一个运行时可以承载多个智能体；一个智能体也可以先后完成多条执行任务。运行时离线时，智能体的身份和历史仍在，但新的任务需要等待运行时恢复。

## 智能体如何参与工作

- **负责 issue**：把智能体设为负责人，任务入队后开始处理。
- **处理一条评论**：在评论中 @智能体，不改变 issue 的负责人。
- **继续讨论**：回复智能体的评论，新的信息会进入后续执行。
- **直接对话**：开始一段不依附于 issue 的对话。
- **参与项目和小队**：智能体可以成为项目 lead，也可以担任小队 leader 或成员。
- **通过自动化运行**：按时间、webhook 或手动触发预设工作。

智能体可以创建 issue、发表评论和修改工作状态，但它没有收件箱，也不会收到 `@all`。对智能体的 @提及是一次执行触发，不是通知。

## 看见、运行和修改是三种权限

每个智能体都有一名 owner 和一个 Access 设置：

| Access | 谁可以运行 |
|---|---|
| **Only me** | 只有智能体 owner。 |
| **Entire workspace** | 工作区中的所有成员。 |
| **Specific people** | 智能体 owner 和指定成员。 |

新建智能体默认使用 **Only me**。

<Callout type="info">
工作区 `owner` 和 `admin` 可以查看并管理所有智能体，但不能绕过 Access 运行别人的 Only me 智能体。普通成员只会看到自己拥有或有权运行的智能体。
</Callout>

三道权限分别是：

- **看见**：智能体 owner、工作区 owner/admin，以及 Access 允许的成员。
- **运行**：由 Access 决定，工作区管理员没有额外旁路。
- **修改**：智能体 owner 和工作区 owner/admin 可以修改配置；只有智能体 owner 可以修改 Access。

环境变量和 MCP 等可能包含凭据的配置有更严格的读取规则，见[创建和配置智能体](/agents-create#环境变量与凭据)。

## 在线、离线和工作状态

智能体列表会同时展示两类状态：

- **可用性**来自运行时，例如在线、离线或不稳定。
- **工作负载**来自执行任务，例如处理中、排队中或空闲。

"离线"不表示智能体被删除。已经入队的任务会等待运行时恢复；"已归档"才表示它不能再接收新工作。

## 归档和恢复

不再使用的智能体可以归档。归档后，它不会出现在日常选择器中，也不能被分配或 @提及，但历史记录仍然保留，之后可以恢复。

<Callout type="warning">
归档会取消该智能体所有尚未结束的执行任务，包括排队中和正在运行的任务。先确认没有需要保留的执行，再进行归档。
</Callout>

智能体 owner 和工作区 `owner`、`admin` 都可以归档或恢复智能体。
