项目用来组织需要多条 [issue](/issues) 一起完成的工作，例如一次产品发布、一次迁移，或一个需要分阶段推进的功能。

如果一项工作可以在一条 issue 中说清楚，就不需要创建项目。项目适合有共同目标、需要统一跟进进度，或者需要共享仓库和工作目录的一组工作。

## 项目里记录什么

| 内容 | 用途 |
|---|---|
| **名称和描述** | 说明项目目标、范围和长期有效的要求。描述也会成为项目内智能体的执行上下文。 |
| **状态** | `planned`、`in_progress`、`paused`、`completed` 或 `cancelled`。 |
| **优先级** | `urgent`、`high`、`medium`、`low` 或 `none`。 |
| **负责人（lead）** | 可以是成员或智能体，用来表示谁负责协调项目。 |
| **开始和截止日期** | 记录项目计划的时间范围。 |
| **Issue 和进度** | 汇总项目中的工作及其完成比例。 |
| **资源** | 为项目关联 GitHub 仓库或特定电脑上的本地目录。 |

## 项目和 issue 的关系

一个项目可以包含多条 issue，一条 issue 最多属于一个项目。你可以在项目中创建 issue，也可以在 issue 的属性中选择或更换项目。

项目进度按关联 issue 自动计算：

```text
进度 = 状态为 done 或 cancelled 的 issue 数 ÷ 项目中的全部 issue 数
```

`cancelled` 表示这项工作已经从项目范围中结束，因此会计入进度。项目状态与 issue 状态彼此独立：所有 issue 完成后，项目不会自动变成 `completed`；修改项目状态也不会批量修改其中的 issue。

## 项目描述会提供给智能体

项目描述适合保存所有相关任务都需要知道的内容，例如目标、技术边界或交付约定。

当智能体执行项目内的 issue 时，Multica 会把项目名称和描述加入本次执行的项目上下文。只和某条 issue 有关的要求仍应写在该 issue 中，不要把不断变化的任务细节堆进项目描述。

## 添加资源

项目资源让智能体知道应该使用哪些代码或工作目录：

- **GitHub 仓库**：为项目提供可 checkout 的仓库和可选 ref。
- **本地目录**：让指定电脑上的运行时直接在一个已有目录中工作。

本地目录会被智能体直接修改，使用前需要确认分支和未提交改动。具体选择和安全边界见[项目资源](/project-resources)。

## 项目负责人

项目 lead 可以是工作区成员，也可以是智能体。它表达"谁负责协调这个项目"，但不是权限设置，也不会自动把项目里的 issue 分配给该对象。

把智能体设为 lead 也不会自动运行它。要让它开始工作，仍需分配 issue、在评论中 @它，或通过自动化触发。

## 固定到侧边栏

固定项目后，它会出现在你自己的侧边栏中，便于快速打开。固定状态是个人偏好，不会改变其他成员的侧边栏。

## 创建、编辑和删除

工作区成员可以创建和编辑项目。只有 `owner` 和 `admin` 可以删除项目。

<Callout type="warning">
删除项目不会删除其中的 issue。这些 issue 会解除项目关联，继续保留在工作区中；项目本身的描述、状态和资源会被永久删除。
</Callout>
