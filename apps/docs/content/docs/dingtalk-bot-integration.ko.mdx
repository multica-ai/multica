import { Callout } from "fumadocs-ui/components/callout";

아무 [에이전트](/agents)나 DingTalk(钉钉) 앱 봇에 바인딩하면, 팀이 DingTalk 안에서 바로 그 에이전트를 사용할 수 있습니다 — 봇에게 개인 메시지를 보내거나, 그룹에서 `@`로 멘션하거나, `/issue`를 입력해 앱을 열지 않고도 [Multica 이슈](/issues)를 생성하세요. 바인딩은 스캔하여 생성하는 방식입니다: DingTalk으로 QR 코드 하나만 스캔하면 앱이 당신 조직의 사내 앱으로 자동 생성되고, 자격 증명도 자동으로 발급됩니다.

각 봇은 하나의 Multica 에이전트와 **일대일**로 바인딩됩니다. 두 번째 에이전트를 바인딩하면 두 번째 앱이 생성되며, 하나의 에이전트가 두 개의 봇을 갖는 일은 없습니다.

## 연동이 하는 일

| 위치 | 동작 |
|---|---|
| **에이전트 → Integrations** | 에이전트 상세 페이지에 **Integrations** 탭이 있습니다. owner와 admin에게는 여기에 **DingTalk에 연결**이 보이며, 바인딩되면 **DingTalk에 연결됨** 상태로 바뀌고 **연결 해제** 버튼과 **DingTalk에서 관리** 링크가 함께 표시됩니다. |
| **봇에게 개인 메시지** | 워크스페이스 멤버가 DingTalk에서 봇에게 직접 메시지를 보냅니다. 각 대화는 그 에이전트와의 Multica [chat](/chat) 세션이 되며, 에이전트의 답변은 메시지로 돌아옵니다. |
| **그룹에서 `@` 멘션** | 봇을 DingTalk 그룹에 추가하고 `@`로 멘션하세요. DingTalk은 로봇을 `@`로 멘션한 메시지만 봇에게 전달합니다 — 봇이 그룹의 다른 메시지를 보는 일은 없습니다. |
| **`/issue` 명령** | `/issue <제목>`(본문 추가 가능)을 보내면 워크스페이스에 새 Multica 이슈가 생성되고, 당신 이름으로 귀속됩니다. |
| **`/new` 명령** | 현재 대화 컨텍스트를 버리고 완전히 새로운 에이전트 세션에서 메시지를 처리합니다(`/reset`은 동일한 별칭입니다). |
| **`/unbind` 명령** | 당신 자신의 DingTalk 계정과 Multica의 연결을 해제합니다. |
| **🤔思考中 리액션** | 메시지가 접수되면 봇이 그 메시지에 🤔思考中("생각 중") 리액션을 달고, 답변이 도착하면 제거합니다. |

## 에이전트 바인딩하기 (owner / admin)

바인딩은 스캔하여 생성하는 방식입니다 — 복사할 앱 시크릿도, 개발자 콘솔에서 수동으로 앱을 만들 필요도 없습니다.

1. **Agents → _당신의 에이전트_**에서 에이전트를 엽니다.
2. **Integrations** 탭으로 이동해 **DingTalk에 연결**을 클릭합니다.
3. QR 코드가 나타납니다. 휴대폰에서 **DingTalk → 스캔**을 열고, 조직 안에 앱을 생성하는 것을 승인하세요. (스캔이 번거로우면 대화상자의 "DingTalk에서 열기" 링크도 사용할 수 있습니다.)
4. 승인이 완료되면 대화상자가 닫히고 에이전트에 **DingTalk에 연결됨**이 표시됩니다. 앱은 당신 조직의 사내 앱으로 생성되었고, 자격 증명은 자동으로 발급되어 저장 시 암호화됩니다.

<Callout type="info">
QR 코드는 일회용이며 짧은 시간 후에 만료됩니다. 승인하기 전에 만료되면 **다시 스캔**을 클릭해 새 코드를 받으세요. 또한 Lark 연동과 달리 스캔 자체는 당신의 DingTalk 신원을 바인딩하지 않습니다 — 봇에게 처음 메시지를 보낼 때 아래 규칙에 따라 식별되며, 조직 멤버는 보통 자동으로 매칭됩니다.
</Callout>

에이전트가 연결되면 **DingTalk에 연결** 버튼은 연결됨 상태로 바뀝니다: 한쪽에는 **연결 해제** 버튼, 다른 쪽에는 **DingTalk에서 관리** 링크가 있습니다. 링크는 [DingTalk 개발자 콘솔의 사내 앱 목록](https://open-dev.dingtalk.com/fe/app#/corp/app)을 엽니다. 스캔으로 생성된 앱이 거기에 있으며, 권한·자격 증명·릴리스는 모두 콘솔에서 관리합니다. (스캔 생성 방식은 앱별 딥링크를 반환하지 않으므로, 진입점은 앱 목록 페이지입니다.)

## 봇 사용하기 (멤버)

### 신원 확인: 보통은 완전 자동

봇은 "DingTalk 속의 당신"이 어느 Multica 계정에 대응하는지 알아야 합니다. 식별은 두 단계로 진행됩니다.

1. **조직 디렉터리를 통한 자동 매칭.** 당신이 그 조직의 멤버라면, 봇은 기업 디렉터리를 통해 당신의 DingTalk 신원을 Multica 계정과 연결합니다 — DingTalk으로 Multica에 로그인한 계정은 unionid로, 이메일로 가입한 계정은 디렉터리 프로필의 이메일로 매칭됩니다. 매칭되면 자동으로 바인딩되며, 아무것도 클릭할 필요가 없습니다.
2. **폴백: 바인딩 링크.** 자동 매칭이 안 되면 봇이 "계정을 연결하세요"라는 메시지로 답합니다. 링크를 열고 Multica에 로그인하면 당신의 DingTalk 계정이 Multica 멤버십에 연결됩니다. 링크는 일회용이며 **15분** 동안 유효합니다. 만료되면 봇에게 메시지를 다시 보내 새 링크를 받으세요.

<Callout type="info">
자동 매칭은 기업 디렉터리 조회에 의존합니다. 스캔으로 생성된 앱에 연락처(通讯录) 읽기 전용 권한이 없으면 자동 매칭이 실패하고 바인딩 링크로 폴백합니다 — DingTalk 개발자 콘솔에서 해당 앱에 관련 읽기 전용 권한을 부여하면 복구됩니다.
</Callout>

<Callout type="warning">
**워크스페이스 멤버**만 봇을 사용할 수 있습니다. 멤버가 아니면 봇은 응답하지 않으며, 메시지는 폐기됩니다(감사 목적의 폐기 사유만 기록되고 내용은 저장하지 않습니다).
</Callout>

### 대화와 명령(`/issue`, `/new`, `/unbind`)

- **무엇이든 에이전트에게 물어보기** — 봇에게 개인 메시지를 보내거나 그룹에서 `@`로 멘션하세요. 이 대화는 일반적인 에이전트 chat 세션입니다. 기다리는 동안 당신의 메시지에는 🤔思考中 리액션이 붙어 있고, 에이전트의 답변은 메시지로 돌아옵니다.
- **이슈 생성** — `/issue 로그인 리디렉션 수정`을 보내면 Multica가 워크스페이스에 그 이슈를 생성하고 "✅ 已创建 MUL-123 — 로그인 리디렉션 수정"이라는 확인 메시지를 보냅니다. 제목 뒤에 줄을 더 추가하면 설명이 됩니다. `/issue`만 단독으로 보내면 직전 메시지가 제목으로 사용됩니다. 명령은 소문자 `/issue`여야 합니다.
- **새로 시작** — `/new`(바로 다음 질문을 이어 써도 됩니다. 예: `/new 배포 이야기를 해보자`)를 보내면 에이전트가 이전 대화 컨텍스트를 버리고 새 세션에서 답합니다. 내용 없이 `/new`만 보내면 즉시 확인 메시지를 받고 다음 메시지부터 적용됩니다. `/reset`도 동일하게 사용할 수 있습니다.
- **연결 해제** — 인자 없이 `/unbind`를 한 줄로 보내면, 봇이 당신의 DingTalk 계정과 Multica의 연결을 해제하고 채팅에서 확인해 줍니다. 다음 메시지부터는 다시 신원 확인이 이루어집니다.

에이전트가 **오프라인**(런타임이 연결되지 않음)이거나 **보관됨** 상태라면, 봇은 메시지를 조용히 폐기하는 대신 짧은 상태 안내로 답합니다. 에이전트가 바쁘면(동시 작업 한도에 도달) 메시지가 대기열에 들어갔고 자리가 나는 대로 처리된다고 알려줍니다.

## 관리 및 연결 해제

워크스페이스 전체 관리는 **설정 → Integrations → DingTalk**에 있습니다.

- **연결된 봇**은 워크스페이스 내 모든 봇과 각 봇이 바인딩된 에이전트를 나열합니다. 이 목록은 모든 멤버에게 보입니다.
- **연결 해제**는 **owner / admin 전용**입니다. 해제하면 봇이 DingTalk 메시지 수신을 멈춥니다. 설치 기록은 감사용으로 유지되며("해지됨" 배지와 함께 표시), 이후 같은 에이전트에서 다시 스캔해 재설치할 수 있습니다.

## 권한

- **바인딩 / 연결 해제**에는 워크스페이스 **owner** 또는 **admin**이 필요합니다. 멤버에게는 연결된 봇 목록은 보이지만 바인딩이나 연결 해제 컨트롤은 보이지 않습니다.
- **봇과 대화하기**에는 DingTalk 신원이 확인된(자동 매칭 또는 수동 바인딩) 워크스페이스 멤버여야 합니다. 그 외의 사람은 모두 폐기됩니다.
- 연동은 폐기된 메시지의 본문을 절대 저장하지 않으며 — 감사용 폐기 사유만 기록합니다.

## 자체 호스팅 설정

Multica Cloud에서는 연동이 이미 사용 가능합니다 — 이 섹션은 건너뛰세요.

자체 호스팅의 경우, **at-rest 암호화 키를 설정하기 전까지 DingTalk은 꺼져 있습니다**. 이 키는 각 앱의 client secret이 데이터베이스에 닿기 전에 암호화합니다.

1. 32바이트 키를 생성해 API 서버에 설정합니다.

   ```bash
   openssl rand -base64 32
   ```

   ```dotenv
   MULTICA_DINGTALK_SECRET_KEY=<base64-encoded 32-byte key>
   ```

2. API를 재시작하세요. 키가 설정되기 전까지 **설정 → Integrations → DingTalk**에는 "DingTalk 연동이 활성화되지 않았습니다" 안내가 표시되고, **DingTalk에 연결** 진입점은 숨겨진 채로 유지됩니다.

<Callout type="info">
나머지 환경 변수는 모두 선택적 오버라이드입니다. 일반 운영에서는 설정하지 않은 채로 두세요.

- `DINGTALK_OPENAPI_BASE` — 신형 DingTalk OpenAPI 호스트(기본값 `https://api.dingtalk.com`).
- `DINGTALK_OAPI_BASE` — 레거시 DingTalk OAPI 호스트(기본값 `https://oapi.dingtalk.com`).
- `MULTICA_DINGTALK_REGISTRATION_BASE_URL` — 스캔 생성(device flow) 등록 엔드포인트 오버라이드. 스테이징이나 mock용.
- `MULTICA_DINGTALK_REGISTRATION_SOURCE` — DingTalk이 할당하는 파트너 소스 라벨. 비워 두면 전송되지 않습니다.
</Callout>

## 문제 해결

### QR 대화상자의 설치 오류

설치 대화상자에 표시되는 오류는 다음 사유(`error_reason`)에 대응합니다.

| 메시지 | 코드 | 의미와 대처 |
|---|---|---|
| QR 코드가 만료되었습니다. 다시 스캔하세요. | `expired` | QR은 시간 제한이 있으며 승인 전에 만료되었습니다. **다시 스캔**을 클릭하세요. |
| DingTalk에서 승인이 실패했거나 취소되었습니다. | `install_failed` | DingTalk에서 취소했거나, DingTalk 쪽에서 앱 생성에 실패했습니다. 다시 시도하세요. 대화상자의 작은 회색 글씨에 DingTalk 쪽의 구체적인 사유가 표시됩니다. |
| DingTalk이 설치 요청을 거부했습니다. 다시 시도하세요. | `dingtalk_protocol_error` | DingTalk API가 프로토콜 수준 오류를 반환했습니다. 다시 시도하고, 계속되면 서버에서 DingTalk까지의 네트워크 연결을 확인하세요. |
| 앱은 생성되었지만 자격 증명 검증에 실패했습니다. 다시 시도하세요. | `credentials_check_failed` | 앱은 존재하지만, 새 자격 증명으로 액세스 토큰을 얻는 데 실패했습니다. 다시 스캔해 설치하세요. |
| 설치 정보를 저장하지 못했습니다. 이 에이전트에 이미 다른 앱이 연결되어 있을 수 있습니다. | `installation_conflict` | 에이전트에 이미 설치가 있습니다. **설정 → Integrations → DingTalk**에서 이전 것을 해제한 뒤 다시 바인딩하세요. |
| 설치 세션이 만료되었거나 유실되었습니다. 다시 스캔해 처음부터 진행하세요. | `session_lost` | 설치 세션은 서버 메모리에만 존재합니다(약 30분). 재시작이나 시간 초과로 유실됩니다. 다시 스캔하세요. |
| 이 워크스페이스에서 DingTalk 봇을 설치할 권한이 없습니다. | `forbidden` | 설치 도중 당신의 역할이 바뀌었습니다. owner / admin에게 계속 진행을 요청하세요. |
| 설치에 실패했습니다. 다시 시도하세요. | `internal_error` | 예기치 못한 서버 측 오류입니다. 다시 시도하고 서버 로그를 확인하세요. |

### 설정 페이지 상태

| 상태 | 의미 |
|---|---|
| **DingTalk 연동이 활성화되지 않았습니다** | 서버에 `MULTICA_DINGTALK_SECRET_KEY`가 없습니다. 위의 자체 호스팅 설정을 따르세요. |
| **DingTalk 봇 설치 기능 곧 제공 예정** | 암호화 키는 설정되어 있지만, 이 서버 빌드에는 스캔 생성 플로가 활성화되어 있지 않습니다(이유는 시작 로그에 있습니다). |

### 대화 문제

- **그룹에서 봇이 반응하지 않음** — 그룹 메시지는 로봇을 `@`로 멘션해야 합니다. DingTalk은 멘션되지 않은 메시지를 봇에게 전달하지 않습니다.
- **개인 메시지에도 아무 답이 없음** — 그 워크스페이스의 멤버가 아닐 수 있습니다. 메시지는 폐기됩니다(폐기 사유만 기록됩니다).
- **바인딩 링크가 만료됨** — 봇에게 아무 메시지나 보내면 새 링크를 받습니다.
- **조직 멤버인데 자동 매칭이 안 됨** — 앱에 연락처 읽기 전용 권한을 부여하거나(DingTalk 개발자 콘솔 → 해당 앱 → 권한 관리), 바인딩 링크를 사용하세요.
- **"⚠️ 本次处理失败了"(처리 실패) 메시지를 받음** — 그 실행이 실패했습니다. 잠시 후 다시 시도하거나 다르게 표현해 보세요. 오류 세부 정보는 Multica에서 해당 에이전트의 작업에서 확인할 수 있습니다.
