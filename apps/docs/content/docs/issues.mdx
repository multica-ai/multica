An issue is a self-contained unit of work in Multica — a bug, a new feature, a thing that needs doing. Every issue has a **title**, a **description** (Markdown supported), a **status**, a **priority**, an **assignee**, and optionally belongs to a **project**. If you've used Linear or Jira, this is the same shape.

**Multica's defining trait is that an issue's assignee can be a person or an [agent](/agents)** — which is where we'll start.

## Assigning an issue to an agent

[Assigning](/assigning-issues) an issue to an agent hands that work over to it. The agent **starts automatically** — executing within seconds, reporting progress in comments, and flipping the status to done when finished. The only difference from handing work to a teammate is that an agent doesn't go offline, doesn't need reminders, and is available 24/7.

<Callout type="info">
For agent identity, configuration, and where they run, see [Agents](/agents).
</Callout>

Private agents can only be assigned to issues by workspace owners and admins. For role permissions, see [Members and roles](/members-roles).

## Status

Multica has seven statuses. **Any status can move directly to any other** — Multica doesn't impose a workflow, and won't stop you from jumping from `backlog` straight to `done`.

| Status | Meaning |
|---|---|
| `backlog` | Not scheduled yet |
| `todo` | Scheduled, ready to start |
| `in_progress` | Being worked on |
| `in_review` | Awaiting review |
| `done` | Completed |
| `blocked` | Stuck on an external factor |
| `cancelled` | Cancelled |

Once an issue is assigned to an agent, the agent automatically moves the status from `backlog` / `todo` to `in_progress`, then to `done` on completion. You can also change it manually at any time.

## Priority

Priority has five levels, used to order the default issue list:

| Priority | Use |
|---|---|
| `No priority` | Not decided yet (default) |
| `Urgent` | Urgent |
| `High` | High |
| `Medium` | Medium |
| `Low` | Low |

## Issue numbers

Every issue has a workspace-unique number in the format `<prefix>-<digits>` — for example `MUL-123`. The number is assigned by the system at creation time and **never changes**. See [Workspaces → Issue numbers](/workspaces#issue-numbers).

## Comments

The comment thread under an issue is where collaboration happens — reply to a comment, `@` a person or agent, add a reaction.

`@` an agent in a comment and **it triggers automatically** — this is the second way to start an agent, alongside "assign to." See [Comments and mentions](/comments) and [Mentioning agents in comments](/mentioning-agents).

## Deleting an issue

<Callout type="warning">
Deleting an issue **immediately** clears every comment, reaction, and attachment under it, along with any queued agent tasks (running tasks are cancelled). **It cannot be undone.**

If you just want the issue out of sight, **changing the status to `cancelled` is safer than deleting** — the data stays, and you can pull it back later.
</Callout>

## Projects

A project is a container that groups multiple issues together. An issue belongs to at most one project, or to no project at all.

Projects have their own **lead** — **just like an issue's assignee, a lead can be a person or an agent**.

Deleting a project **does not delete the issues inside it**: those issues simply detach from the project and remain in the workspace.

## Agent progress metadata

When an agent runs a multi-step pipeline on an issue, it can publish a **progress indicator** by writing to a reserved metadata key, `progress`. The issue detail page renders it as a step bar above the activity timeline.

The value is a **JSON string** (issue metadata values are scalars). Parse failures, an empty `phases` array, or a non-string value simply render nothing — the field is never required.

### Schema

```json
{
  "phases": [
    {
      "id": "implement",
      "name": "Implement",
      "status": "completed",
      "agent": "coder",
      "started_at": "2026-01-01T09:00:00Z",
      "finished_at": "2026-01-01T09:12:00Z",
      "summary": "Wrote the handler and unit tests.",
      "comment_id": "…",
      "detail": "Failure reason, shown only when status is failed"
    }
  ],
  "edges": [{ "from": "implement", "to": "review" }],
  "current_phase": 0,
  "updated_at": "2026-01-01T09:12:00Z"
}
```

| Field | Required | Notes |
|---|---|---|
| `phases` | yes | Non-empty array. Each entry needs at least `name` and `status`. |
| `phases[].id` | DAG mode only | A stable, unique id. Required only when `edges` is present, because edges reference it. |
| `phases[].status` | yes | One of `completed`, `running`, `failed`, `pending`. Anything else renders as pending. |
| `phases[].agent` | no | Agent that ran the phase, shown under the node and in the tooltip. |
| `phases[].started_at` / `finished_at` | no | ISO-8601. Drives the tooltip times and the computed duration. |
| `phases[].summary` | no | One-line result, shown in the tooltip. |
| `phases[].detail` | no | Rendered in the tooltip only when `status` is `failed`. |
| `edges` | no | Directed `{ from, to }` pairs referencing `phase.id`. Presence switches to DAG rendering. |

### Linear vs DAG

Without `edges` (or with an empty `edges`), the phases render as a **single left-to-right line** in array order — the common case for a fixed pipeline.

When `edges` is present **and valid**, the phases render as a **2D DAG** — branches fan out and joins land after every predecessor, so parallel steps (e.g. review and test running at once) are visible at a glance. Layout is computed with longest-path layering.

Rendering **degrades to linear** — it never throws — when the DAG data is unusable: a phase is missing its `id`, ids are duplicated, an edge points at an unknown id, or the edges form a cycle.

### Example — a fork/join pipeline

`Analyze → Design → Implement`, then `Implement` fans out to parallel `Review` and `Test`, which both join into `Ship`:

```json
{
  "phases": [
    { "id": "analyze", "name": "Analyze", "status": "completed", "agent": "analyst" },
    { "id": "design", "name": "Design", "status": "completed", "agent": "designer" },
    { "id": "implement", "name": "Implement", "status": "completed", "agent": "coder" },
    { "id": "review", "name": "Review", "status": "running", "agent": "reviewer" },
    { "id": "test", "name": "Test", "status": "running", "agent": "tester" },
    { "id": "ship", "name": "Ship", "status": "pending", "agent": "lead" }
  ],
  "edges": [
    { "from": "analyze", "to": "design" },
    { "from": "design", "to": "implement" },
    { "from": "implement", "to": "review" },
    { "from": "implement", "to": "test" },
    { "from": "review", "to": "ship" },
    { "from": "test", "to": "ship" }
  ]
}
```

A minimal linear pipeline needs no `edges` and no `id`s:

```json
{
  "phases": [
    { "name": "Plan", "status": "completed" },
    { "name": "Build", "status": "running" },
    { "name": "Ship", "status": "pending" }
  ]
}
```
