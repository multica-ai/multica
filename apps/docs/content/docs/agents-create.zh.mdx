创建智能体前，需要先有一个可用的[运行时](/daemon-runtimes)。运行时决定这名智能体使用哪台电脑和哪款 AI 编程工具；智能体则保存它长期使用的身份、指令和能力。

在工作区的 **智能体** 页面选择"新建智能体"开始。

## 选择起点

创建页面提供三种方式：

| 方式 | 适合什么情况 |
|---|---|
| **从空白开始** | 已经知道智能体的职责，想自己填写每项配置。 |
| **使用模板** | 从已有指令和 skill 开始，再按团队需要修改。 |
| **通过 AI 创建** | 先描述目标，让 Agent Builder 追问关键信息并生成草稿。 |

通过 AI 创建需要一个在线运行时来完成 Builder 对话。无论选择哪种方式，创建前都可以检查并修改最终配置。

## 最少需要填写什么

创建智能体只要求两项：

- **名称**：在当前工作区中必须唯一。
- **运行时**：选择实际执行任务的环境。

其他字段可以先使用默认值，创建后再调整。新智能体默认只有创建者可以运行。

## 描述和指令不要混用

**描述**是写给团队看的简短介绍，只显示在智能体列表和详情中，不会进入 AI 编程工具的提示。

**指令**会在每次执行时提供给智能体，应写清：

- 它负责什么、不负责什么；
- 收到工作后先检查什么；
- 允许修改哪些内容；
- 结果应该如何提交；
- 什么时候需要向成员确认。

例如：

```text
你负责审查前端 Pull Request。

先阅读改动和相关测试，只检查：
- React 与 TypeScript 的正确性
- 可访问性
- 与现有组件模式是否一致

不要直接修改代码。把问题按严重程度写在 issue 评论中；
没有阻塞问题时，明确说明可以合并。
```

相比堆叠角色形容词，可执行的步骤、边界和输出要求更有用。

## 添加 skill

创建时可以从工作区选择一个或多个 [skill](/skills)。使用模板时，模板自带的 skill 会和你额外选择的 skill 一起加入智能体。

Skill 适合保存可以跨智能体复用的方法和资料；只属于这名智能体的长期要求应写在指令中。

## 选择运行时、模型和思考强度

运行时已经对应一款 AI 编程工具。选择运行时后，可以继续选择该工具支持的模型和思考强度：

- 留空时，使用运行时或本地 CLI 的默认配置。
- 设置模型后，该智能体之后领取任务时会使用这个覆盖值。
- 某些运行时自行管理模型，因此不会显示模型选择。

不同工具支持的模型、会话恢复、skill 和 MCP 能力不同，见[AI 编程工具对照](/providers)。

## 设置 Access

Access 控制谁能分配、@提及或直接对话这名智能体：

| Access | 含义 |
|---|---|
| **Only me** | 只有你可以运行。默认选项。 |
| **Entire workspace** | 所有工作区成员都可以运行。 |
| **Specific people** | 只有你和选中的成员可以运行。 |

只有智能体 owner 可以修改 Access。工作区 `owner` 和 `admin` 可以管理其他配置，但不能借管理员身份运行未授权的智能体。

## 创建后可以继续配置什么

打开智能体详情页，可以继续调整：

| 配置 | 作用 |
|---|---|
| **并发上限** | 同一名智能体最多同时执行多少条任务，默认是 6；达到上限后继续排队。 |
| **环境变量** | 在 AI 编程工具启动时注入变量。 |
| **自定义参数** | 逐项追加到 AI 编程工具的 CLI 参数中。 |
| **MCP** | 为支持的 AI 编程工具提供 MCP server 配置。 |
| **集成** | 连接这名智能体可以使用的外部服务。 |

运行时本身也可能设置总并发上限，实际并发取两层限制中较小的一项。

修改配置不会改变已经运行中的执行任务。后续任务被运行时领取时，会使用当时保存的智能体配置。

## 环境变量与凭据

环境变量适合放这名智能体执行时需要的有限权限凭据，例如只读 API key 或单一 scope 的 token。

<Callout type="warning">
`custom_env` 的值保存在 Multica 服务器数据库中，不属于"只留在本机"的数据。普通智能体列表和详情接口不会返回明文；只有工作区 `owner`、`admin` 可以解锁和修改，并且每次读取或修改都会留下审计记录。运行中的智能体不能调用管理接口读取其他智能体的变量。
</Callout>

不要使用生产数据库的管理员密码或其他高价值长期凭据。为智能体单独创建最小权限凭据，并定期轮换。

`PATH`、`HOME` 和 `MULTICA_*` 等运行环境关键变量不能通过这里覆盖。

## 自定义参数和 MCP

自定义参数按数组逐项传给 AI 编程工具，不经过 shell 展开。参数名称是否有效由对应工具决定；不要在这里重复设置已经有独立字段的模型配置。

MCP 配置可能包含 token，同样应按敏感信息处理。只有支持托管 MCP 配置的运行时会显示这一项；不支持的工具不会因为保存了配置就自动获得 MCP 能力。

## 复制智能体

复制会带上名称之外的大部分工作配置，例如指令、skill、模型和自定义参数。出于安全考虑，环境变量的值不会复制，需要在新智能体中重新设置。

复制完成后检查运行时和 Access，避免把原智能体的执行位置或共享范围直接带到不合适的场景。

## 使用 CLI 创建

```bash
multica agent create \
  --name "Frontend Reviewer" \
  --runtime-id <runtime-id> \
  --description "审查前端 Pull Request" \
  --instructions "先阅读改动和测试，只在 issue 评论中提交审查结论。"
```

处理环境变量或 MCP 凭据时，优先使用 stdin 或权限受限的文件，不要把明文直接放进命令行历史。完整参数见 [CLI 命令速查](/cli)。
