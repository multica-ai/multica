**작업**은 모든 [에이전트](/agents) 실행의 단위입니다 — [에이전트에게 이슈를 할당](/assigning-issues)하거나, [댓글에서 에이전트를 @-멘션](/mentioning-agents)하거나, [채팅](/chat)에서 메시지를 보내거나, [오토파일럿](/autopilots)이 예약된 시각에 실행되면 모두 작업이 생성됩니다. Multica는 이를 대기열에 넣고, [데몬](/daemon-runtimes)이 가져가 해당 [AI 코딩 도구](/providers)에 넘긴 다음, 완료되면 결과를 서버에 다시 기록합니다.

작업과 [이슈](/issues)는 서로 다른 두 객체입니다. 하나의 이슈는 여러 번 할당되거나 @-멘션되거나 수동으로 재실행될 수 있으며 — 그때마다 **새로운** 작업이 생성됩니다.

## 작업이 거치는 상태

<Mermaid chart={`
graph LR
    Q["Queued<br/>queued"] -->|daemon claims| D["Dispatched<br/>dispatched"]
    D -->|agent starts| R["Running<br/>running"]
    D -->|needs local directory| W["Waiting<br/>waiting_local_directory"]
    W --> R
    R -->|success| C["Completed<br/>completed"]
    R -->|error or timeout| F["Failed<br/>failed"]
    Q -->|user cancels| X["Cancelled<br/>cancelled"]
    D -->|user cancels| X
    R -->|user cancels| X
    F -.retryable reason.-> Q
`} />

- **Queued(대기 중)** — 작업이 영구 대기열에 들어가, 자신의 런타임 데몬이 점유하기를 기다리는 상태. 런타임이 오프라인이라고 해서 작업이 실패하지는 않으며 — 여기서 그저 기다립니다(최대 2시간; 아래 참고)
- **Dispatched(파견됨)** — 데몬이 작업을 점유하고 AI 코딩 도구를 시작하는 중
- **Waiting(대기)** (`waiting_local_directory`) — 작업에 필요한 로컬 프로젝트 디렉터리가 점유한 기기에 아직 준비되지 않은 상태; 디렉터리가 준비될 때까지 보류됩니다
- **Running(실행 중)** — AI 코딩 도구가 실제로 작업을 수행하는 중
- **Completed(완료)** — 성공적으로 끝났으며, 산출물(댓글, 코드 커밋, 상태 변경)이 서버에 다시 기록됩니다
- **Failed(실패)** — 오류 또는 타임아웃으로 중단됨; 실패 사유가 재시도 가능한 경우 작업은 자동으로 `queued` 상태로 돌아가 다시 시도됩니다
- **Cancelled(취소됨)** — 사용자가 취소한 경우

내부에서 볼 수 있는 상태가 하나 더 있습니다: **`deferred`** — 이미 실행 중인 에이전트에게 댓글이 도착한 경우, 병렬 실행을 새로 시작하는 대신 Multica가 이를 보류했다가 현재 실행이 끝났을 때 하나의 후속 실행으로 합칩니다. [에이전트 멘션하기](/mentioning-agents#what-happens-when-you--multiple-agents)를 참고하세요.

## 작업이 타임아웃되면 일어나는 일

Multica 서버는 30초마다 스캔합니다. 세 종류의 타임아웃이 실패를 유발합니다:

| 상황 | 타임아웃 | 실패 사유 |
|---|---|---|
| 대기열에 있으나 아무도 점유하지 않음(예: 런타임이 계속 오프라인) | **2시간** | `queued_expired` — 재시도 안 함 |
| 파견되었으나 시작되지 않음(데몬이 가져갔지만 AI 도구를 실행하지 않음) | **5분** | `timeout` — 재시도함 |
| 너무 오래 실행됨 | **2.5시간** | `timeout` — 재시도함 |

관련된 런타임 누락 검사는 [데몬과 런타임 → 런타임이 오프라인으로 표시되는 시점](/daemon-runtimes#when-a-runtime-is-marked-offline)을 참고하세요.

## 어떤 실패가 자동으로 재시도되고 어떤 실패는 그렇지 않은지

실패는 두 가지 범주로 나뉩니다: **재시도 가능**과 **재시도 불가**.

**재시도 가능**(Multica가 자동으로 다시 대기열에 넣음):

- `runtime_offline` — 작업이 파견된 후 데몬이 사라짐
- `runtime_recovery` — 데몬이 충돌 후 재시작되어 끝내지 못한 작업을 회수함
- `timeout` — 런타임 또는 파견 타임아웃
- `codex_semantic_inactivity` — Codex 세션이 실행 도중 조용해짐
- `provider_network` — AI 도구가 제공자와 통신하다 일시적 네트워크 오류를 만남

**재시도 불가**(작업이 실패 상태로 유지됨):

- `agent_error` — AI 코딩 도구 자체가 오류를 보고함(API 오류, 할당량 초과, 내부 버그). 근본적인 문제는 재시도하지 않습니다 — 무한 반복될 수 있기 때문입니다. (위의 `provider_network`는 이 계열에서 유일한 예외입니다.)
- `queued_expired` — 2시간 안에 아무도 작업을 점유하지 않음; 여전히 부재중인 같은 런타임으로 재시도해봐야 소용없습니다. 런타임을 고친 뒤 수동으로 재실행하세요.

자동 재시도에는 두 가지 추가 조건도 있습니다:

1. **최대 2회 시도** — 원본 1회 + 재시도 1회. 예외는 `provider_network`로, 일시적 네트워크 장애는 다음 시도에서 해소되는 경우가 많으므로 최대 **3회**까지 시도합니다(즉시 재시도 1회, 그다음 짧은 지연 후 마지막 1회). 마지막으로 허용된 시도까지 실패하면 더 이상 재시도하지 않습니다.
2. **이슈 및 채팅으로 트리거된 작업에만 해당** — 오토파일럿으로 트리거된 작업은 자동으로 재시도되지 **않습니다**.

<Callout type="warning">
**오토파일럿 작업은 자동으로 재시도되지 않습니다** — 의도된 설계입니다. 오토파일럿은 자체적인 실행 주기(예: 매일)를 갖고 있으며, 실패 시 자동 재시도가 일어나면 다음 예약 실행과 겹치게 됩니다. 실패 후 즉시 재실행이 필요하다면 수동 재실행을 사용하세요(다음 섹션).

**오토파일럿 작업이 실패했음을 알 수 있는 방법**: [인박스](/inbox)에 알림이 도착하고, 연관된 이슈의 상태가 `in_progress`에서 다시 `todo`로 되돌아갑니다. [오토파일럿](/autopilots) 페이지에서도 오토파일럿별 최근 실행 결과를 확인할 수 있습니다.
</Callout>

## 수동 재실행 vs. 자동 재시도

**수동 재실행**은 CLI 또는 API(`POST /api/issues/{id}/rerun`)에서 직접 트리거하는 것입니다:

```bash
multica issue rerun <issue-id>
```

동작:

- 기본적으로 이슈의 **현재 에이전트 담당자**를 대상으로 합니다 — 이전 작업을 누가 실행했는지와 무관하게 재실행이 현재 할당을 따르도록 하고 싶을 때 유용합니다.
- 실행 로그의 특정 행에 있는 재시도 버튼은 해당 행의 작업 ID를 함께 전송하므로, 재실행은 **현재 담당자가 아니라 바로 그 작업을 실행했던 에이전트**를 대상으로 합니다. 덕분에 스쿼드 워커, 병렬 @-멘션 에이전트, 또는 재할당으로 인해 에이전트가 교체된 행에 대해서도 행 단위 재시도가 의미를 갖게 됩니다.
- 이 이슈에 대한 대상 에이전트의 대기 중이거나 실행 중인 작업을 **취소합니다**(있는 경우). 같은 이슈에서 다른 에이전트가 소유한 작업(예: 병렬 @-멘션 실행)은 그대로 둡니다.
- **완전히 새로운** 작업을 생성합니다 — 원본 작업이 시도 횟수 상한에 도달했더라도 시도 횟수가 1로 재설정됩니다.
- **작업 디렉터리를 재사용하며, 세션 재개 여부는 어디에서 트리거했는지에 따라 달라집니다:**
  - **실행 로그의 행 단위 재시도**(task ID 포함): 해당 실행의 작업 디렉터리를 재사용하므로 디스크에 이미 기록된 파일이 보존됩니다. 그 실행의 세션도 재개합니다 — 단, 실패가 대화를 오염시킨 경우(컨텍스트 오버플로, 반복 한도, invalid request, Codex 시맨틱 무활동, 에이전트 폴백)에는 재사용한 작업 디렉터리 위에서 새 세션을 시작합니다. 원본 디렉터리가 사라진 경우(GC됨, 다른 런타임에 있음, 또는 기록되기 전에 실패)에는 새 작업 디렉터리로 폴백합니다. 따라서 네트워크 순간 장애, provider의 5xx / 속도 제한, 잠시 오프라인이 된 런타임, 이미 수정한 auth/할당량/설정 오류 같은 **일시적 실패**는 처음부터 다시 시작하지 않고 중단된 지점부터 이어집니다.
  - **task ID 없는 CLI / API 재실행**: 이어갈 특정 실행이 없으므로 **새 세션 + 새 작업 디렉터리**로 시작합니다.

비교:

| 항목 | 자동 재시도 | 수동 재실행 |
|---|---|---|
| 트리거 | 시스템, 실패 사유 기반 | 사용자, 수동 |
| 상한 | 2회 시도(`provider_network`은 3회) | 제한 없음 |
| 적용 가능한 소스 | 이슈, 채팅 | 에이전트 담당자가 있는 이슈 |
| 선택되는 에이전트 | 실패한 작업과 동일한 에이전트 | 소스 작업의 에이전트(UI 행 단위 재시도) 또는 이슈의 현재 담당자(CLI / task_id 없음) |
| 작업 디렉터리 | 재사용(이전 파일에서 이어감) | 행 단위 재시도: 아직 남아 있으면 재사용; CLI / task_id 없음: 새로 시작 |
| 세션 상속 | 예(이전 세션 재개) | 행 단위 재시도: 실패가 대화를 오염시키지 않는 한 재개; CLI / task_id 없음: 새로 시작 |

## 실행 지켜보기: 실행 로그

모든 이슈의 상세 페이지에는 **실행 로그**가 있습니다 — 그 이슈에 대한 모든 실행을 진행 중인 것과 지난 것 모두 나열한 목록입니다. 각 행은 누가 실행했는지(에이전트), 무엇이 트리거했는지(트리거 댓글의 텍스트, "오토파일럿 실행", "재시도 #N" 등), 현재 상태, 그리고 — 실행 중인 행의 경우 — 실시간 경과 타이머를 보여줍니다. 여러 댓글을 아우르는 병합된 실행에는 "N개 댓글 포함"이라고 표시됩니다. 지난 실패 행에는 실패 사유가 표시됩니다.

각 행에서 할 수 있는 일:

- **트랜스크립트 보기(View transcript)** — 에이전트의 트랜스크립트를 엽니다. 실행이 진행 중이면 실시간으로 스트리밍됩니다. 트랜스크립트 대화상자에는 귀속 정보("on behalf of …" — 이 실행이 책임지는 대상 멤버)도 함께 표시됩니다.
- **작업 중지(Stop task)** — 대기 중이거나 실행 중인 작업을 취소합니다(단일 에이전트의 실행을 멈추는 방법은 이것이며, 담당자나 상태를 바꾸는 것으로는 멈추지 않습니다).
- **작업 재시도(Retry task)** — 실패했거나 취소된 행에서, 바로 그 행을 실행했던 에이전트로 새 실행을 시작합니다(자세한 동작은 위의 [수동 재실행 vs. 자동 재시도](#manual-rerun-vs-automatic-retry) 참고).

## 실패한 작업이 이슈 상태에 미치는 영향

재시도 대기가 없는 상태로 실행이 실패하고, 이슈가 `in_progress`에 머물러 있으며 그 이슈에 다른 활성 실행이 없다면, **이슈의 상태가 자동으로 `todo`로 되돌아갑니다** — 그래서 보드를 열면 "이건 다시 봐야겠다"는 것을 즉시 알 수 있습니다. (다른 에이전트가 여전히 그 이슈에서 활성 실행을 갖고 있다면 상태는 그대로 둡니다.) [이슈와 프로젝트](/issues)를 참고하세요.

## 작업이 이전 컨텍스트에서 이어질 수 있는지

가능합니다 — AI 코딩 도구가 세션 재개를 지원하는 한 그렇습니다.

Multica는 작업 중에 세션 ID를 **두 번** 고정합니다: 시작 시 한 번(AI 도구가 첫 번째 시스템 메시지를 반환할 때), 종료 시 한 번(완료 또는 실패 시). 첫 번째는 데몬이 실행 도중 충돌하더라도 복구할 수 있게 해주고, 두 번째는 다음 **자동 재시도**와 **실행 로그의 행 단위 재시도**에서 재사용되어, 그 ID를 다시 전달함으로써 에이전트가 이전 대화와 파일 상태를 이어받게 합니다 — 단, 실패가 대화를 오염시킨 경우에는 작업 디렉터리만 재사용하고 새 세션을 시작합니다. **task ID 없는 CLI / API 재실행**은 새 세션 + 새 작업 디렉터리로 시작합니다 — [수동 재실행 vs. 자동 재시도](#manual-rerun-vs-automatic-retry)를 참고하세요.

하지만 **실제로 어떤 AI 코딩 도구가 이를 지원하는지**는 크게 다릅니다 — 도구별 상태는 [제공자 매트릭스 → 세션 재개](/providers#session-resumption-who-really-supports-it)를 참고하세요.
