Multica connects to a self-hosted [Forgejo](https://forgejo.org) instance per workspace. Once connected, any pull request whose branch name, title, or body contains an issue identifier (for example `MUL-123`) is **auto-linked** to that [issue](/issues), appears under **Pull requests** in the issue sidebar, and — when merged with a closing keyword — moves the issue to **Done**. Commit statuses (CI) for the PR's head commit show as a checks bar on the card.

Forgejo runs alongside GitHub: a workspace can use either or both.

<Callout type="info">
Unlike GitHub, Forgejo has no "App" model. Each workspace stores its own instance URL plus an access token, and registers a webhook on its Forgejo repository or organisation. The token and webhook secret are encrypted at rest.
</Callout>

## Prerequisites (server)

Set a base64-encoded 32-byte key so the server can encrypt stored Forgejo credentials. Without it, the connect form is disabled.

```sh
# generate once
openssl rand -base64 32
```

```sh
MULTICA_FORGEJO_SECRET_KEY=<base64 32-byte key>
```

Set `MULTICA_PUBLIC_URL` to the server's public base URL so Multica can show a ready-to-paste webhook URL. Without it, the UI shows only the webhook path and you prepend your own origin.

## Connect a workspace

1. In Forgejo, create an access token under **Settings → Applications**. It needs read access to the repositories you want mirrored.
2. In Multica, open **Settings → Integrations → Forgejo**.
3. Enter the instance URL (for example `https://forgejo.example.com`) and the access token, then **Connect**. Multica validates the token against the instance before saving.
4. Copy the **webhook URL** and **webhook secret** shown after connecting.

<Callout type="warn">
The webhook secret is shown only once. Copy it before leaving the page. Reconnecting the same instance rotates the token and secret.
</Callout>

## Register the webhook in Forgejo

On the Forgejo repository (or organisation, to cover every repo) open **Settings → Webhooks → Add Webhook → Forgejo/Gitea** and set:

- **Target URL**: the webhook URL from the previous step.
- **HTTP method**: `POST`, **content type** `application/json`.
- **Secret**: the webhook secret from the previous step.
- **Trigger events**: select **Pull Request**. To mirror CI, also select **Commit Status** (sometimes labelled "Status").

Multica verifies every delivery's `X-Gitea-Signature` against the stored secret, so a webhook without the matching secret is rejected.

## What gets mirrored

- **Pull requests** — open, closed, merged, and draft state, author, branch, and diff stats.
- **Issue links** — identifiers in the PR title, body, or branch link the PR to the issue. A closing keyword (`Closes`/`Fixes`/`Resolves MUL-123`) on a merged PR moves the issue to **Done** once no linked PR is still open.
- **CI** — commit statuses for the PR's head commit aggregate into a passed/failed/pending checks bar.

## Agents opening pull requests

Creating PRs needs no Forgejo configuration in Multica. Agents check out repositories in the runtime and push branches and open PRs using the runtime host's own Git credentials. To let agents work against Forgejo, ensure the daemon host can authenticate to it — for example an SSH deploy key, or a token in the host's Git credential helper — and add the Forgejo repository URL as usual. Repository checkout works with any Git URL, so no Forgejo-specific wiring is required.
