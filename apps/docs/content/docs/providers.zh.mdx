Multica 不会替代你使用的 AI 编程工具。运行时会调用电脑上已经安装并登录的工具，再把执行进度和结果写回 issue。

如果你已经在使用其中一款，通常可以直接沿用。选择时主要看四件事：电脑上是否已经安装、账号可以使用哪些模型、是否需要由 Multica 配置 MCP，以及团队要复用哪些 skill。

安装方法见 [安装 AI 编程工具](/install-agent-runtime)。

| Claude Code | `claude` | ✓ | ✓ | `.claude/skills/` |

| Grok | `grok` | ✓ | ✓ | `.grok/skills/` |

"Multica 管理 MCP" 表示你可以在智能体配置中填写 MCP server，由 Multica 在执行前传给对应工具。它不代表该工具本身没有其他 MCP 配置方式。

## 模型从哪里来

创建智能体时，模型列表来自对应的运行时。有些工具提供固定的模型名称，有些会根据本机配置、登录账号和订阅权益返回可用模型。

不选择模型时，Multica 使用该工具的默认设置。列表为空时，先确认运行时在线并且工具已经登录，再刷新模型列表。

<Callout type="info">
不同工具可能使用不同的模型名称。更换 AI 编程工具后，请重新选择模型，不要直接沿用原来的值。
</Callout>

## 会话如何继续

这些工具都支持在后续执行中继续已有会话，但前提是原来的会话仍然存在，并且任务由能够访问它的运行时执行。

Multica 会保存工具返回的会话标识。下一次在同一 issue 中继续工作时，运行时会尝试恢复这段会话；无法恢复时，会创建一个新会话继续执行。Pi 比较特殊，它保存的是本地会话文件路径，因此更依赖原来的电脑和文件。

有关重试和会话复用的规则，见 [执行任务](/tasks#手动重试)。

## MCP 配置

Multica 可以为以下工具配置 MCP：

- Claude Code、CodeBuddy、Codex、Cursor Agent
- Grok、Hermes、Kimi CLI、Kiro CLI
- OpenCode、OpenClaw、Qoder CLI、Trae CLI

Antigravity、GitHub Copilot CLI、DevEco Code 和 Pi 暂不读取智能体中的 MCP 配置，因此创建智能体时不会显示这一项。

<Callout type="warning">
MCP server 的命令、参数和环境变量会随智能体配置保存在服务器端。不要把不需要共享的本地凭据直接写入配置。
</Callout>

## Skill 如何交给工具

执行开始前，Multica 会把智能体绑定的 skill 写入上表对应的位置，让工具通过自己的原生机制发现它们。

仓库里原本存在的 skill 不会被覆盖。如果名称冲突，Multica 会为注入的副本使用另一个目录名；执行结束后，只清理自己创建的文件。

Codex、Hermes 和 OpenClaw 的处理稍有不同：

- Codex 使用单次执行的 `CODEX_HOME`，不会把 skill 写进系统级 Codex 目录。
- Hermes 使用单次执行的 `HERMES_HOME`；已有的本地配置仍然可以被读取，但绑定的 skill 与会话数据按任务隔离。
- OpenClaw 使用当前任务工作目录下的 `skills/`，并由单次执行配置把该目录设为工作区。

如何创建和绑定 skill，见 [Skills](/skills)。
