The Multica CLI mirrors almost everything the Web UI can do (create [issues](/issues), assign [agents](/agents), start the [daemon](/daemon-runtimes), and more). This page lists every top-level command with a one-line description. For the full set of flags and examples, run `cs-workflow <command> --help`.

## Getting authenticated

Run this the first time you use the CLI to obtain a **personal access token (PAT)**:

```bash
cs-workflow login
```

Your browser opens automatically. After you approve in the web app, the CLI saves the PAT (prefixed with `mul_`) to `~/.multica/config.json`. Every subsequent command authenticates with that PAT.

<Callout type="tip">
For CI or headless environments, skip the browser flow: create a PAT in the web app under **Settings → Personal Access Tokens**, then run `cs-workflow login --token <mul_...>` to supply it directly.
</Callout>

For the difference between token types, see [Authentication and tokens](/auth-tokens).

## Auth and setup

| Command | Purpose |
|---|---|
| `cs-workflow login` | Log in and save a PAT |
| `cs-workflow auth status` | Show current login status, user, and workspace |
| `cs-workflow auth logout` | Clear the local PAT |
| `cs-workflow setup cloud` | One-shot setup for Multica Cloud (login + install daemon) |
| `cs-workflow setup self-host` | One-shot setup for a self-hosted backend |

## Workspaces and members

| Command | Purpose |
|---|---|
| `cs-workflow workspace list` | List every workspace you can access |
| `cs-workflow workspace get <slug>` | Show details for one workspace |
| `cs-workflow workspace member list` | List members of the current workspace |
| `cs-workflow workspace update <id> --name "..." [--description "..."] [--context "..."] [--issue-prefix "..."]` | Update workspace metadata (admin/owner). Long fields accept `--description-stdin` / `--context-stdin`. |

## Issues and projects

<Callout type="info">
`list` commands (`cs-workflow issue list`, `autopilot list`, `project list`, etc.) print short, copy-paste-ready IDs by default — issue keys like `MUL-123` for issues, short UUID prefixes for the rest. The `<id>` argument on the follow-up commands below accepts either the short ID or the full UUID, so the typical flow is `cs-workflow issue list` → copy the key → `cs-workflow issue get MUL-123`. Pass `--full-id` to a list command when you need the canonical UUID.
</Callout>

| Command | Purpose |
|---|---|
| `cs-workflow issue list` | List issues (prints copy-paste-ready issue keys) |
| `cs-workflow issue get <id>` | Show a single issue (accepts an issue key or a UUID) |
| `cs-workflow issue create --title "..."` | Create a new issue |
| `cs-workflow issue update <id> ...` | Update an issue (status, priority, assignee, etc.) |
| `cs-workflow issue assign <id> --agent <slug>` | Assign to an agent (triggers a task immediately) |
| `cs-workflow issue status <id> --set <status>` | Shortcut to change status |
| `cs-workflow issue search <query>` | Keyword search |
| `cs-workflow issue runs <id>` | Show agent runs on an issue |
| `cs-workflow issue rerun <id>` | Re-enqueue a fresh task for the issue's current agent assignee |
| `cs-workflow issue comment <id> ...` | Nested: view / post comments |
| `cs-workflow issue subscriber <id> ...` | Nested: subscribe / unsubscribe |
| `cs-workflow project list/get/create/update/delete/status` | Project CRUD |

## Agents and skills

| Command | Purpose |
|---|---|
| `cs-workflow agent list` | List the workspace's agents |
| `cs-workflow agent get <slug>` | Show an agent's configuration |
| `cs-workflow agent create ...` | Create an agent |
| `cs-workflow agent update <slug> ...` | Update an agent |
| `cs-workflow agent archive <slug>` | Archive |
| `cs-workflow agent restore <slug>` | Restore an archived agent |
| `cs-workflow agent tasks <slug>` | Show an agent's task history |
| `cs-workflow agent skills ...` | Nested: attach / detach skills |
| `cs-workflow skill list/get/create/update/delete` | Skill CRUD |
| `cs-workflow skill import ...` | Import a skill from GitHub, ClawHub, or the local machine |
| `cs-workflow skill files ...` | Nested: manage a skill's files |

## Squads

| Command | Purpose |
|---|---|
| `cs-workflow squad list` | List squads in the workspace |
| `cs-workflow squad get <id>` | Show a single squad |
| `cs-workflow squad create --name "..." --leader <agent>` | Create a squad (owner / admin) |
| `cs-workflow squad update <id> ...` | Update name, description, instructions, leader, or avatar |
| `cs-workflow squad delete <id>` | Archive (soft-delete) — transfers assigned issues to the leader |
| `cs-workflow squad member list/add/remove <squad-id>` | Manage squad members |
| `cs-workflow squad activity <issue-id> <action\|no_action\|failed> --reason "..."` | Used by squad leader agents to record an evaluation per turn |

See [Squads](/squads) for the full model.

| Command | Purpose |
|---|---|

## Daemon and runtimes

| Command | Purpose |
|---|---|
| `cs-workflow daemon start` | Start the daemon (background by default; add `--foreground` to run in the foreground) |
| `cs-workflow daemon stop` | Stop the daemon |
| `cs-workflow daemon restart` | Restart the daemon |
| `cs-workflow daemon status` | Check whether the daemon is online and its concurrency |
| `cs-workflow daemon logs` | View daemon logs |
| `cs-workflow runtime list` | List runtimes in the current workspace |
| `cs-workflow runtime usage` | Show resource usage |
| `cs-workflow runtime activity` | Recent activity log |
| `cs-workflow runtime update <id> ...` | Update a runtime's configuration |

## Miscellaneous

| Command | Purpose |
|---|---|
| `cs-workflow repo checkout <url>` | Clone a repo locally for agents to use |
| `cs-workflow config` | View or edit local CLI configuration |
| `cs-workflow version` | Print the CLI version |
| `cs-workflow update` | Upgrade the CLI to the latest release |
| `cs-workflow attachment download <id>` | Download an attachment from an issue or comment |
