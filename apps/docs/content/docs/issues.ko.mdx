이슈는 Multica에서 독립적인 작업 단위입니다 — 버그, 새 기능, 처리해야 할 어떤 일이든 될 수 있습니다. 모든 이슈에는 **제목**, **설명**(Markdown 지원), **상태**, **우선순위**, **담당자**가 있으며, 선택적으로 **프로젝트**에 속할 수 있습니다. Linear나 Jira를 사용해 보셨다면 동일한 형태입니다.

**Multica의 가장 큰 특징은 이슈의 담당자가 사람일 수도, [에이전트](/agents)일 수도 있다는 점입니다** — 여기서부터 시작하겠습니다.

## 에이전트에게 이슈 할당하기

이슈를 에이전트에게 [할당](/assigning-issues)하면 그 작업을 에이전트에게 넘기는 것입니다. 에이전트는 **자동으로 시작합니다** — 실행이 즉시 대기열에 들어가고, 런타임이 온라인이면 보통 몇 초 안에 시작하며(오프라인이면 대기열에서 기다립니다), 댓글로 진행 상황을 보고합니다. 결과물을 하나 완성하면, 관례상 이슈를 `in_review`로 옮기고 사람이 마무리하도록 둡니다 — 실행 하나가 끝났다고 해서 이슈가 완료된 것은 아닙니다(자세한 내용은 아래 [상태](#status) 참고).

<Callout type="info">
에이전트의 정체성, 구성, 실행 위치에 대해서는 [에이전트](/agents)를 참고하세요.
</Callout>

특정 에이전트를 할당할 수 있는지는 그 에이전트의 **접근(Access)** 설정(나만 / 워크스페이스 전체 / 특정 사용자)에 따라 결정됩니다 — [에이전트 → 누가 에이전트를 실행할 수 있나](/agents#who-can-run-an-agent)를 참고하세요. 역할 권한에 대해서는 [멤버와 역할](/members-roles)을 참고하세요.

## 상태

Multica에는 일곱 가지 상태가 있습니다. **어떤 상태든 다른 어떤 상태로도 바로 이동할 수 있습니다** — Multica는 워크플로를 강제하지 않으며, `backlog`에서 곧바로 `done`으로 건너뛰어도 막지 않습니다.

| 상태 | 의미 |
|---|---|
| `backlog` | 보류됨. `backlog` 상태에서 에이전트를 할당해도 **시작되지 않습니다** — 이슈가 `backlog`을 벗어날 때(예: `todo`로) 비로소 실행이 대기열에 들어갑니다 |
| `todo` | 일정이 잡혔고 시작할 준비됨 — 여기서 에이전트를 할당하면 즉시 트리거됩니다 |
| `in_progress` | 작업 중 |
| `in_review` | 결과물이 사람의 리뷰를 기다리는 중 |
| `done` | 완료됨 |
| `blocked` | 외부 요인으로 막힘 |
| `cancelled` | 취소됨 |

**에이전트가 이슈를 작업할 때 상태가 어떻게 바뀌는가.** 이 전환은 플랫폼이 강제하는 것이 아니라, 에이전트가 (자신의 지침에 따라) 작업하면서 스스로 수행합니다 — 일반적인 흐름은 이렇습니다: 에이전트가 시작할 때 이슈를 `in_progress`로 옮기고, 여러분이 살펴볼 무언가를 전달했을 때 `in_review`로 옮깁니다. **실행이 끝났다고 자동으로 `done`이 되는 것은 아닙니다** — `in_review` → `done`으로 마무리하는 것은 리뷰 후 여러분의 몫입니다. 언제든지 직접 상태를 변경할 수도 있습니다.

플랫폼 자체가 상태를 자동으로 바꾸는 경우는 정확히 두 가지뿐입니다:

- 에이전트의 실행이 **실패**하면(재시도 대기가 없고 그 이슈에 다른 실행도 없는 경우), 멈춰 있는 `in_progress`가 `todo`로 되돌아가 다시 봐야 함을 알 수 있게 합니다.
- close 의도(`Closes MUL-123`)가 있는 연결된 GitHub PR이 **머지되면**, 이슈가 `done`으로 이동합니다.

## 우선순위

우선순위에는 다섯 단계가 있으며, 기본 이슈 목록의 정렬에 사용됩니다:

| 우선순위 | 용도 |
|---|---|
| `No priority` | 아직 결정되지 않음 (기본값) |
| `Urgent` | 긴급 |
| `High` | 높음 |
| `Medium` | 중간 |
| `Low` | 낮음 |

## 이슈 번호

모든 이슈에는 워크스페이스 내에서 고유한 번호가 `<prefix>-<digits>` 형식으로 부여됩니다 — 예를 들어 `MUL-123`처럼요. 번호는 생성 시점에 시스템이 부여하며 **절대 변하지 않습니다**. [워크스페이스 → 이슈 번호](/workspaces#issue-numbers)를 참고하세요.

## 댓글

이슈 아래의 댓글 스레드는 협업이 일어나는 곳입니다 — 댓글에 답글을 달고, 사람이나 에이전트를 `@`로 멘션하고, 반응을 추가할 수 있습니다.

댓글에서 에이전트를 `@`로 멘션하면 **자동으로 트리거됩니다** — 이것은 "할당"과 더불어 에이전트를 시작하는 두 번째 방법입니다. [댓글과 멘션](/comments)과 [댓글에서 에이전트 멘션하기](/mentioning-agents)를 참고하세요.

## 이슈 삭제하기

<Callout type="warning">
이슈를 삭제하면 그 아래의 모든 댓글, 반응, 첨부 파일과 대기 중인 에이전트 작업이 **즉시** 삭제됩니다(실행 중인 작업은 취소됩니다). **되돌릴 수 없습니다.**

이슈를 단지 보이지 않게 하고 싶을 뿐이라면, **상태를 `cancelled`로 변경하는 것이 삭제보다 안전합니다** — 데이터가 남아 있어 나중에 다시 가져올 수 있습니다.
</Callout>

## 프로젝트

프로젝트는 여러 이슈를 하나로 묶는 컨테이너입니다. 이슈는 최대 하나의 프로젝트에 속하거나, 아예 어떤 프로젝트에도 속하지 않을 수 있습니다.

프로젝트에는 자체 **리더**가 있습니다 — **이슈의 담당자와 마찬가지로, 리더도 사람일 수도, 에이전트일 수도 있습니다**.

프로젝트를 삭제해도 **그 안의 이슈는 삭제되지 않습니다**: 해당 이슈들은 단지 프로젝트에서 분리되어 워크스페이스에 그대로 남습니다.
