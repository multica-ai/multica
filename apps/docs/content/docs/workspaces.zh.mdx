工作区是 Multica 最上层的协作范围。成员、issue、项目、智能体、skill 和执行记录都属于一个工作区；切换工作区后，你看到的是另一套独立的数据和配置。

## 工作区里有什么

| 内容 | 包含什么 |
|---|---|
| **工作** | Issue、项目、评论、附件和执行记录 |
| **团队** | 成员、角色和邀请 |
| **智能体** | 智能体配置、skill、运行时和自动化 |
| **共享设置** | 工作区说明、上下文、标签、自定义属性和集成 |

工作区的**上下文**会提供给其中的智能体，适合写团队背景、产品约定或长期有效的工作要求。具体任务仍应写在对应的 issue 中。

## 创建工作区

创建时只需要填写：

- **名称**：成员看到的工作区名称，之后可以修改。
- **工作区 URL**：地址中的 slug，例如 `multica.ai/my-team` 里的 `my-team`。

Slug 只能包含小写字母、数字和连字符，不能与已有工作区或系统保留地址重复。创建后不能修改。

Multica 会根据工作区名称自动生成 issue 前缀，并把创建者设为 `owner`。例如前缀为 `MUL` 时，issue 编号会显示为 `MUL-1`、`MUL-2`。

## 修改工作区设置

`owner` 和 `admin` 可以在设置中修改工作区名称、图标、描述、上下文和 issue 前缀。普通 `member` 可以查看这些信息，但不能修改。

<Callout type="warning">
修改 issue 前缀会让所有 issue 使用新的显示编号。例如 `MUL-5` 会变为 `NEW-5`。旧编号仍可能留在 PR、分支名、文档或聊天记录中，因此不要把修改前缀当作日常操作。
</Callout>

Issue 后面的数字由工作区独立维护。创建和删除其他工作区都不会影响这里的编号。

## 切换和离开工作区

一个账号可以加入多个工作区。切换工作区只会改变当前查看和操作的范围，不会移动任何数据。

成员可以主动离开工作区，离开后需要重新接受邀请才能回来。最后一名 `owner` 不能离开；应先把另一位成员提升为 `owner`。

离开工作区还会停用该成员拥有的运行时，归档绑定在这些运行时上的智能体，并取消尚未结束的执行任务。离开前应先把仍在使用的智能体和运行时交接好。

## 删除工作区

只有 `owner` 可以删除工作区。删除会永久移除工作区及其中的工作、成员、智能体配置和执行记录，无法撤销。

<Callout type="warning">
删除前请先导出需要保留的数据。只是暂时不参与时，应选择离开工作区，不要删除整个工作区。
</Callout>
