在[快速上手](/cloud-quickstart)中，你把一个 issue 分配给了 Multica Helper。Multica 随后创建一条执行任务，由连接电脑上的运行时领取。运行时调用本地 AI 编程工具，并把进度和结果写回原来的 issue。

## 一次执行如何完成

<AgentWorkFlowDiagram />

1. **Issue 提供上下文。** Issue 保存工作说明、讨论和负责人。分配给智能体后，系统会使用该智能体的指令、模型、skill 和运行时配置。
2. **Multica 创建执行任务。** 执行任务进入队列；没有在线运行时时，它会在队列中等待。
3. **运行时领取任务。** 连接电脑上的在线运行时领取任务，并调用智能体配置的 AI 编程工具。
4. **AI 编程工具在本地执行。** 工具读取工作目录、运行命令并生成结果。
5. **结果写回 issue。** 进度、评论和结果显示在 issue 的时间线与执行日志中。

## 五个核心概念

| 概念 | 它是什么 |
|---|---|
| **工作区** | 团队共同工作的范围。Issue、智能体、项目、skill 和执行记录都属于某个工作区。 |
| **Issue** | 一项要完成的工作，以及围绕它积累的描述、讨论、状态和历史。 |
| **智能体** | 可复用的身份和配置，包括名称、指令、模型、skill、访问范围和运行时。智能体本身不是一个正在运行的进程。 |
| **执行任务（task）** | 智能体的一次具体执行。一个 issue 可以先后产生多条执行任务，每次都会单独记录。 |
| **运行时** | 执行任务实际在哪里完成。当前由连接到 Multica 的电脑和其中一款 AI 编程工具组成。 |

## 什么留在 Multica，什么留在电脑

| Multica | 连接的电脑 |
|---|---|
| 工作区、issue、评论和状态 | AI 编程工具及其凭据 |
| 智能体配置与 skill | 代码目录和本地文件 |
| 执行任务的状态、执行记录和结果 | 实际的文件修改与命令执行 |

无论使用 Multica Cloud 还是自托管，AI 编程工具都在连接的电脑上运行。Multica 保存团队协作所需的上下文和执行记录，但服务器不运行这些工具。

<Callout type="warning">
有一个例外：如果你主动把 secret 保存到智能体的 `custom_env`，它会存放在 Multica 服务器端，再在运行时执行时传给智能体。请按服务器侧凭据管理它，并限制访问范围。
</Callout>

## 怎么让智能体开始工作

| 方式 | 适合什么情况 |
|---|---|
| **分配 issue** | 把智能体设为负责人，让它接手这项工作。 |
| **在评论中 @智能体** | 请智能体处理一条具体评论，不改变 issue 的负责人。 |
| **直接对话** | 开始一段不依附于 issue 的独立对话。 |
| **自动化** | 按时间、webhook 或手动触发预先设置的工作。 |

## 一次执行结束，不代表 issue 已完成

执行日志显示"已完成"，只表示这一次执行已经结束。Issue 仍然可以继续讨论、补充要求或再次触发智能体。Issue 是否完成，以实际工作进展和它的状态为准。
