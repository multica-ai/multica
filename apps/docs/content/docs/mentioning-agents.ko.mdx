[댓글](/comments)에서 [에이전트](/agents)를 `@`-멘션하는 것은 더 가벼운 트리거입니다 — **담당자 변경도, 상태 변경도 없이**, 그저 에이전트가 현재 [이슈](/issues)를 살펴보도록 살짝 알리는 것입니다. [**할당**](/assigning-issues)(에이전트를 담당자로 만들고 이슈를 넘기는 것)과 비교하면, @-멘션은 "이 부분 좀 봐줘", "다른 관점으로 분석해줘", "잠깐 끌어들여서 같이 논의하자" 같은 상황에 잘 맞습니다.

## 댓글에서 에이전트 멘션하기

멤버를 멘션하는 것과 동일합니다 — `@`를 입력해 피커를 열고 에이전트를 선택하세요. 댓글이 게시되면 Multica는 멘션된 각 에이전트에게 **그 댓글**을 트리거 컨텍스트로 삼아 즉시 `task`를 대기열에 넣습니다. 에이전트가 작업을 받으면 다음을 읽을 수 있습니다.

- 이슈 전체(설명 + 모든 과거 댓글)
- 트리거 댓글 자체 — 이번 실행의 시작점으로

`@mention` Markdown 문법, 피커, 그리고 `@all` 의미는 [**댓글**](/comments)에서 다룹니다.

<Callout type="info">
**댓글에서 [스쿼드](/squads)도 `@`-멘션할 수 있습니다.** 동일한 피커가 멤버 및 에이전트와 함께 스쿼드도 보여줍니다. 스쿼드를 선택하면 `[@SquadName](mention://squad/<uuid>)`이 삽입되고, 스쿼드의 **리더 에이전트**가 응답을 조율하도록 트리거됩니다 — 담당자와 상태는 그대로 유지됩니다.
</Callout>

## 할당과 어떻게 다른가

둘 다 에이전트를 일하게 만들지만, 동작 방식은 완전히 다릅니다.

| 항목 | 할당 | @-멘션 |
|---|---|---|
| `assignee` 변경 | ✓ | ✗ |
| `status` 변경 | ✗ | ✗ |
| `task` 대기열 추가 | 즉시(백로그가 아닌 경우) | 즉시 |
| 트리거 댓글 ID | 선택 사항 | 항상 현재 댓글을 포함 |
| 한 번의 동작당 대상 에이전트 | 1명(담당자 한 명) | 다수(댓글 하나에서 여러 명을 @ 가능) |
| 우선순위 | 이슈에서 상속 | 이슈에서 상속 |

판단 기준은 간단합니다. **에이전트가 "지금부터 이 이슈를 책임지길" 원하면 할당을, "현재 컨텍스트를 한번 살펴보길" 원하면 @-멘션을 사용하세요.**

## 여러 에이전트를 @ 하면 어떻게 되는가

댓글 하나에서 **서로 다른** 여러 에이전트를 @-멘션하면, 각 에이전트는 자신의 런타임에서 독립적인 `task`를 대기열에 받습니다 — **서로를 막지 않고 병렬로 실행됩니다**. 같은 실행 에이전트로 귀결되는 멘션(예: `@AgentA`와, 리더가 A인 `@Squad`)은 하나의 `task`로 합쳐집니다.

**같은 에이전트에 대한 연속된 댓글은 쌓이지 않고 병합됩니다.** 각 에이전트는 이슈당 대기 중인 `task`를 최대 하나만 가지며, Multica는 후속 댓글을 거기에 접어 넣습니다:

- 이 이슈에서 에이전트의 이전 `task`가 아직 **`queued`**(아직 가져가지 않음) 상태라면, 새 댓글은 **그 같은 task에 병합됩니다** — 하나의 실행이 두 댓글을 모두 다룹니다. 실행 로그에는 "N개 댓글 포함"으로 표시됩니다.
- 에이전트가 이미 **실행 중**이라면, 새 댓글은 **연기됩니다**: 두 번째 병렬 실행은 시작되지 않습니다. 현재 실행이 끝나면, Multica는 실행 도중 도착한 댓글을 확인하고 그것들을 모두 아우르는 **하나의** 후속 실행을 대기열에 넣습니다.

따라서 댓글마다 실행이 하나씩 생기는 것을 걱정하지 않고 후속 지시를 계속 입력할 수 있습니다 — 아무것도 누락되지 않고, 아무것도 두 번 실행되지 않습니다.

<Callout type="info">
**댓글을 편집하면 다시 트리거됩니다.** 편집으로 내용이 바뀌면, Multica는 이전 버전이 트리거한 아직 끝나지 않은 실행을 취소하고 새 텍스트를 기준으로 다시 라우팅합니다 — 그래서 깜빡한 `@agent`를 편집으로 추가하면 그 에이전트가 **깨어납니다**. 편집하는 동안에도 (아래의) 트리거 미리 보기가 표시됩니다.
</Callout>

## 일반 댓글은 어떤 에이전트를 깨우는가

아무도 `@`할 필요가 없습니다. 명시적인 `@agent` / `@squad` 없이 멤버가 작성한 일반 댓글은 다음 우선순위에 따라 자동으로 라우팅됩니다:

1. **에이전트의 댓글에 답글** → 그 에이전트가 트리거됩니다. 에이전트와의 대화는 매번 다시 `@`하지 않아도 계속 이어집니다.
2. **스레드 안에서 답글** → 그 스레드의 확립된 주인 — 스레드 루트가 라우팅되었던 에이전트(또는 루트에서 명시적으로 `@`된 첫 번째 에이전트).
3. **그 외** → 이슈의 **담당자**가 폴백입니다: 할당된 에이전트는 직접 트리거되고, 할당된 [스쿼드](/squads)는 그 리더를 깨웁니다.

세 가지는 결코 자동 라우팅되지 않습니다: `@all`이 포함된 댓글(에이전트 라우팅을 완전히 억제함), 멤버만 `@`하는 댓글, 그리고 `/note`로 시작하는 댓글(아무도 트리거하지 않는 자기 메모).

## 컴포저의 트리거 미리 보기

보내기를 누르기 전에, 컴포저는 **트리거 칩**을 보여줍니다 — 이 댓글이 어떤 에이전트를 시작시킬지("보낼 때 시작됨"(Will start when sent))와 그 이유(담당자 / @멘션 / 스쿼드)를 정확히 나타냅니다. 작성한 내용을 바꾸지 않고도, 칩의 체크를 해제하면 이번에는 그 에이전트를 시작시키지 않습니다("이번에는 시작 안 함"(Won't start this time)).

칩의 툴팁은 접속 상태도 보여줍니다: 온라인 에이전트에는 "바로 시작됨"(Starts right away), 오프라인 에이전트에는 "지금 오프라인 — 온라인이 되면 시작"(Offline now — starts once online) — 오프라인 에이전트도 여전히 task를 받으며, 단지 대기열에서 기다릴 뿐입니다. 빨간 칩은 트리거가 완전히 차단되었음을 뜻합니다: 그 에이전트를 실행할 권한이 없거나, 보관되었거나 사용할 수 없거나, 런타임이 아예 없는 경우입니다.

## `@all`은 어떤 에이전트도 트리거하지 않는다

`@all`로 전체를 호출할 때, **워크스페이스 멤버만 인박스에 들어가며 — 에이전트는 `@all` 확장에 포함되지 않습니다.** 이는 의도된 설계입니다. 에이전트는 인박스 알림을 받지 않으므로 `@all`은 에이전트에게 아무 의미가 없습니다. 에이전트를 일하게 하려면 이름으로 직접 멘션하세요.

`@all`은 "에이전트에게 도달하지 않는다"는 것에서 한 걸음 더 나아갑니다: 그 댓글에 대해 위에서 설명한 **자동 라우팅도 꺼버립니다** — 이슈의 담당 에이전트가 깨어나지 *않습니다*. 어떤 실행도 시작하지 않고 사람에게만 이야기하고 싶을 때 사용하세요.

## 에이전트 간 멘션이 폭주하지 않는 이유

에이전트는 실행 중에 댓글을 게시할 수 있으며, 그 댓글에는 자기 자신을 포함한 `@mention`이 들어갈 수 있습니다. "자기 멘션은 무시한다"는 식의 포괄적 규칙은 없습니다; 루프를 억제하는 것은 위의 **병합 동작**입니다: 이미 이슈에 대기 중이거나 실행 중인 `task`를 가진 에이전트는 두 번째 병렬 실행을 받지 않으며 — 후속 요청은 하나의 다음 실행으로 합쳐집니다.

이것은 완충 장치일 뿐, 완전한 차단은 아닙니다. 에이전트 A가 에이전트 B를 @-멘션하는 것은 정상적으로 동작하며, 그 후 B가 응답에서 A를 @-멘션하면 A가 다시 실행됩니다 — **에이전트들이 서로를 멘션하는 순환은 계속됩니다**. 에이전트 지침을 작성할 때, 끝에 `@`를 남기지 않고 대화를 끝내도록(침묵도 유효한 응답임) 일러두고, 상호 멘션 루프를 만들지 않도록 주의하세요.
