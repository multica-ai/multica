# 技术栈和工程结构

后端开发默认从这里进入，用于快速确认技术栈、子系统边界、数据库环境规则和依赖方向。

## 技术栈基线

- 语言与运行时：`Java 21`
- 构建体系：`Maven` 多模块（父工程：`pom.xml`）
- 框架基线：`Spring Boot 3.3.13`
- 微服务与服务调用：`Apache Dubbo RPC`
- 配置中心与注册发现：`Nacos`
- Web 与参数处理：`Spring MVC`
- 数据访问：`MyBatis-Plus`
- 数据库迁移：`Liquibase`
- 服务器数据库：`MySQL`
- 本地开发数据库：内存数据库
- 流程编排：`LiteFlow 2.15.2`
- 消息队列：`RabbitMQ / AMQP`
- 定时任务：`XXL-JOB`
- 事务管理：`Spring TransactionTemplate`
- 参数校验：`Jakarta Validation`
- JSON 处理：`fastjson2`
- 配置加密：`Jasypt`
- 日志与链路观测：`SkyWalking`
- 测试基线：`JUnit 5 + Spring Boot Test`
- 子系统形态：`prod`（HTTP/用户端）+ `manage`（HTTP/管理端）+ `runtime`（RPC/运行时）

## 模块边界

- `prod` 子系统（用户侧）：`obita-llm-prod/backend/` - 提供用户侧 API
- `manage` 子系统（管理侧）：`obita-llm-manage/backend/` - 提供管理侧 API 和统一认证入口
- `runtime` 子系统（运行时）：`obita-llm-runtime/` - 原子运行时服务，通过 Dubbo RPC 被 prod/manage 调用
- 共享能力：`backend-common/` - 统一认证组件

约束：

- `prod`、`manage` 和 `runtime` 的 `domain`、`application`、`infrastructure` 必须隔离，不共享领域模型、领域服务和仓储实现。
- `prod` 和 `manage` 通过 Dubbo RPC 调用 `runtime` 的接口。
- 跨子系统共享仅通过明确契约（例如 `backend-runtime-rpc-api`）或 `backend-common` 中的通用能力完成，不允许跨边界直接耦合基础设施实现。

## 数据库环境硬规则

- 本地启动使用 H2 文件数据库（`~/obita-config_db`），通过 Spring Boot `sql.init` 自动执行 `schema.sql` 初始化。
- 服务器环境默认使用 MySQL，不复用本地 H2 配置。
- 每一次数据库结构变更，必须同步更新 `schema.sql` 和相关 guidance / cleanup 约束，不能只改其中一处。
- cleanup 必须确认数据库结构变更前后与代码清理逻辑一致，避免遗留旧表、旧字段、旧查询路径或失效兼容逻辑。

## 开发入口

- 后端任务执行顺序：`AGENT.md/CLAUDE.md` -> 本文档 -> 对应子系统目录。
- 启动命令：
  - `./obita-llm-prod/backend/scripts/dev.sh` (prod-backend, 端口 8081)
  - `./obita-llm-manage/backend/scripts/dev.sh` (manage-backend, 端口 8080)
  - `./obita-llm-runtime/scripts/dev-runtime.sh` (runtime, 端口 20880)
  - `./frontend/scripts/dev.sh` (前端, 端口 5173)
- 一键启动所有服务：
  - `./start.sh` (默认启动 frontend, prod, manage, runtime)
- 测试命令：
  - `make admin-test`
  - `make runtime-test`
  - `make backend-test`
