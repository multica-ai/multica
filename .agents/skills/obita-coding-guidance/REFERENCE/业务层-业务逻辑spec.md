### 编码要求

1. 业务规则约束分析
    * 明确规则约束：分析每个功能在实现过程中需要满足的规则约束（如权限控制、业务逻辑、规则条件等），这是实现业务逻辑的前提。
2. 代码实现要求
   * 优先使用已有依赖：在实现过程中，优先复用已有的类或工具方法，避免重复造轮子。
   * 对象转换方法：如果涉及对象之间的转换，请编写专门的转换方法，确保字段映射完整，不要遗漏字段。禁止使用 BeanUtils 或类似工具。
   * 路径规范：参考__工程结构.md__，将代码文件放置到合适的路径下，优先选择已存在的路径，确保模块划分清晰。
   * 注释标准：所有代码文件必须添加符合 JavaDoc 标准的注释，包括类注释、方法注释以及关键逻辑的行内注释。
   * 参数要求：确保含义清晰
     * 入参规范：入参超过 3 个时必须使用封装后的类型作为入参，比如相关实体的DO类型，避免使用Map类型。
     * 出参规范：出参必须要有意义，避免使用void类型（优先使用boolean类型），禁止使用 xxxResult 等额外二次封装。
3. 编码规范
   * 命名规范：
     * 变量名、方法名和类名应具有语义化，避免使用无意义的缩写，DO、DTO、VO等专有词汇保持大写。
     * 接口类名以 DomainServiceI 作后缀，实现类名以 DomainServiceImpl 作后缀。
   * 代码格式：遵循团队统一的代码格式规范（如缩进、空格、换行等）。
   * 注释清晰：对于复杂的逻辑，务必添加清晰的注释说明其目的和实现方式。
4. 编码质量
   * 性能与健壮性：
       * 确保代码具有良好的性能，避免不必要的计算或查询。
       * 添加必要的异常处理逻辑，以提升代码的健壮性。
   * DRY原则：不要重复代码，避免逻辑、校验、数据的代码重复，合理抽取公共代码。
   * 可读性：
       * 确保代码具有良好的易读性、简单性以及快速返回原则。
   * 异常处理：
     * 区分系统异常和业务异常，优先复用已定义的异常类型，新定义异常类型时继承RuntimeException（包含错误码和错误信息属性）
     * 业务逻辑异常务必使用明确的业务错误码和错误信息，避免使用系统默认的错误码和错误信息。
     * 错误码规范：区分业务异常、系统异常错误码，优先复用已定义的错误码和错误信息
       * 业务异常：以BIZ开头，例如：BIZ_ACCOUNT_NOT_EXIST，账户ID不存在 
       * 系统异常：以SYS开头
     * 任何异常信息都要直接使用错误码和错误信息 throw 出去，禁止字符串拼接错误码、错误信息，不要 catch。如带有错误码、错误信息参数的异常：`throw new BizException(ErrorCode.BIZ_ACCOUNT_NOT_EXIST.getCode(), ErrorCode.BIZ_ACCOUNT_NOT_EXIST.getMessage())`
