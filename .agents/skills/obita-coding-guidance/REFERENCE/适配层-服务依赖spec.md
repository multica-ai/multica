# 任务目标
根据上下文资料识别出要接入的依赖服务，实现依赖服务的接入。代码实现需要满足良好的性能、健壮性和可读性要求，同时遵循代码规范和工程结构。

## 任务清单
1. 分析POM依赖：根据上下文资料，分析每个服务需要的POM依赖，然后根据规范添加依赖并更新本地仓库。
2. 依赖服务分析：针对每行依赖的服务信息及接入示例，分析出需要接入的服务信息，包括服务类名称、方法名称、服务版本号。
3. 依赖服务方法参数分析：根据每行依赖方法的入参、出参、接入示例说明，初步分析出每个依赖服务方法的入参、出参信息及校验约束；再结合 read_file 工具分析出每个依赖服务方法的入参、出参完整信息。
4. 代码实现：按照以下要求完成 任务目标 的代码实现。

## 具体要求
1. POM接入规范：
   * 根据分析出的POM依赖，参考__工程结构.md__目录规范和pom文件风格，分别在工程的主pom文件、服务接入相关的子模块pom文件中添加依赖，主pom文件依赖优先放在 dependencyManagement 标签下。
   * 禁止重复添加，每个pom文件中相同POM依赖只添加一次，如果POM依赖版本号不同则只需修改版本号。禁止修改或添加文档中没提到的POM依赖。
   * POM依赖接入是代码实现前的步骤，务必首先完成主pom文件和子pom文件修改，再开始编写代码。
2. 依赖服务与校验约束分析
   * 遵照事实：仅从上下文资料提到的内容进行分析，不要编造内容。
   * 明确校验约束：根据*方法入参及出参说明*分析每个服务在接入过程中需要满足的参数校验约束。
   * 列出每个依赖服务详细信息：根据每行分析出的依赖服务，依次列出每个服务方法的具体入参、出参信息，包括入参和出参的父类相关字段。请按照如下提示完成解析：
     * I.优先使用 read_file 工具读取依赖的服务类、方法参数类以及参数类的父类，解析出依赖服务及方法参数的完整信息。注意直接使用类名称作为 read_file 工具的其中参数，不要添加java或class等后缀。
     * II.如果 read_file 工具不能使用，可以尝试通过javap命令反编译字节码文件完成服务及参数解析。
3. 代码实现要求
   * 注意，在编写代码之前，务必先完成*依赖服务与校验约束分析*！！！
   * 禁止重复接入：相同依赖服务只接入一次（服务签名、服务版本、服务参数相同）。
   * 路径规范：参考__工程结构.md__，将代码文件放置到合适的路径下，优先选择已存在的路径，确保模块划分清晰。
   * 参数校验：入参字段为字符串时优先使用 `org.apache.commons.lang3.StringUtils` 工具校验。
   * 注释标准：所有代码文件必须添加符合 JavaDoc 标准的注释，包括类注释、方法注释以及关键逻辑的行内注释。
   * 单一职责：相同依赖服务类放在同一个实现类；不同依赖服务类放在不同实现类。
4. 编码规范
   * 依赖服务注入规范：依赖服务版本号不为空时，则使用 @HSFConsumer 注解完成依赖服务注入。
   * 命名规范：
     * 变量名、方法名和类名应具有语义化，避免使用无意义的缩写。
     * 接入代码实现类名以 Adapter 作后缀，并使用，并使用 @Component 注解标注。
   * 入参及出参规范：
     * 方法入参：
       * 直接使用依赖服务方法的原始入参作为实现方法入参，请保持二者方法入参类型及名称一致。
       * 严格根据*方法入参说明*解析校验约束并完成入参校验，**不要自主增加文档中没提到的校验**。
     * 方法出参：如果依赖服务方法的调用结果被 xxxResult 或 xxxResponse 等类型包装，则需要去除包装类型，规则如下：
         * 判断包装类型中 success 等类似字段为真时，解析出真实的内部出参作为实现方法出参并返回。
         * 判断包装类型中 success 等类似字段非真时，根据强/弱依赖的处理要求进行不同处理。
   * 代码格式：遵循团队统一的代码格式规范（如缩进、空格、换行等）。
5. 编码质量
    * 可读性：
        * 确保代码具有良好的易读性、简单性以及最短层级嵌套。
   * 性能与健壮性：
       * 确保代码具有良好的性能，避免不必要的计算或查询。
       * 添加必要的异常处理逻辑，提升代码的健壮性。
         * 所有代码都必须放在 try catch 代码块中，包括参数校验及调用结果解析等所有实现代码。finally 代码块中统一记录日志，仅依赖服务调用结果、异常结果放在 try 之前定义。
         * catch 代码块需要根据服务的强/弱依赖进行不同处理：如果服务为强依赖则需要继续抛出异常，默认使用 RuntimeException；如果服务为弱依赖则 catch 后不要再向外抛异常。
   * 日志记录：
       * 日志只在 finally 代码块中统一记录，且记录内容要全面，包括依赖服务方法原始的入参对象、出参对象，异常信息，堆栈信息等，复杂对象使用 JSON.toJSONString 工具转换为字符串。
       * 日志格式要清晰易懂，异常不为空时需要输出异常堆栈，例代码如：`logger.error("签约失败，异常信息：{}，param：{}，result：{}”, e.getMessage(), JSON.toJSONString(param), JSON.toJSONString(result), eStack)`。其中`param`代表复杂对象入参，`result`代表复杂对象出参，`eStack`代表异常堆栈，编码中需要将param、result等字符替换为依赖服务方法参数名称。
