### 编码要求
区别于应用层，此规范适用于HTTP协议交互的服务实现，不要单独设计接口定义，服务实现符合SOLID原则。

1. 服务设计：表现层有独立的服务设计，包括入参、出参、服务类，确保含义清晰，相关代码放在表现层模块中。
   - 入参设计：复杂入参类定义使用 `Param` 作为后缀，比如 OrderParam。
   - 出参设计：复杂出参类定义以 `VO` 作为后缀，比如 OrderVO。
     - 出参需要用 xxxResult 或 xxxResponse 类型再包装作为方法完整出参，比如 `PageResult<OrderDTO>`、`PageResult<Boolean>`（需替换为实际输入的合适代码类）。
   - 服务类：服务类设计以 `Controller` 作为后缀。
   - 禁止直接使用业务层或数据层模型，必须定义表现层的出、入参模型
2. 校验约束分析
   * 明确校验约束：分析每个方法在实现过程中需要满足的校验约束（如参数校验、数据校验、权限校验等）。
3. 代码实现要求
   * 优先使用已有依赖：在实现过程中，优先复用已有的类或工具方法，避免重复造轮子。
   * 对象转换方法：务必在本层完成调用下层服务的接口协议转换，请编写专门的转换方法，确保字段映射完整，不要遗漏字段。禁止使用 BeanUtils 或类似工具。
   * 路径规范：参考__工程结构.md__，将代码文件放置到合适的路径下，优先选择已存在的路径，确保模块划分清晰。
   * 参数校验：入参字段为字符串时优先使用 `org.apache.commons.lang3.StringUtils` 工具校验。
   * 注释标准：所有代码文件必须添加符合 JavaDoc 标准的注释，包括类注释、方法注释以及关键逻辑的行内注释。
4. 编码规范
   * 代码风格：服务设计符合 RESTful 风格，按照下述规范执行，不要混淆 HTTP 请求类型：
     * GET请求类型：请求参数默认使用查询参数方式，如果参数超过 5 个则使用 POST 请求体方式
     * DELETE请求类型：请求参数默认使用查询参数方式，如果参数超过 5 个则使用 POST 请求体方式
     * POST请求类型：请求参数默认使用请求体方式
     * PUT请求类型：请求参数默认使用请求体方式
   * 注释清晰：对于复杂的逻辑，务必添加清晰的注释说明其目的和实现方式。
5. 编码质量
   * 性能与健壮性：
     * 确保代码具有良好的性能，避免不必要的计算或查询。
     * 添加必要的异常处理逻辑，提升代码的健壮性。
   * 可读性：
     * 确保代码具有良好的易读性、简单性以及最短层级嵌套。
   * 日志记录：
     * 方法体使用 try catch 包裹整个代码块，且在异常日志中要打印异常对象，catch 后不要再向外抛异常。
     * 添加必要的中文日志，日志内容应包含问题定位的关键信息，例如：异常原因、方法入参和出参、异常堆栈等，复杂类型使用 JSON.toJSONString 工具转换为字符串。
     * 日志格式要清晰易懂，包含异常堆栈的日志示例代码，`logger.error("签约失败，异常信息：{}，paramId：{}，param：{}”, e.getMessage(), paramId, JSON.toJSONString(param), eStack)`。其中`param`代表复杂参数，`eStack`代表异常堆栈，编码中需要将param*等字符替换为实际方法参数名称。
