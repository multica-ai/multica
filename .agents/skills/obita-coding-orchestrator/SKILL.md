AI 编码流程中
你正在运行一个编码流水线。
默认按顺序执行每个步骤，步骤1 不允许跳过。 
使用中文与用户沟通。
在开始任何步骤前，必须先按 skill 名称使用 `superpowers:using-superpowers`，确认当前任务涉及的 superpowers skill 已被正确发现和使用。
在 Codex 环境中，“使用 skill” 的最低要求是：
1. 先打开对应 skill 的 `SKILL.md`
2. 明确对用户说明当前正在使用哪个 skill、用于哪个步骤
3. 按该 skill 的流程继续执行，而不是只把它当作参考资料
4. 步骤2 到步骤 5 遵守 `docs/system-design/系统设计原则.html` 中的本系统设计原则

## 澄清保护规则（最小改动）

在步骤 1 前后，避免陷入重复澄清循环：
- 如果用户意图已经明确，仅在确有必要时最多做一次聚焦确认。
- 用户已经给出明确选择或回答后，除非范围发生变化，否则不要重复问同一个澄清问题。
- 保持原有流水线顺序不变；当意图已经足够清晰时，继续进入下一步。

## 步骤 1 — 用户意图

**调用 `obita-coding-user-intent` skill 澄清用户意图与范围。**

必须在阶段输出中显式标记：`当前执行步骤：Step 1 - 用户意图`。
在进入步骤 2 之前，必须先完成一次基于用户当前意图的最小范围文档发现与范围判定；默认不得读取业务代码来澄清需求范围，除非用户明确要求“读代码确认一下”。
不要在步骤 1 完成前直接调用 `superpowers:brainstorming` 或其他方案类 skill。
默认要求通过固定确认话术完成一次收口；若用户意图已非常明确，仅允许做一次最小范围的聚焦确认，不要重复围绕同一问题循环澄清。
执行要求（必须同时满足）：
1. 必须先输出一份基于需求/系统设计文档的最小需求确认文档，只包含三段：`需求澄清`、`变更范围`、`影响面分析`。
   - 不展示模型读取到的文件清单、扫描过程、项目背景介绍或与本次变更无关的模块信息。
   - 默认不得基于代码扫描结果扩展需求范围；只有用户明确要求读代码确认时，才允许把最小代码发现作为内部依据。
   - `需求澄清` 聚焦用户要实现的业务变化、触发条件、目标行为和不做边界。
   - `变更范围` 只列本次需要修改的代码/测试/文档范围。
   - `影响面分析` 只列本次确实影响的 API、数据、权限、外部集成、补偿/幂等、测试点；无影响类别不要展开。
2. 必须使用明确确认话术：`这样是否准确表达了你的需求？`
3. 在用户未明确确认前，不得进入 Step 2；不得用“意图已经比较清晰”为由跳过确认闸门。
例外：
- 如果意图已经明确，仅在确有必要时最多做一次聚焦确认，然后继续。

## 步骤 2 — 方案设计

**调用 `superpowers:brainstorming` skill 形成技术方案。**
执行要求（必须同时满足）：
1. 必须先按 skill 名称加载 `superpowers:brainstorming` 的 `SKILL.md`，不要依赖固定本机绝对路径。
2. 必须在阶段输出中显式标记：`当前执行步骤：Step 2 - 方案设计`。
3. 必须基于该 skill 的输出产出方案，不可跳过。

如果任务范围小、目标明确，且不存在需要用户决策的分支，可继续后续执行。

## 步骤 3 — 编写计划

**遵循步骤 2 `superpowers:brainstorming` skill 的引导和规范，在用户确认方案设计之后，调用 `superpowers:writing-plans` 编写计划。**
执行要求（必须同时满足）：
1. 必须先按 skill 名称加载 `superpowers:writing-plans` 的 `SKILL.md`，不要依赖固定本机绝对路径。
2. 必须在阶段输出中显式标记：`当前执行步骤：Step 3 - 编写计划`。
3. 必须输出可执行计划后再进入 Step 4。

## 步骤 4 — 执行计划

**遵循步骤 3 `superpowers:writing-plans` skill 的引导和规范执行计划。**
执行要求（必须同时满足）：
1. 默认在当前会话本地执行；只有用户明确要求并行、委派或多 agent 协作时，才按 skill 名称加载并使用 `superpowers:subagent-driven-development`。
2. 必须在阶段输出中显式标记：`当前执行步骤：Step 4 - 执行计划`。
3. 必须按计划逐项执行，并在阶段输出中同步进度。

## 步骤 5 — 测试与代码审查

**必须调用`obita-code-review` skill 确认改动是否实现完整，并且和spec设计要求一致。**
- 如果改动包含前后端联动或接口行为变更，调用 `obita-e2e-blacktest-curl` skill 进行回归测试。
- 端到端测试只能从两个真实对外发布服务入口发起：
  - `web`：后端管理 CRUD 接口
  - `api`：运行时公开接口
- 禁止在测试模块中自建 Controller、Fake Application、测试专用启动类、H2、内存数据库或替代运行时。
- 测试必须基于真实接口、真实参数、真实鉴权和真实环境动态构造，不得伪造内部链路。
- 测试框架规范以 `docs/system-design/测试框架设计原则.html` 为准。
- CURL 或其他联调请求必须基于“本次改动对应的真实接口、真实参数和真实环境”动态构造，不要复用与当前模块无关的固定示例。
- 当测试环境不可用时：
  1. 说明缺失的是哪一个真实入口服务（web 或 api）
  2. 优先尝试启动真实服务或执行真实联调脚本
  3. 不得退化为测试模块内自建应用、自建 controller 或 H2 替代方案
- 核心注释一致性门禁：
  1. 必须检查本次改动涉及的核心注释是否与 `docs/system-design` 下的系统设计保持一致。
  2. 核心注释包括但不限于：状态机流转、状态映射、错误码映射、幂等、补偿、重试、事务边界、外部系统调用、关键异常兜底、失败返回语义、接口契约、字段含义、权限/归属校验。
  3. 注释不得描述过期设计、过渡实现或已废弃字段；不得与系统设计文档中的接口契约、状态流转、错误码映射冲突。
  4. 如系统设计与代码实现存在差异，必须先明确是修文档、修代码，还是修注释。
  5. 若发现核心注释与系统设计不一致，必须作为 code review finding 输出，并在进入 Step 6 或最终交付前修复。

## 步骤 6 — 收尾

**必须调用`obita-coding-guidance` skill 检查代码，确认全部符合规范。**
- 如果本次产生了代码或文档改动，必须调用 `obita-coding-versioning` skill 更新版本记录或相关设计文档。
- 如果本次实现过程中引入了临时调试代码、兼容分支或待退役逻辑，必须调用 `obita-coding-cleanup` skill 完成清理。

## 工具拒绝恢复规则（最小改动）

当任何工具调用返回 `rejected` 或 `blocked` 时：
1. 用一句话说明具体被拒原因。
2. 如果可能，使用更安全或更小范围的参数重试一次。
3. 如果重试仍失败，提出一个聚焦的解阻问题。
4. 除非用户目标发生变化，否则不要跳回泛化的 A/B/C 需求访谈。

## 技能降级规则（最小改动）

当某个下游 skill 可被调用但实际不可用、已弃用或不适合当前任务时：
1. 明确说明该 skill 当前不可用或不适用。
2. 按当前流水线目标选择最接近的直接下一步继续，不要重复回到前面的泛化澄清。
3. 如果该 skill 原本承担的是“可选”的方案设计职责，而当前任务并不依赖它，直接跳过并继续后续步骤。
