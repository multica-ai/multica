## 全局写入确认门禁

- 任何模式下，未获得用户对当前章节或当前明确修改点的确认前，禁止写入、修改或覆盖任何系统设计 HTML 文件。
- “已定位目标文档/章节/修改点”只代表可以生成拟修改草案，不代表可以执行写入。
- 迭代场景中，即使 Step 0 Fast Path 已唯一定位目标文档、修改区域和修改动作，也必须先展示本次拟修改内容，并提供内容确认数字选项让用户选择：
  - `1. 确认内容`
  - `2. 返回调整`
  - `3. 取消本次修改`
- 内容确认问题固定为：`请选择下一步：1 确认 / 2 调整 / 3 取消`
- 用户对内容确认问题回复 `1`、`确认`、`可以`、`同意`、`按这个改`、`OK` 等肯定表达时，只代表拟修改内容通过，不代表允许立即写入文件。
- 内容确认后，必须再回显“将写入的文件路径 + 明确章节/修改点 + 只改这些内容”，并提供数字选项让用户选择：
  - `1. 现在写入文档`
  - `2. 返回调整`
  - `3. 取消本次修改`
- 写入授权问题固定为：`请选择下一步：1 写入 / 2 调整 / 3 取消`
- 用户回复 `1`、`写入`、`现在写入`、`可以写入`、`开始写入`、`执行写入`、`修改文档` 任一表达，才允许将该章节或修改点写入文档。
- 用户回复 `确认`、`可以`、`同意` 等未明确对应写入选项的表达时，不视为写入授权；应再次提示用户回复 `1` 即可写入。
- 用户只提供纠偏信息时，不视为内容确认或写入授权；应按纠偏结果重新生成拟修改内容并再次确认。
- 若用户在确认后表达质疑、打断、撤回、暂停、先别改、等一下等意思，立即停止写入流程；未写入则保持文件不变，已启动但未完成的写入必须说明风险并等待用户重新授权。

## 用户确认阶段展示规则

- 给用户确认时，只展示用户可读的中文说明、步骤列表、字段清单或 Markdown 表格，不展示 HTML 标签、HTML 表格行、Mermaid 源码或大段最终落盘代码。
- 对系统设计 HTML 的修改，确认阶段必须明确列出目标文件、改动章节/小节、修改前含义、修改后含义、字段是否保留、运行时是否填充等业务语义；不要用 `<tr>`、`<td>`、`<code>` 这类 HTML 片段让用户判断。
- 迭代修改确认阶段必须使用文档中的精确章节编号或章节范围，例如 `2.2-2.2.1 Merchant 创建提现订单流程时序`；禁止使用 `2.x`、`某章节`、`相关章节` 等泛化章节名。若修改点横跨父章节和子章节，必须用 `父章节-子章节` 的范围表达。
- 迭代修改确认阶段必须提供“改动前后对比”，使用 Markdown 表格或“修改前 / 修改后”中文条目；禁止只给笼统修改摘要后询问确认。
- 改动前后对比必须按章节分开展示；禁止把多个章节混在同一个总表里让用户确认。
- 每个章节块必须包含：章节名、改动对象、修改前、修改后、影响范围。若同一章节有多个改动对象，可在该章节块内使用 Markdown 表格；若涉及多个章节，必须使用多个章节块分别展示。
- 若本轮只允许一次改一个修改点，必须说明本轮只确认其中一个修改点。
- 测试用例确认使用 Markdown 表格，不使用 HTML 表格代码。
- 只有用户明确要求“看 HTML”“给我代码片段”“展示最终落盘内容”时，才允许展示 HTML 或 Mermaid 源码。
- 写入文档阶段仍按目标文件格式生成 HTML；确认阶段的可读草案不等同于最终 HTML 原文。

## 版本记录更新规则

- 本 skill 的职责是生成或迭代系统设计文档；若本次只修改 `docs/system-design/` 或 `docs/requirements/` 下的文档，不同步修改 `docs/versioning/README.md` 或 `docs/versioning/version_v*/record.md`。
- 只有当同一轮任务还包含代码、DDL、接口实现、配置、测试等工程文件变更时，才按项目版本管理规则更新版本记录。
- 用户明确要求“更新版本记录”“记版本”“发版记录”时，才允许为纯文档修改追加版本记录。

## Step 0 - 需求类型判断

优先执行“迭代场景快捷定位”。

### Step 0 Fast Path - 迭代场景快捷定位

若用户输入已明确表达“对现有系统设计做局部修改”，允许跳过标准的类型选择问答，先基于上下文自动定位。

满足以下条件时，可直接判定为“迭代现有系统设计”：
1. 出现明确修改动作词：`删除` / `删掉` / `修改` / `替换` / `补充` / `新增` / `调整`
2. 出现系统设计对象词：`系统设计` / `时序图` / `ER图` / `状态机` / `测试用例` / `接口定义` / `影响面分析` / `章节`
3. 出现业务范围、接口名、章节名或原文案线索，如：`法币提现`、`法币充值`、`/withdraw/order/list`、具体文字描述

执行规则：
1. 扫描目录 `docs/system-design/需求系统分析/` 下的 HTML 文件
2. 根据业务关键词、接口名、章节名、原文案，分层判断：
   - 目标文档
   - 修改章节或修改区域
   - 具体修改动作或目标文案
3. 同时保留少量候选项，供用户纠偏
4. 每次只问一个问题
5. 不允许做“空转确认”，确认必须和下一步信息采集或执行合并

分层推进规则：

- 若只能判断目标文档：
  - 向用户说明模型判断的目标文档
  - 可附带最多 3 个候选文档
  - 不单独询问“是否确认”
  - 直接要求用户：若判断正确，请回复要修改的章节/区域；若不正确，请直接回复正确文档名

- 若能判断目标文档和修改章节/区域：
  - 向用户说明模型判断的目标文档与章节/区域
  - 可附带最多 3 个候选章节/区域
  - 不单独询问“是否确认”
  - 直接要求用户：若判断正确，请回复具体要修改的内容；若不正确，请直接指出正确的文档或章节

- 若能唯一判断目标文档、修改章节/区域和具体修改动作：
  - 直接向用户说明模型判断结果
  - 生成本次拟修改内容，必须包含目标文件、改动章节/小节，并按章节分开展示改动前后对比
  - 不得直接写入文档
  - 必须提供选项：`请选择下一步：1 确认 / 2 调整 / 3 取消`
  - 用户确认后，只能进入写入范围确认，不得直接写入
  - 必须回显“将写入的文件路径 + 明确章节/修改点 + 只改这些内容”，并提供选项：`请选择下一步：1 写入 / 2 调整 / 3 取消`
  - 若用户发现偏差，可直接指出正确文档、章节或修改点；用户纠偏不视为内容确认或写入授权

候选项规则：
- 备选文档最多展示 3 个
- 备选章节/区域最多展示 3 个
- 按相关性从高到低排序
- 若只有 1 个候选，则直接展示该候选，不必凑满

用户纠偏规则：
- 若用户指出正确文档、章节或修改点，模型应直接基于纠偏结果继续流程
- 不得再次退回最初的类型选择步骤

规则：

## Step 1 - 需求输入对焦（仅全新需求）

1. 提供本地需求文档（推荐）
2. 提供 Feishu 文档地址

规则：

### Step 1A - 本地文档
- 读取本地需求文档。
- 让用户确认”需求输入已就绪，可以进入系统设计”。

### Step 1B - Feishu 文档

1. **自动获取 UAT**：
   - 按以下顺序扫描配置文件获取 UAT：
     - `~/.claude/mcp.json`
     - `{项目根目录}/.mcp.json`
     - `~/.claude/projects/*/{项目名}/.mcp.json`
     - `/Users/owen/Documents/system_analysis/.mcp.json`
   - 从 `mcp.json` 中提取 `X-Lark-MCP-UAT` 值

2. **验证 UAT 有效性**：
   - 调用飞书 API 验证 token 是否有效
   - 若 token 过期或无效，引导用户获取新 UAT

3. **获取新 UAT（如需要）**：
   - 生成 OAuth 授权链接
   - 指导用户在浏览器完成授权
   - 从回调 URL 中提取 code，换取 UAT
   - 更新本地配置

4. **拉取文档**：
   - 从 URL 提取文档 Token：例如 `https://xxx.feishu.cn/wiki/Ijgbwoy7yiMNqPkE7Jjcq791ngf` → `Ijgbwoy7yiMNqPkE7Jjcq791ngf`
   - 如果是 wiki 类型，先调用 `/open-apis/wiki/v2/spaces/get_node` 获取实际的 docx token
   - 调用飞书文档 API 获取内容，使用分页机制遍历所有页面：
     ```
     GET /open-apis/docx/v1/documents/{doc_token}/blocks?page_size=500
     - 如果 has_more=true， 使用 next_page_token 获取下一页
     - 重复直到 has_more=false
     ```
   - 遍历所有分页，提取所有文本块内容
   - 文本块类型：2(段落)、3-7(各级标题)

5. 保存为 HTML 格式到 `docs/requirements/`。
   - HTML 需包含完整的文档内容和结构
6. 文件命名：`需求名称-YYYYMMDD.html`，重名追加 `-v2`。
7. 回显”本地保存路径 + 文档标题”，等待用户确认后进入 Step 2。
8. 若拉取失败，提示用户重新选择三项输入方式。

### Step 1C - 无文档澄清
1. 立即调用 `obita-coding-user-intent` skill 做需求澄清。
2. 用户确认后，落盘为 HTML 格式到 `docs/requirements/`，命名同上。
3. 进入 Step 2。

## Step 2 - 系统设计草案（HTML）

### 迭代模式（Step 0A 选择了迭代）
1. 若已在 Step 0 Fast Path 中唯一定位到目标文档、章节/区域及具体修改动作，只能先生成拟修改草案，不得直接执行写入
2. 若仅定位到目标文档，则继续收集修改章节/区域
3. 若已定位到目标文档和修改章节/区域，则继续收集具体修改内容
4. 每次只修改一个章节或一个明确修改点
5. 每章节或每个修改点先按“用户确认阶段展示规则”向用户展示拟修改内容；必须明确目标文件、改动章节/小节，并按章节分开展示改动前后对比，然后提供选项：`请选择下一步：1 确认 / 2 调整 / 3 取消`
6. 用户确认内容后，必须再回显写入范围并提供选项：`请选择下一步：1 写入 / 2 调整 / 3 取消`
7. 用户明确授权写入后，才允许将该章节或修改点写入文档
8. 写完后询问用户是否继续修改其他章节

### 全新模式（Step 0B 选择了全新）
1. 固定加载 `assets/system-design-template.html`。
2. 固定加载 `references/design-principles.md`。
3. 先判定变更类型：`完全新增` / `现有能力升级` / `混合型`。
4. 如信息缺失，每次只问一个问题补齐。
5. 按模板产出系统设计草案（HTML）。

Step 2 固定章节顺序（不得调整）：
1. 接口定义（含接口层错误码定义）
2. 接口实现时序
3. 数据库 ER 图
4. 状态机
5. 下游依赖模块集成（含下游错误码处理逻辑）
6. 测试用例
7. 影响面分析
8. 讨论记录

Step 2 编写过程规则：
- 必须按 1 -> 6 逐章与用户确认。
- 每次只确认一个章节，当前章节未确认不得进入下一章节。
- 每章内容确认必须提供选项：`请选择下一步：1 确认 / 2 调整 / 3 取消`

**接口定义章节细化流程：**
1. **接口路径** - 用户只需提供接口名称和路径，模型根据 RESTful 规范自行判断请求方法和 Content-Type：
   - `POST /xxx/create` → POST, application/x-www-form-urlencoded
   - `GET /xxx/info` → GET, application/x-www-form-urlencoded
   - `POST /xxx/submit` → POST, application/json
2. **入参确认** - 模型一次性列出所有请求参数清单，用户确认或修改
3. **出参确认** - 模型一次性列出所有响应参数清单，用户确认或修改
4. **错误码确认** - 模型一次性列出所有错误码清单，用户确认或修改

**1.1 接口定义表格只包含三列：**
| 接口名 | 路径 | 方法 |
|--------|------|------|
| xxx | /xxx/create | POST |

详细参数在 1.2、1.3、1.4 中分别说明。

补充规则：
- 若变更类型为”升级/混合”，必须提供”与现有能力差异对比表”。

**写入一致性检查规则：**
每次在用户确认并写入文档后，检查以下内容：
1. 是否与模板结构一致（章节顺序、表格格式）
2. 是否与用户讨论的内容一致（参数名、类型、说明等）
3. 如有不一致，立即修正

**接口实现时序章节细化流程：**
1. **初版生成** - 模型根据需求文档理解，先形成一版完整的时序内容（包含主流程和异常流程）
2. **逐节点确认** - 每次只问用户一个节点/步骤的问题
3. **用户确认阶段表达方式** - 与用户讨论、确认时，只使用文字步骤列表，方便用户理解：
   ```
   【接口：OpenAPI提现 /withdraw/order/create】

   步骤1：外部网关 → 提现服务：POST /withdraw/order/create
   步骤2：提现服务 → 商户服务：商户鉴权(接口待定)
   步骤3：提现服务 → 商户服务：查询商户信息
   步骤4：提现服务 → 提现服务：创建提现订单（状态=MERCHANT_APPROVED）
   步骤5：提现服务 → 计费服务：/feign/charge/calculate（计算手续费）
   步骤6：提现服务 → 支付服务：冻结余额

   【异常分支-冻结失败】
   步骤6.1：更新单据状态为 FINAL_FREEZE_FAILED（一锁二判三更新）
   步骤6.2：返回给网关 (status=FAILED, errorMsg=BALANCE_INSUFFICIENT)
   ```
4. **写入文档阶段表达方式** - 只有在写入 HTML 文档时，才直接生成 `mermaid sequenceDiagram`：
   ```
   sequenceDiagram
   participant GW as 外部网关
   participant WD as 提现服务
   participant MR as 商户服务
   participant CS as 计费服务
   participant PM as 支付服务

   GW->>WD: POST /withdraw/order/create
   WD->>MR: 商户鉴权(接口待定)
   WD->>MR: 查询商户信息
   WD->>WD: 创建提现订单（状态=MERCHANT_APPROVED）
   WD->>CS: /feign/charge/calculate（计算手续费）
   WD->>PM: 冻结余额

   alt 冻结失败
       WD->>WD: 更新单据状态为 FINAL_FREEZE_FAILED（一锁二判三更新）
       WD-->>GW: 返回(status=FAILED, errorMsg=BALANCE_INSUFFICIENT)
   end
   ```
5. **接口命名规则**：
   - 已知外部接口：直接用接口名，如 `/feign/charge/calculate`
   - 未知接口：中文描述，如 `商户鉴权(接口待定)`
6. **确认阶段时序规则**：
   - 主流程使用文字步骤序号：1、2、3；分支使用 2.1、2.2
   - 用 `→` 表示调用方向，用 `←` 表示返回
   - 返回信息需明确 status 和 errorMsg
7. **文档写入时序规则**：
   - 文档中主流程按消息先后顺序用 mermaid 表达
   - 分支场景使用 `alt / else / opt / loop` 表达
   - 必要时使用 `Note over` 或 `Note right of` 补充上下文
   - 文档中不要保留用户确认阶段的文字步骤列表
8. **状态更新规则**：
   - 使用"更新"而非"推进"
   - 所有状态更新必须标注"（一锁二判三更新）"
   - 不更新数据库的场景（等待补偿）：描述为"不推进状态机，等待补偿"

时序讨论规则：
- 每次讨论时只展示一个接口的时序
- 讨论阶段只展示文字步骤列表，不展示 mermaid 代码
- 用 `→` 表示调用方向，用 `←` 表示返回
- 返回信息需明确 status 和 errorMsg
- 每个接口的异常分支单独列出，不混在主流程中

时序写文档规则：
- 每次写入 HTML 文档时，直接生成 `mermaid sequenceDiagram`
- 写入文档时不保留讨论阶段的文字步骤列表
- 文档中的主流程与异常分支都要落在图里，不额外附同义文字描述

**迭代场景时序展示规则（重点）**：
- 迭代场景只展示本次修改的步骤及其上下文
- 不展示完整流程，只展示：修改点 → 修改步骤 → 下一步
- 讨论阶段用 `[修改]` 标记本次修改的步骤
- 文档写入时用 `Note over`、`rect rgba(...)` 或注释明确圈定修改范围
- 每一步的完整处理逻辑都要展开，包括：
  - 幂等处理（如订单重复判断）
  - 状态更新（一锁二判三更新）
  - 异常分支
  - 返回给调用方的信息

**测试用例章节细化流程：**
1. **测试框架** - JUnit 5 + Mockito + AssertJ，单元测试 Mock 下游依赖
2. **用例组织** - 按接口组织，每个接口包含：
   - 正常流程测试
   - 异常流程测试
   - 边界值测试

4. **用例格式**：

**确认流程：**
- 每次只确认一个接口 + 一个用例分类（正常/异常/边界值）

## Step 3 - 生成对应系统设计文件

### 迭代模式
1. 仅将用户已明确确认内容且明确授权写入的章节或修改点增量写入原文档 `docs/system-design/需求系统分析/系统设计-<需求名>.html`

### 全新模式

2. 输出目录固定为：`docs/system-design/需求系统分析/`。
