# obita-code-review

## 指令说明
* 使用方法：当识别到用户意图为编码实现质量检查、代码质量CR时，会自动触发此技能并加载 REFERENCE 文件夹中的一个或几个相关的参考文件作为规范约束。
* 适用范围：除完整需求实现检查外，也适用于对某个局部实现点做定点 CR，例如 `orderId` / `requestSeqNo` / `batchId` 生成、幂等处理、状态流转、异常恢复等。
* 参考文件：在文件夹`{.Codex} 或 {.codex}/skills/obita-code-review/REFERENCE`下，采用分层设计的规范文件，具体说明如下：
  - `代码实现检查spec.md`：用于检查需求编码实现质量，包括功能逻辑实现、代码设计、代码规范、代码质量等。

* 代码质量检查报告文件：多次检查则以版本号递增的方式生成，文件名格式为`{日期}-{需求名称}-code-review.md`，存放在 `docs/superpowers/codereview/` 目录下。
* 代码质量检查报告索引文件：`docs/superpowers/codereview/`，如果版本文件夹或文件不存在，则新建。
* 定点 CR 要求：当用户只指定一个实现点时，仍需按 REFERENCE 中的功能逻辑、并发、数据一致性、线上风险几个维度完成审查，避免只做“代码位置定位”而忽略真实缺陷判断。
* 行级注释检查约束：审查时必须检查关键逻辑的行级注释是否齐全、准确且与实现一致，注释缺失或过期需在报告中明确标注风险。
* 命名语义门禁：审查时必须检查公开接口、分层入口、核心方法的命名是否具备明确业务含义；若命名泛化到无法直接看出业务职责，按检查失败处理，不能仅作为“代码风格建议”轻描淡写带过。
