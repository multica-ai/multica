# 任务目标
根据用户输入的**目标需求**和当前分支代码的修改内容，检查功能逻辑实现、代码设计、代码质量、代码规范等。

## 任务清单
1. 代码分析：通过 git 命令等工具找出当前分支改动的代码文件，分析代码修改的详细内容，说明代码改动实现的需求功能逻辑。
2. 需求分析：根据用户**目标需求**充分理解需要实现的需求功能和技术实现要求。
3. 代码实现检查：结合前2步的 需求分析 和 代码分析，根据“具体要求”章节检查代码实现。
4. 生成检查报告：综合以上分析结果生成代码实现检查报告。
   
## 具体要求
1. spec需求文档优先：如果有**目标需求**分析后的 spec.md 文档则以 spec.md 文档为基准。
2. 代码分析范围：分析当前分支从创建以来未提交的改动、以及已提交的排除 merge 外的所有改动。
3. 功能逻辑：
   * 一致性：检查代码实现是否与需求功能逻辑一致，是否满足需求功能和技术实现要求。
   * 完整度：检查代码实现是否完整，是否有待完善的功能逻辑。
   * 充分性：是否有未考虑到某些业务规则细节，边界条件是否考虑充分等。
4. 代码设计：
   * 可读性：保持代码的可读性，避免使用奇淫巧技，避免过度设计。
   * 可扩展性：代码是否方便扩展，容易持续迭代，避免强耦合。
   * 重复性：遵守 Don’t Repeat Yourself 原则，复用已有的代码、在职责单一的前提下尽量减少重复代码的编写。
   * 可维护性：代码的可维护性是由很多因素协同作用的结果，代码的可读性好、简洁、可扩展性好，会使得代码更易维护。
   * 可测性：代码可测试性的好坏，同样可以反应代码质量的好坏，代码的可测试性差难以编写单元测试用例。
   * 语义自描述性：公开接口、Facade、Integration、Repository、Orchestrator、Processor 以及核心方法的名称，必须让评审者在不阅读实现细节的前提下，直接理解其业务对象、业务动作和职责边界。
5. 代码质量：
   * 安全问题：是否存在安全隐患，例如是否存在 SQL 注入、CSRF、越权等安全漏洞。
   * 性能问题：是否会产生性能上的损害，例如慢 SQL、热点 key、数据倾斜等性能问题。
   * 资损问题：是否存在资损问题，例如是否存在金额计算错误、数据重复写入等资损问题。
   * 并发问题：是否存在并发问题，例如数据重复写入、数据丢失等并发问题。对于 `orderId`、`requestSeqNo`、`batchId`、外部请求号等全局标识生成逻辑，必须额外检查唯一性粒度是否足够、并发下是否会碰撞、时钟回拨或多实例场景下是否会重复、冲突后是否存在重试或降级处理。
   * 数据一致性：是否存在数据不一致问题，例如缺少事务约束导致数据不一致等。若唯一索引承担幂等或主键约束职责，必须检查不同唯一键冲突的处理语义是否正确区分，避免将“业务幂等命中”和“内部单号碰撞”混为一类。
   * 线上风险：是否会对稳定性产生潜在影响，例如是否产生死锁、死循环、NPE、FullGC、OOM 等问题。 
   * 未知影响：是否对体验产生潜在影响，例如某 Feature 的上线会潜在影响了其它功能的表现等。
6. 代码规范：
   * 是否符合当前工程统一的代码风格要求以及分层次的代码规范要求。
   * 命名语义门禁：对外接口、跨层接口、核心抽象和关键方法名称必须具备明确业务含义，能够直接回答“它服务哪个业务对象、执行什么动作、处于哪一层职责”。禁止仅使用 `Manager`、`Helper`、`Processor`、`Handler`、`Service`、`Facade`、`Util`、`execute`、`process`、`handle` 等泛化词汇作为主要语义承载，除非名称中已经带有足够明确的业务限定。
   * 命名判定标准：若评审者仅看名称仍无法判断该接口是“创建/查询/审核/冻结/计费/回放/补偿”中的哪类能力，或容易将“调用方 / 协调方 / 执行方 / 持久化方”职责混淆，则按检查失败处理，至少归类为 `P1`。
   * 行级注释约束：关键分支、状态流转、异常兜底、外部依赖调用前后必须具备可读的行内注释，禁止仅保留空泛注释或 TODO 占位。
   * 核心注释一致性：检查本次改动涉及的核心注释是否与 `docs/system-design` 中的系统设计一致，尤其是状态机、错误码、接口字段、外部依赖、补偿、幂等、事务边界、权限/归属校验。若注释和设计不一致，按实现缺陷处理。

## 检查报告要求
> 检查报告应包含以下章节：

### 1. 概述
- 本次改动涉及的文件列表
- 改动代码行数统计
- 改动类型分类（新增功能、Bug修复、重构等）

### 2. 需求分析
- 目标需求描述
- 技术实现要求
- 关键功能点列举

### 3. 代码分析
- 改动文件详细分析
- 实现的功能逻辑说明
- 技术方案概述

### 4. 功能逻辑检查
- 需求一致性分析
- 功能完整度评估
- 边界充分性评估

### 5. 代码设计检查
- 可读性评估
- 可扩展性评估
- 重复性评估
- 可维护性评价
- 可测性评估

### 6. 代码质量检查
- 性能问题分析
- 安全问题分析
- 资损问题分析
- 并发问题分析
- 数据一致性分析
- 线上风险分析
- 未知影响分析

### 7. 代码规范检查
- 代码风格检查
- 项目规范检查
- 命名语义检查

### 8. 优化建议
按优先级分类：
- **P0（必须修复）**：严重问题，影响功能或安全
- **P1（建议修复）**：重要问题，影响性能或体验
- **P2（可选优化）**：优化建议，提升代码质量

### 9. 总体评价
- 代码质量评分（1-5分）
- 主要优点总结
- 主要问题总结
- 整体建议

## 检查流程

```
1. 获取代码改动
   ↓
2. 分析需求文档（如有）
   ↓
3. 逐文件分析代码改动
   ↓
4. 按检查维度进行检查
   ↓
5. 汇总问题和建议
   ↓
6. 生成检查报告
```

## 注意事项

1. **客观公正**：检查应基于事实和规范，避免主观臆断
2. **建设性**：提出问题的同时给出具体的优化建议
3. **优先级**：明确问题的严重程度和优化优先级
4. **上下文**：考虑项目实际情况，避免过度理想化
5. **平衡性**：在代码质量和开发效率之间找到平衡
6. **持续改进**：检查规范应随项目发展持续优化

## 定点CR补充检查项

当用户只要求 review 某个局部实现点时，除定位代码外，至少补充以下判断：

1. **职责判断**：确认该逻辑真实发生在哪一层，避免把“调用方 / 校验方 / 使用方”误判为“生成方 / 决策方”。
2. **唯一性判断**：确认标识生成规则是否足以支撑单实例并发、多实例部署、时间回拨、重试补偿等场景。
3. **约束判断**：确认数据库唯一索引、下游幂等键、外部系统 requestId / bizNo 等是否依赖该标识，评估碰撞后的影响范围。
4. **异常语义判断**：确认不同冲突来源是否被正确区分处理，例如业务幂等命中、内部单号碰撞、下游重复受理，避免错误吞并或错误重放。
5. **测试覆盖判断**：确认是否存在针对碰撞、重试、回放、边界值的单测或集成测试；若没有，需作为 review 风险明确指出。
6. **行级注释判断**：确认关键逻辑是否具备行级注释，且注释与真实行为一致，避免“代码已变更但注释未更新”导致误导。
7. **系统设计一致性判断**：确认核心注释是否与 `docs/system-design` 中的接口契约、状态流转、错误码映射、外部依赖、补偿/幂等/事务边界保持一致；若不一致，必须明确指出应修文档、修代码或修注释。
8. **命名语义判断**：确认本次 review 涉及的接口、抽象、方法命名是否直接表达业务对象、业务动作和层次职责；若名称泛化到需要依赖实现细节才能理解，应明确作为门禁问题指出。
