# Obita 平台业务上下文

## 平台概述

Obita 是一个跨境支付平台，面向 B 端客户提供支付、充值、提现、资金管理、兑换、Payout 等能力。平台基于多牌照架构运营，支持法币和数币双轨业务。

## 牌照体系

| 牌照 | 全称 | 说明 |
|------|------|------|
| TCSP | 信托或公司服务提供商 | 一类牌照主体，包含数币+法币账本 |
| MSO | 货币服务经营者 | 二类牌照主体，仅包含法币账本 |

- 同一客户可签约一个或两个牌照主体
- 不同牌照的监管要求、资金属性、业务场景不同
- 法币余额按牌照主体分开展示，不按 VA 展示
- 补签约机制：已入驻客户可后续补签另一个牌照，需重新进行 onboard 审核

## 核心产品模块

| 模块 | 说明 |
|------|------|
| Checkout 收银台 | 面向商户的收单/收银产品 |
| SGB VA 充值/提现 | 基于虚拟账户的法币充值和提现功能 |
| 统一余额模型 | 对客展示多牌照、多币种余额的综合模型 |
| 账户体系 | 法币多牌照账户 + 稳定币账户 + 客户/公司业务账户结构 |
| OOMP 商户管理 | 运营后台，支撑 KYB 审核、商户配置、交易管理、Risk Case 审核等 |
| 审批流 | 支持客户自配置规则和邮件邀约授权的审批系统 |
| VelaFi 渠道接入 | 稳定币支付基础设施集成，支持商户 API Key 和授权 API Key 两套认证 |
| 风控系统 | 入金/出金审核、合规校验、制裁筛查、EDD 等 |
| Payout 付款 | Obita 代商户向第三方（非同名企业/个人）进行体外付款的资金类产品，支持数币/法币、同币种/跨币种 |
| Conversion 兑换引擎 | 统一的可扩展兑换引擎，负责法币↔数币、数币↔数币、法币↔法币之间的安全、合规、可审计转换服务 |
| ObitaPocket（C 端钱包） | 基于 Web3Auth 的 C 端钱包，商户 Payout 向 C 端用户发放薪资/奖励/佣金/补贴 |

## 账户架构

- **账户域作为主数据和关系中枢**，不负责账务分录和最终余额
- Legal Entity → User ID → Account 三层结构
- VA 作为 Account 层的虚拟银行账户，与客户 ID 建立绑定关系
- 一个 Legal Entity 可绑定多个 VA（按签约牌照+币种维度隔离，如 TCSP-USD-SGB-VA、TCSP-USD-GLDB-VA）
- 生命周期同步：当客户 onboard 状态变更为「审核不通过」或「已注销」时，关联的 VA 状态同步更新为「Closed」

## 统一余额模型

### 对客展示逻辑

法币余额不按 VA 维度展示，按牌照主体维度展示：

| 客户签约状态 | TCSP VA | MSO VA | 余额卡片展示 |
|-------------|---------|--------|------------|
| 仅签约 TCSP | 可用 | 不存在 | 仅显示 TCSP 卡片 |
| 仅签约 MSO | 不存在 | 可用 | 仅显示 MSO 卡片 |
| 同时签约两者 | 可用 | 可用 | 同时显示两张卡片 |
| 未签约任何 | 不存在 | 不存在 | 无卡片，引导签约 |

### 余额卡片交互

- 点击 TCSP 卡片的「充值」→ 进入 TCSP 充值页 → 展示 TCSP VA 收款账号
- 点击 MSO 卡片的「充值」→ 进入 MSO 充值页 → 展示 MSO VA 收款账号
- 两者的收款账号不同（VA 号不同、开户行主体不同）

### 前置条件

- 法币资产页面展示已开通法币功能的前置要素：onboard 完成
- 币种展示逻辑：查询 Merchant ID 开通的 VA 数据，展示对应币种卡片
- 如客户未完成 onboard，法币资产模块引导客户完成 onboard

### 双卡片余额展示（TCSP/MSO 分离）

余额卡片分两张展示：

| 卡片 | 对应余额 | 说明 |
|------|---------|------|
| 数币业务可用余额 | TCSP 账户余额 | 用于 On Ramp/Off Ramp/Swap 场景 |
| 付款业务可用余额 | MSO 主体可用余额 | 用于法币 Payout 场景 |

- 未开通对应主体时展示 0
- 提示文案："Obita 账户中的法币余额来源包括数币兑换、银行账户充值等方式，余额支持的使用场景如下"
- Navro 充值账户卡片增加提示："付款业务专用充值账户"，账户详情下提示此账户资金仅能用于法币付款

## 账本体系

### TCSP / MSO 账本隔离规则

- TCSP 账本：包含数币+法币
- MSO 账本：仅包含法币
- 两套账本数据完全独立隔离，不得混用
- 跨账本交易需使用桥币（如 USD）实现多跳路径

### BVI 自营交易主体

- BVI 是 Obita 旗下的自营交易主体，在 TCSP 牌照下承担**所有商户的唯一交易对手方**
- 普通商户不得直接与二级 LP（Velafi）交易，必须通过 BVI 中转
- BVI 库存不足时需手动补库（SOP），系统自动 Lark 通知运营
- OOMP 交易台账和头寸看板基于 BVI 视角记录资金动账

## LP 层级与流动性提供方

| LP | 层级 | 定位 | 适用场景 | 支持币种 | 关键约束 |
|----|------|------|---------|---------|---------|
| BVI | 一级 LP（自营） | TCSP 下交易对手方 | On/Off Ramp、Swap | USD + USDT/USDC | 库存需手动补库 |
| Navro | 一级 LP（直连） | MSO 下法币 FX | 法币 FX + Payout | 全球主流法币 | 无银行牌照，资金需在清算周期内 payout |
| Velafi | 二级 LP | 小币种兑换 | 体外小币种 Payout | MXN/BRL/ARS/COP/PEN + USDT/USDC | 无银行牌照，资金在清算周期内必须 payout；需 OOMP 手动补录交易台账 |
| SGB | 银行渠道 | 法币入金/出金 | On/Off Ramp 的法币腿 | USD/EUR/GBP/SGD/HKD | TCSP+MSO 双账本；Conversion 内部记账无需真实转账 |
| GLDB | 银行渠道 | 法币入金/出金（人工） | On/Off Ramp 的法币腿 | USD/AED/AUD/CNY/EUR/HKD/JPY | 需人工操作转账 |

## 渠道信息

| 渠道 | 类型 | 能力 |
|------|------|------|
| SGB | 银行渠道（已对接 API） | 自动开户、入金通知、出金付款、webhook 回调、对账清算、订单查询 |
| GLDB | 银行渠道（未对接 API） | 手动录入 VA、手工补录交易 |
| Navro | 法币流出渠道（LP） | 法币 Payout（含 FX 兑换）、Payout 批次管理、余额查询、状态回调、入金充值 |
| Cobo | 数币托管钱包/转账渠道 | 按协议提币（数币从大账户到商户 Web3 钱包地址）、数币 Payout 到收款人钱包地址 |
| Reap | 企业信用卡渠道 | U 卡（企业信用卡）充值、授权扣费、卡/交易/商家层面风控配置 |

- 金融网关侧维护渠道配置（名称、Logo、Swift Code、地址、支持币种等）
- 渠道有优先级配置，数字越小优先级越高（1-N）
- 渠道有状态配置，如渠道状态异常降级向上游提供渠道功能
- 金融网关需按渠道功能维度设置配置文件，以便业务进行查询
- 提现路由：SGB 自动出金 / GLDB 人工出金 / 双渠道默认 SGB / 审核标签优先人工

### SGB 银行渠道详情

**开户方式**：
- 自动开户：调用 SGB 开户接口，无需输入 VA ID
- 手动开户：运营手动录入 VA ID 和银行开户证明

**Webhook 回调类型**：

| 回调类型 | 触发场景 | 说明 |
|---------|---------|------|
| 入金通知 | 客户银行来账 | SGB 收到资金后推送，携带 VA_Account_Number、来账金额、来款方账户信息 |
| 出金结果 | 出金交易完成 | SGB 受理出金后返回受理结果和交易结果 |
| 退回通知（Return） | 出金后被收款行退回 | 交易被收款行退回时触发，需创建退款入账订单 |
| 订单查询响应 | 运营手动查询 | 用于校验手工补录订单是否存在于渠道侧 |

**SGB 特殊要求字段**（绑卡时）：
- purpose code：后端固定值 OAT（Own Account Transfer）
- country、city、address：取客户 onboard 信息的地址

### Navro 渠道详情

| 项目 | 内容 |
|------|------|
| 渠道类型 | 法币流出渠道 / 法币入金渠道 |
| 资金模式 | License 模式（MSO 主体在 Navro 以商户身份 onboard） |
| 资金准备方式 | Prefunding（预充值，商户在 Navro 账户预存资金） |
| 核心能力 | 法币 Payout（含 FX 兑换）、Payout 批次管理、余额查询、状态回调、入金充值 |
| API 认证 | Client ID + Secret → Generate access token |
| 下游伙伴 | Thunes（最终执行向收款人银行账户打款） |
| 三方协议 | Obita ↔ Navro ↔ Thunes |

**Navro 入金（充值）流程**：

| 项目 | 内容 |
|------|------|
| 入金通知 | Navro 通过 webhook 余额变更通知推送入金（含 VA ID、来账金额、币种） |
| 订单认领 | 金融网关广播后，deposit 服务根据 VA ID 认领订单 |
| 入账主体 | 仅计入 MSO 主体可用余额（付款业务可用余额），不支持 TCSP 主体入账 |
| 风控 | 完全复用现有 SGB/法币入金风控处置链路（通过/不通过/补充材料/重复通知/未匹配订单） |
| 对客状态 | 完全复用现有 Top Up 订单定义，仅增加渠道与主体差异 |
| 邮件通知 | 复用现有入金邮件通知逻辑 |
| 主动查询 | Get transactions from a balance（用于 webhook miss 后手工补单） |

**Navro 充值对客交互**：
- 选择 Navro 渠道后展示充值账户信息，结构复用 SGB 现有格式
- 账户详情下增加专属提示语："请注意，此账户充值的资金，仅能用于法币付款交易场景，如需进行数币兑换交易，请您选择其他充值账户"

**Navro 开户流程**：

| 项目 | 内容 |
|------|------|
| onboarding 入口 | OOMP — OTC — KYB，KYB 流程新增 MSO 主体选项 |
| 开户方式 | 运营小二代商户在 Navro hub 手工开户 |
| VA 配置 | 渠道选项新增 Navro，填写：VA ID（必填）、币种 USD、银行开户证明（文件上传） |
| VA 审核 | 提交后双人审核，审核通过后 VA 生效 |
| 信息同步 | 开户后商户的 Navro 信息（Client ID 等）手工登记至金融交换、会员/商户系统 |
| 编辑/解绑 | 支持解绑，后端校验余额和订单状态 |

**Navro Payout 状态映射**：

| Navro 状态 | 说明 | Obita 对客状态 |
|------------|------|---------------|
| Draft | 已创建未行动 | proceeding |
| Requested | 等待确认汇率 | Committed |
| Committed | 进行 AML 检查 | Processing |
| Processing | Navro 最终检查 | Sent |
| Sent | 已发送至支付伙伴 | Accepted |
| Accepted | 合作伙伴已验证 | success / failed |
| Completed: Paid | 收款人已收到款项 | success |
| Completed: Rejected | 被合作伙伴或银行拒绝 | failed |
| Completed: Cancelled | 被审批人/Navro/合作伙伴取消 | failed |
| Completed: Failed | 通用错误 | failed |
| Completed: Reversed | 被收款行退回 | failed |

**MSO 主体与 Navro 账户关系**：
- Obita 使用 MSO 牌照主体在 Navro 以商户身份 onboard
- Navro 为每个商户开设银行账户（命名格式：`Navro-商户xxxxx@银行`）
- Navro 充值资金仅计入 MSO 主体可用余额，不支持 TCSP 主体入账

## Payout 付款产品

### 产品定义

Obita 代商户向第三方（非同名企业/个人）的体外账户（钱包地址/银行卡号）进行付款的资金类产品。

### 支持的 4 种场景

| 场景 | 资金来源 | 收款目标 | 是否涉及兑换 | 是否涉及 Web3 钱包 |
|------|---------|---------|------------|------------------|
| 数币→数币（同币种） | 数币余额 | 收款人钱包地址 | 否 | 是 |
| 数币→数币（Swap） | 数币余额 | 收款人钱包地址 | 是 | 是 |
| 法币→数币 | USD 余额 | 收款人钱包地址 | 是 | 是 |
| 法币→法币 | USD 余额 | 收款人银行卡 | 是（USD/小币种） | 否 |

### 暂不支持的能力

- 数币→法币
- TCSP 下的法币→法币付款能力

### 余额不足提示逻辑

| 场景 | 提示文案 |
|------|---------|
| 法币收款账户余额不足 | "请联系 Obita 运营人员进行资金预处理" |
| 数币收款账户余额不足 | "请进行充值" |

### 审批节点

- Payout 订单有**商户侧审批**和**运营侧（OOMP）审批**两个审批节点

### 退款规则

- 非客户因素导致的链路失败（包括风控 pending），**不做业务失败处理**
- 客户因素导致的失败：按原汇率反转退回，汇率损差由客户承担
- 法币→法币场景：
  - Navro 已发生换汇后失败（COMPLETED+REJECTED）：Obita 吸差不退
  - 未发生换汇的失败：吸差金额退回

### 业务主体打标

- Payout 订单在存储层需要有 Obita 业务主体的打标（区分 TCSP/MSO）
- Portal 显示上暂不体现

## Conversion 兑换引擎

### 产品定位

统一的可扩展兑换引擎，负责法币↔数币、数币↔数币、法币↔法币（跨币种）之间的安全、合规、可审计转换服务。

### 分层架构

| 层级 | 名称 | 职责 |
|------|------|------|
| L1 | 接入层 | 接收兑换请求，参数校验 |
| L2 | 报价与路径决策层 | 汇率报价、路径规划 |
| L3 | 交易编排与恢复层 | 执行计划生成、原子执行、异常恢复 |
| L4 | 内部资源抽象层 | 内部账本操作抽象 |
| L5 | 外部依赖适配层 | 外部 LP/API 对接适配 |

### 核心原则

- **原子执行**：N-Legs（可变腿数），每条腿独立执行
- **报价唯一凭证**：Quote Package 作为全流程唯一报价标识
- **状态隔离**：对客极简（Created→Processing→Completed/Failed），内部异常通过 Recovery 闭环

### 报价类型

| 报价类型 | 说明 | 适用场景 |
|---------|------|---------|
| Firm | 锁价可成交 | 自动化交易 |
| Indicative | 参考不可成交 | 路径规划参考 |
| Dealer | 人工报价 | 复杂/大额交易 |

多腿聚合优先级：dealer > indicative > firm

### Quote Package

报价唯一凭证，包含以下要素：

| 要素 | 说明 |
|------|------|
| client_quote | 客户侧报价信息 |
| route_plan | 路径规划 |
| quote_legs | 报价腿明细 |
| price_snapshot | 价格快照 |
| quote_type | 报价类型（Firm/Indicative/Dealer） |
| TTL | 报价有效期 |

### 支持的 7 种业务场景

| 场景 | 交易方向 | 账本归属 | 流动性提供方 |
|------|---------|---------|------------|
| On Ramp | 法币→数币（USD→USDT/USDC） | TCSP | BVI 商户（自营） |
| Off Ramp | 数币→法币（USDT/USDC→USD） | TCSP | BVI 商户（自营） |
| Swap | 数币↔数币（USDT↔USDC） | TCSP | BVI 商户（自营） |
| 体外小币种 Payout | 数币/法币→小币种+Payout | TCSP+体外记账 | Velafi（体外 LP） |
| 法币 FX Payout | 法币→其他法币+Payout | MSO | Navro（一级 LP 直连） |
| 跨账本 FX Payout | TCSP→MSO 跨账本+FX+Payout | TCSP+MSO 双账本 | Navro |

### 汇兑路径

| 路径 | 说明 | 处理方 |
|------|------|--------|
| 数币跨币种 | USDC↔USDT swap | Conversion 内部 |
| 法币→数币 | USD→USDT/USDC | Conversion 内部（通过 BVI） |
| 法币→法币 | USD→小币种 | Navro FX quote（Obita 加点差） |

### 汇率吸差机制

- Obita 在 Navro 开设**汇率吸差账户**，用于覆盖汇率波动
- 执行过程资金变动客户无感（通过内部户/业务过渡户规避）
- 收款人精准到账：收款人收到的币种和金额不受汇率波动影响

### 执行腿（Execution Leg）状态机

| 状态 | 说明 | 是否终态 |
|------|------|---------|
| not_started | 已创建，未执行 | 否 |
| processing | 执行中 | 否 |
| pending_manual | 等待人工处理（GLDB 手动转账） | 否 |
| success | 成功 | 是 |
| failed | 失败（系统原因/库存不足，可恢复） | 否（可触发 Recovery） |
| cancelled | 取消 | 是 |
| skipped | 因路径变化/补偿而跳过 | 是 |

### 主单（Conversion Order）状态

| 状态 | 说明 |
|------|------|
| Created | 主单已创建 |
| execution_planning | 生成执行计划中 |
| Processing | 执行中 |
| Recovering | 异常恢复中 |
| Completed | 全部腿成功 |
| Failed | 异常不可恢复或人工判定失败（已成功的腿保留 success，后续通过 Recovery/Compensation 处理） |

### 异常恢复机制

- **Recovery Case**：恢复工单，执行异常后的恢复处理对象
- **Compensation Record**：补偿记录，已产生影响需冲正/回滚/补偿的记录
- failed 状态的执行腿可通过 Recovery 重新执行

## Payee（收款人）管理

- 分为**个人**和**公司**两种类型
- 基础信息：姓名/公司名（必填）、邮箱、国家（必填）、手机号、地址
- 收款账户支持两种类型：
  - **银行账户**：国家/币种/支付方式/SWIFT/IBAN
  - **钱包地址**：链名/地址
- 支付方式按国家+币种路由：WIRE/SWIFT、SEPA、FPS
- 状态机：init → active → suspend → reject
- 禁用/删除状态的收款人在 Payout 时不可选择

## Web3 钱包地址（代收代付）

### 用途

合规要求，数币 Payout 的资金中转账户。

### 规则

- **开通**：在 Invoice 项目中完成，Payout 复用
- **校验**：小额打款验证（Small Deposit Verification，详见下方专节）
- 所有数币 Payout 订单都必须经过 Web3 钱包地址
- 暂不支持从商户代收代付 Web3 钱包地址发起 Payout
- 钱包地址需要定期更新（在会员/商户模块中维护）

### to_address 两种模式

| 场景 | to_address 取值 |
|------|----------------|
| 商户白名单场景 | TCSP 托管主地址 |
| 第三方外部地址场景 | Obita 代收代付 Web3 地址（来款不退回） |

## 小额打款验证（Small Deposit Verification）

### 产品定位

轻量、通用的 Web3 地址控制权验证能力，与具体业务场景解耦。

### 流程

1. 上层业务传入待验证地址+链信息
2. 系统创建验证单据
3. 生成随机金额（0.01-0.99）
4. 验证者使用该地址转入指定金额
5. 系统按 `from_address + chain + currency + amount` 精准匹配
6. 匹配成功 → verified

### 状态机

pending → verified（终态）

## 安全与合规模块（OOMP Security & Compliance）

### Risk Case

- 系统根据 Sumsub（KYC/KYB/KYT/AML）和内部风控规则自动生成风险事件
- 角色与分层：Ops（初审）→ Risk Control（风控审核）→ Compliance（终审）
- 处理原则：系统先判断，人工再判断，合规兜底
  - 100% 确认风险 → 系统自动拒绝
  - 非 100% → 人工审核
  - 事后审查类 → 不阻塞交易但留痕

### 风险类型

| 类型 | 全称 | 处理方式 |
|------|------|---------|
| Sanction | 制裁名单 | 系统自动拒绝 |
| PEP | 政治敏感人物 | 人工审核 |
| ADN | 负面媒体 | 人工审核 |
| SAR | 可疑活动报告 | 事后审查，不阻断交易 |
| LVT | 大额交易 | 事后审查，不阻断交易 |

### 数据来源

- Sumsub（现阶段直连）+ 自研风控系统（后续直连）+ 业务系统

### SLA

| 环节 | SLA |
|------|-----|
| 运营初审 | 24H |
| 风控审核 | 2D |
| 合规审核 | 2D |
| 材料提交 | 7 自然日 |

### 分期范围

- 一期：AML/SAR/LVT/黑白名单
- 二期：KYC/KYB/KYT/Travel Rule

## ObitaPocket（C 端钱包）

### 产品定位

商户 Payout 向 C 端用户发放薪资/奖励/佣金/补贴的钱包产品。

### 技术方案

- 基于 Web3Auth（本期仅 Email 登录，MFA 禁用）

### 商户预开户

- 商户添加 payee 邮箱 → 系统自动创建 C 端账户 → 欢迎邮件由 Obita 发送

### 记账模型

- **Individual Ledger**（C 端内部账本）
- 资金流入先不写链上，仅在账本记账
- Send 请求时才发起链上转账
- 总可用余额 = Σ 流入 − Σ 已 Send

### Send 逻辑

- 按 Tx ID 时间顺序依次匹配
- 可拆分 Tx ID 金额
- 按商户维度统计并推送 Send 申请

### KYC Level-1

- 首次进入 C 端触发
- 提交姓名（不做真实性校验，自动通过）
- 终身累计 Send < 50,000 USDT/USDC

### Send 状态

| 状态 | 说明 |
|------|------|
| Merchant in Review | 商户审核中 |
| Processing | 处理中 |
| Completed | 全部完成 |
| Partial Completed | 部分完成 |
| Failed | 失败 |

### 登录方式

- Payout 邮件直达
- 邮箱 OTP
- Google 一键登录
- 2FA session 24h 有效

## 风控相关

### 交易风控

- 审制裁量维度：交易金额阈值、交易对手方身份（制裁筛查）、交易频率
- 处理方式：自动通过、补充材料、拦截交易、线下处理
- 同步事件：实时返回风控结果（通过/高风险/补充材料）
- 异步事件：事后同步风控侧结果信息（如绑卡完成、解绑结果）

### 交易金额风控策略参考

| 交易金额 | 风控策略 | 处理方式 | 对客表现 |
|---------|---------|---------|---------|
| <500 USD | 自动通过 | 交易成功 | 无感知 |
| 500~10000 USD | 基于历史交易数据判断 | 自动通过 or 补充材料 | 成功 or Need Action |
| >10000 USD | EDD | 补充材料 | Need Action |
| 命中制裁 | 拦截 | 交易失败，不可申诉 | 失败 |

### 绑卡风控

- 同名校验：Account Name 与 onboard 注册主体名称是否一致，支持匹配度范围阈值
- 格式校验：Account Number、BIC 等字段格式是否符合目标国家银行规范
- Bank Statement 验证：核对 Bank Statement 上的账户号、户名与填写信息是否一致
- 支持 OCR 自动审核 Bank Statement 文件

### 风控处置（高风险资金交易）

- 部分订单基于风控审核后被定义为风险资金交易
- 此时资金已在 SGB 侧完成资金入账，风控系统需进行系统报警
- 由风控、合规一同确认资金处置方式（线下打款退回 or 资金冻结）
- Portal 内部错误应技术报警而非对客暴露

## 审批规则

### 优先评估模型

两级优先：商户级 > 牌照级 > 全局级，同级别首条匹配生效。

### 规则匹配逻辑

- 审批规则按「业务类型 + 币种 + 审批模式」进行重复校验，同条件仅允许配置一条
- 支持基于金额阈值的分级审批人路由（如 >100 A 审核，>500 B 审核）
- 金额匹配示例：300 订单命中 A 审核，800 订单命中 A + B 审核

### 配置保护机制

| 机制 | 说明 |
|------|------|
| 遮罩警告 | 编辑已生效规则时展示风险提示 |
| 自动排序 | 按优先级自动排列规则，防止人为排序错误 |
| 空阈值置底 | 金额阈值为空的规则自动排到同级别末尾 |

### 审批人邀请

- 邮箱作为唯一身份键，未注册邮箱自动创建账户
- 审批流程：客户发起交易 → 冻结资金 → 进入「待审批」状态 → 审批人审批 → 通过推进流程 / 拒绝交易失败
- 双人审核：审核人不能与制单人为同一人

### 审批节点（按业务场景）

| 业务场景 | 审批要求 |
|---------|---------|
| 提现（Portal） | 全部需要审批（运营侧双人审核） |
| 提现（API） | 不考虑商户审批规则 |
| 手工补录订单 | 必须双人审核 |
| 绑卡 | 风控审核，无需人工审批 |

## API 认证体系

- 商户 API Key：用于标准商户接口调用
- 授权 API Key：用于委托/授权场景的接口调用

## 计费模型

### 费率层级

| 层级 | 说明 | 优先级 |
|------|------|--------|
| 全局兜底费率 | 系统级默认费率，适用于所有无特殊约定的客户 | 最低 |
| 商户级费率 | 针对特定客户/合同的单独定价 | 最高（优先于全局生效） |

### 查询逻辑

客户发起交易时，系统查询该客户是否存在商户级费率配置：
- 存在 → 取商户级费率
- 不存在 → 取全局兜底费率
- 新订单以创建时的费率为准，费率变更不影响已存在的进行中订单

### 提现计费

- 扣费模式：预扣制（提现交易冻结资金 = 提现本金 + 手续费）
- 失败退回：提现失败时，冻结的本金和手续费全额解冻
- 对客展示：提现提交前展示预估手续费和预计到账金额，手续费为 0 时展示「免手续费」
