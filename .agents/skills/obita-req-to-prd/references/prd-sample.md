# PRD 样例 — 写作格式参考

> **用途声明**：本文件展示 Obita PRD 的标准写作格式和组织方式。
> 以下 21 种写作手法（表格/流程图/泳道图/Use Case/可交互原型/名词解释/改造对比/逐步骤描述等）为**通用格式**，适用于任何业务场景的 PRD 撰写。
> 样例中的具体内容（字段名、状态值、角色、系统名等）取自法币充提场景，仅供参考格式，撰写其他业务 PRD 时应替换为对应场景的实际内容。

## 关键写作手法示例

### 1. 用表格定义字段

订单流水字段定义示例：

| 字段 | 说明 |
|------|------|
| 交易流水号 | 业务订单号 |
| 交易类型 | Top Up |
| 交易对手方 | 展示充值来款账户名称 |
| 交易金额 | 实际入账金额（扣除手续费后的到账金额），带币种 |
| 订单状态 | 成功、处理中、待处理（补充材料）、失败 |
| 操作 | 详情、去处理 |

### 2. 用表格做模块清单

| 模块 | 改造内容 |
|------|----------|
| 业务端（Portal） | 充值功能页、提现功能页、订单查询页 |
| 运营平台（OOMP） | 商户管理、交易管理、手工补录、审核流程 |
| Payment | 入金服务、付款服务 |
| 账务系统 | 入金记账、付款记账 |
| 风控系统 | 入金/出金审核、合规校验 |
| 金融网关 | SGB 渠道对接、入金通知、付款指令、对账清算 |
| 会员系统 | 客户 ID 与 VA ID 绑定关系、集团会员体系下账户查询权限 |

### 3. 用表格做状态映射

| 对客展示状态 | 系统内部状态 | 说明 |
|-------------|-------------|------|
| pending | pending | 业务认领订单，风控审核 |
| Need action | risk-pending | 风控要求客户提供证明材料，待客户补充材料 |
| success | Success | 风控审核通过，资金已入账 |
| failed | Failed | 风控审核拒绝，或系统异常 |
| reversed | reversed | 渠道冲正，资金退回 |

### 4. 用表格做异常场景覆盖

| 异常场景 | 处理链路 |
|----------|----------|
| 客户银行退票 | SGB 通知 → 入金服务创建逆向订单 → 账务冲减余额 → 通知客户 |
| 订单未匹配 | 入金服务 → 差错系统挂账池 → 运营手工认领 → 入账 |
| 重复入账 | 账务系统幂等校验 → 重复订单自动拒绝并告警 → 运营手工处理 |
| 收款账户信息有误 | 渠道返回失败状态码 → 金融网关回调提现服务 → 订单状态更新为 Failed → 预冻结资金自动解冻 |
| 重复付款（幂等拦截） | 提现服务生成唯一 Withdraw_Order_ID，渠道基于流水号做幂等校验，重复请求自动拒绝 |

### 5. 用表格做风控策略

| 判断项 | 校验策略 | 是否命中 | 处理方式 | 对客 |
|--------|---------|---------|---------|------|
| 交易对手方身份 | 查询交易对手方账户是否命中制裁 | 是 | 拦截交易 | 交易失败，不可申诉 |
| 交易对手方身份 | 同上 | 否 | 通过筛查 | 无影响 |
| 交易金额 | 小额交易（<500USD） | 是 | 自动通过 | 交易成功 |
| 交易金额 | 中额交易（500~10000USD） | 是 | 基于历史交易数据判断 | 交易成功 or 补充材料 |
| 交易金额 | 大额交易（>10000USD） | 是 | EDD | 补充材料 |

### 6. 用表格做邮件通知触发节点

| 系统节点 | 对客通知邮件 |
|---------|------------|
| success | 模版1（入账成功） |
| failed | 模版2（交易失败） |
| risk_pending | 模版3（补充材料） |

### 7. 客户旅程泳道图

客户旅程章节开头必须包含泳道图，用 Mermaid 按角色分泳道展示客户操作完整链路。泳道图侧重**用户视角的操作路径**，与第六章系统交互流程图（侧重系统间调用时序）形成互补。

**重要：泳道图和所有 Mermaid 流程图在 HTML 输出中必须使用 Mermaid.js 直接渲染为图形**（引入 CDN，`<pre class="mermaid">` 标签），不要输出代码块。

**sequenceDiagram 泳道图示例**：

```
sequenceDiagram
    participant C as 客户
    participant P as Portal
    participant S as 业务服务
    participant O as OOMP

    rect rgb(234, 243, 222)
    Note over C,P: 客户操作阶段
    C->>P: 进入充值页面
    P->>C: 展示法币资产卡片列表
    C->>P: 点击「充值」
    P->>C: 展示充值方式选择页
    C->>P: 选择「银行转账」
    P->>C: 展示 VA 收款账户信息
    end

    rect rgb(220, 240, 255)
    Note over P,S: 系统处理阶段
    C->>P: 完成外部转账
    S->>S: 渠道来账通知 → 订单认领 → 风控审核
    end

    rect rgb(255, 248, 230)
    Note over S,O: 异常处理阶段（可选）
    S->>O: 风控标记需补充材料
    O->>P: 通知客户补充材料
    C->>P: 上传补充材料
    S->>S: 重新风控审核
    end

    rect rgb(200, 255, 200)
    Note over S,C: 完成阶段
    S->>P: 订单状态更新为 success
    P->>C: 展示充值成功 + 邮件通知
    end
```

**泳道图颜色分区规则**：
- 客户操作：`rect rgb(234, 243, 222)` 浅绿区块
- 系统处理：`rect rgb(220, 240, 255)` 浅蓝区块
- 异常处理：`rect rgb(255, 248, 230)` 浅黄区块
- 完成阶段：`rect rgb(200, 255, 200)` 绿色区块

### 8. 名词解释表格

第二章「核心概念说明」末尾必须包含名词解释表格，列出本 PRD 涉及的所有专业术语。先引用 `business-context.md` 术语表已有条目，再补充本 PRD 新增术语。

| 术语 | 解释 |
|------|------|
| VA (Virtual Account) | 虚拟账户，Obita 为每个客户在渠道侧开立的法币收款账户 |
| TCSP | 信托公司服务提供商（Trust Company Service Provider），Obita 的数币业务牌照主体 |
| MSO | 货币服务运营机构（Money Service Operator），Obita 的法币业务牌照主体 |
| Onboard | 客户开户流程，完成 KYC 后激活账户 |
| Payout | 付款功能，支持客户将资产支付至外部收款方 |
| Payee | 收款方，记录客户常用的收款方信息（个人/公司、钱包地址/银行账户） |

**格式要点**：
- 术语列：中文术语名，首次出现的术语可括号标注英文原文
- 解释列：简洁明了的一句话定义，不超过 50 字
- 排列顺序：按术语在 PRD 中首次出现的顺序排列
- 已有术语标注来源：「(详见 business-context.md)」

### 9. 可交互原型

第三章「客户旅程」开头必须包含可交互原型文件的链接。该文件是一个独立的 HTML 页面，基于 PRD 客户旅程内容自动生成，模拟完整的用户操作路径。

**在 PRD 中的引用格式**：

```html
<div class="interactive-prototype">
  <p>▶ <a href="./Interactive-法币充值提现-v2.2.html" target="_blank">点击查看可交互原型</a></p>
  <p class="note">该原型模拟了客户从法币主页到充值完成的全链路交互过程，
  支持页面跳转、按钮点击、表单填写等操作。</p>
</div>
```

**可交互原型文件规范**：
- 文件命名：`Interactive-[功能简称]-v[版本号].html`，与 PRD 文件版本号一致
- **视觉风格必须严格遵循 Obita Portal 设计规范**（详见 `references/portal-style-guide.md`，来源：OOMP 前端代码仓库 React + Tailwind CSS + shadcn/ui），除非用户提供新的交互风格要求，否则不得偏离：
  - 品牌蓝色系：按钮/强调色 `#0072F7`（`hsl(221.2 83.2% 53.3%)`）、输入框边框 `border-blue-200`（`#BFDBFE`）、卡片/表格边框 `border-blue-100`（`#DBEAFE`）、表头渐变背景 `#F7FAFF`、表格行 hover `#EFF6FF`
  - 组件尺寸：按钮 36px 高 / 6px 圆角（`h-9 rounded-md`）、输入框/Select 36px 高 / 8px 圆角（`h-9 rounded-lg`）、卡片 12px 圆角（`rounded-xl`）、对话框 8px 圆角最大 512px（`sm:max-w-lg`）
  - 页面布局：侧边栏 256px（`w-64`）+ 顶栏 64px（`h-16`）+ 内容区 `p-6` / `gap-6`、最小宽度 1100px（`min-w-[1100px]`）
  - 状态色：完成 `#0072F7`（蓝）、进行中 `#F59E0B`（amber）、错误 `#EF4444`（红）、待处理 `gray-200`
  - 字体：`system-ui, -apple-system, sans-serif`，正文 14px（`text-sm`），标题 `font-semibold`
  - 步骤指示器：圆形 `w-3 h-3 rounded-full border-2`，使用上述状态色
- 交互能力：支持页面跳转（步骤间前进/后退）、按钮点击切换状态、表单填写模拟
- 内容范围：覆盖客户旅程的所有 Step，每个 Step 一个页面
- 与 PRD Mockup 保持视觉一致，可视为 Mockup 的可交互版本

### 10. Use Case 用例格式

Use Case 章节使用表格描述每个业务场景的完整用例：

| 项目 | 内容 |
|------|------|
| **用例名** | 来账入金 |
| **触发条件** | 客户通过外部银行向 Obita VA 账户发起转账 |

### 11. Mermaid 系统交互流程图用法

**重要：所有 Mermaid 流程图在 HTML 输出中必须使用 Mermaid.js 直接渲染为图形**（引入 CDN，`<pre class="mermaid">` 标签），不要输出代码块。

**sequenceDiagram 示例（整体业务流程）**：

```
sequenceDiagram

    金融网关->>业务域: 广播入账通知
    业务域->>金融网关: 认领订单
    金融网关->>业务域: 推进入账
    业务域->>风控系统: 入金订单风控审核
    风控系统->>业务域: 返回风控结果

    alt 需补充材料
        风控系统->>业务域: 补充材料流程
    else 风控通过
        业务域->>payment: 推进入账
    end
```

### 12. 用户旅程逐步骤描述格式（页面原型 + 功能描述双列布局）

客户旅程章节采用**双列布局**：左列展示自动生成的页面 Mockup（HTML/CSS 模拟的 UI 组件），右列描述该页面模块的功能内容。

**Mockup 生成规则**：
- 每个步骤根据需求描述自动生成页面 Mockup，不需要用户提供截图
- Mockup 使用 HTML/CSS 构建，模拟真实页面 UI 元素：卡片、表单、按钮、列表、状态标签、面包屑导航等
- **Mockup 风格必须与 Obita Portal 一致**，严格遵循 `references/portal-style-guide.md` 中的设计规范（品牌色系、组件样式、排版、间距、圆角等），除非用户提供新的交互风格要求，否则不得偏离
- Portal 核心视觉特征：蓝色系（卡片/输入框使用蓝色边框，表格使用蓝色表头和分隔线），状态色为蓝色=完成、amber=进行中、红色=错误
- Mockup 中填充基于需求描述的模拟数据，展示真实的页面结构和交互元素
- 如用户提供 Figma 截图，优先使用截图替换自动生成的 Mockup

### 20. 场景对比表格

法币入金场景分类使用对比表格，清晰区分不同入金方式的链路和特点：

| 入金方式 | 客户链路 | 特点 |
|---------|---------|------|
| 来账入金 | 客户获取收款账户 → 客户向收款账户转账 → Obita 收到来账通知 → 匹配入账 | 入金时间不可控，入金金额不确定；入金订单弱关联业务流程 |
| 订单入金 | 客户发起入金交易 → Obita 请求渠道生成付款订单 → 客户支付 → Obita 查询状态 → 入账 | 入金状态可控，金额明确；入金交易强关联业务流程 |
| 渠道入金 | 客户发起收款交易 → 渠道生成付款订单 → 客户支付 → 定时清算 → 入账 | 入金时间依赖渠道清结算时间；交易强关联业务流程 |

### 21. 改造前后功能对比表格

第五章「版本说明与改造范围」中必须包含改造前后功能对比表，用于清晰展示每个功能点改造前后的差异，帮助研发和测试快速理解变更范围。

| 功能点 | 改造前 | 改造后 |
|--------|--------|--------|
| 充值方式 | 仅支持 SGB 来账充值 | 新增订单入金方式，客户可主动发起充值 |
| VA 账户展示 | 展示单一 VA 账户信息 | 按牌照主体分组展示多个 VA 账户 |
| 入金通知 | 仅支持邮件通知 | 新增 Portal 站内通知 + 邮件通知 |
| 手续费展示 | 固定显示 0.5% | 动态查询费率配置，按金额区间展示 |
| 充值订单状态 | 3 种状态（Pending/Success/Failed） | 新增 Pending-Risk 状态，支持风控审核流程 |

**格式要点**：
- 三列表格：功能点（改造涉及的模块或页面）/ 改造前（当前行为）/ 改造后（改造后行为）
- 按模块或页面分组排列，与 5.2 涉及改造模块的顺序一致
- 改造前描述当前系统行为，改造后描述本次改动后的预期行为
- 如果是新功能（无改造前），改造前列填写「无（新增）」
- 如果是移除功能，改造后列填写「移除」
