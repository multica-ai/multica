# 产品需求文档编写（PRD Writer）

## Overview

为 Obita 支付平台产品经理生成功能需求文档（PRD）。输出格式为独立 HTML 文件，支持浏览器直接预览，也可导出为 Word 文档。文档严格遵循 Obita PRD 模板结构、写作规范和排版标准。

## 写作规范

### 核心原则

1. **两大板块结构**：PRD 文档必须分为「第一部分：业务解释」（一~四章：背景→概念→旅程→Use Case）和「第二部分：产品实施细节」（五~八章：版本→功能详解→配套→附录）两大板块。**绝对禁止**不按两大板块组织内容或混淆板块归属。这是红线规则。
2. **结构化优先**：所有功能描述必须使用表格对比、枚举列表、结构化编号呈现，拒绝纯段落叙述
3. **表格是标配**：字段定义、状态映射、模块清单、异常场景等一律用 Markdown 表格
4. **流程图必备且必须渲染为图形**：涉及多角色/多系统交互的流程，必须附带 Mermaid 流程图（sequenceDiagram 或 flowchart），并使用颜色分区标注不同阶段。**所有 Mermaid 图在 HTML 输出中必须使用 Mermaid.js CDN 直接渲染为图形（`<pre class="mermaid">` 标签），绝对禁止输出为代码块（`<pre><code>`）**。这是红线规则，每次输出必须遵守。
5. **第三章客户旅程必须完整**：第三章**必须**完整包含以下三个部分，**缺一不可**，缺少任何一部分均视为文档不合格：
    - **3.X.1 可交互原型链接**：蓝色卡片样式链接引用 Interactive HTML 文件
    - **3.X.2 泳道图**：Mermaid sequenceDiagram 按角色分泳道，`<pre class="mermaid">` 渲染为图形，带颜色分区
    - **3.X.3 关键环节详解**：拆分为 ≥3 个 Step，每个 Step 采用左列 Mockup + 右列功能描述的双列布局
    **绝对禁止**省略其中任何一节，也**绝对禁止**用其他章节的内容替代客户旅程。这是红线规则。
6. **客户旅程每一步必须输出页面 Mockup**：第三章「客户旅程」的每个 Step 必须采用双列布局——左列输出可交互的 HTML/CSS 页面 Mockup（模拟真实 Portal UI 组件：卡片、表单、按钮、列表、状态标签等），右列输出功能描述（链路、前置条件、展示内容、数据来源、特殊情况）。**绝对禁止只输出文字描述而不生成 Mockup**。Mockup 必须遵循 Obita Portal 视觉风格规范。这是红线规则。
7. **对客描述清晰**：面向客户的文案必须可理解、可直接使用，不可仅展示内部状态码或技术术语
8. **边界情况覆盖**：每个功能模块需单独列出异常场景和处理规则
9. **先总览后细节**：文档先给出整体概览，再逐模块展开详细设计
10. **知识回流必须执行**：PRD 交付完成后**必须**执行知识回流（Step 6），检查是否有新业务规则、新系统模块、新状态机、新渠道信息、新术语、新写作格式需要沉淀到 `business-context.md` 或 `prd-sample.md`。这是红线规则，不可跳过。
11. **可交互原型必须生成**：第三章「客户旅程」**必须**产出独立的可交互 HTML 原型文件（`Interactive-[功能简称]-v[版本号].html`），并在 PRD 中以链接引用。**绝对禁止**只输出 PRD 文档而不生成可交互原型。这是红线规则。
12. **名词解释表格必须有**：第二章「核心概念说明」末尾**必须**包含「名词解释」表格，列出本 PRD 涉及的所有专业术语。**绝对禁止**省略此表格。这是红线规则。
13. **改造前后功能对比表必须有**：第五章「版本说明与改造范围」**必须**包含 5.3 改造前后功能对比表（功能点/改造前/改造后）。**绝对禁止**省略此表格。这是红线规则。
14. **Portal 风格必须一致**：所有 Mockup 和可交互原型的视觉风格**必须**严格遵循 Obita Portal 设计规范（品牌蓝色系、组件尺寸、布局结构、状态色、字体等，详见 `references/portal-style-guide.md`）。**绝对禁止**使用通用/默认/非 Portal 风格。这是红线规则。
15. **双文件交付**：PRD 交付**必须**产出两个文件：PRD 正文 HTML + 可交互原型 HTML。**绝对禁止**只交付单文件。这是红线规则。
16. **泳道图必须使用颜色分区**：所有 Mermaid 流程图（泳道图、系统交互图、资金流图）**必须**使用 `rect rgb(...)` 标注不同阶段色彩分区（客户操作/系统处理/异常处理/完成等）。**绝对禁止**输出无颜色分区的纯白流程图。
17. **写前必须读取知识库**：开始撰写 PRD 之前（Step 2 输出大纲之前），**必须**先读取以下四个知识库文件，获取业务上下文、写作格式、Portal 风格规范和组件 API 参考。这是红线规则，不读取知识库就动笔视为流程违规：
    - `references/business-context.md`：Obita 平台业务知识库（产品模块、牌照体系、账户架构、渠道信息、术语表等），确保名词解释引用已有术语、业务规则不与已沉淀知识冲突
    - `references/prd-sample.md`：PRD 写作手法 & 格式规范（21种写作手法），确保文档格式和写作风格符合规范
    - `references/portal-style-guide.md`：Portal 前端设计规范，确保 Mockup 和可交互原型的视觉风格与真实 Portal 一致
    - `references/antd-skill.md`：Ant Design 组件查询参考（基于 @ant-design/cli），Portal 中使用的 antd 组件（Table、Form、Select、DatePicker 等）的 API、Props、Demo 必须通过此文件指引的 CLI 查询确认，不得凭记忆编写
18. **Mockup 和原型必须基于实际页面结构产出**：第三章客户旅程的左列 Mockup 和可交互原型 HTML **必须**基于以下来源之一确定页面结构和内容布局（按优先级从高到低）：
    1. **用户提供的 Portal 实际源码/文件路径**（最高优先级）：如果用户提供了具体代码路径或源码片段，**必须**先读取并理解其中的页面结构、字段定义、组件组合方式，再据此生成 Mockup。不得跳过阅读直接凭印象输出。
    2. **用户提供的截图/Figma/原型图**：逐字段、逐组件地还原截图中的页面布局，包括字段名称、排列顺序、表单分组、表格列头等细节。
    3. **business-context.md 中的模块描述**：如果知识库中有相关业务模块的功能描述和界面说明，以此为基础构建页面结构。
    **绝对禁止**：在不了解目标页面实际结构的情况下，用通用模板（如 report center）套用到所有业务模块。**绝对禁止**：不同业务模块（账单、交易流水、Payout、充值、审批等）使用相同的页面模板仅替换标题。每个涉及的模块必须按其真实页面结构独立构建 Mockup。这是红线规则。

### 排版要求

- 使用中文标点符号
- 一级标题用「一、」「二、」等中文数字编号
- 二级标题用「1.1」「1.2」数字编号
- 三级标题用「1.1.1」「1.1.2」
- 表格必须有表头行
- 流程图中角色/系统用正式中文名称（客户、Portal、提现服务、风控系统、金融网关、OOMP、渠道）

### 红线规则

- **两大板块结构**：PRD 必须分为「第一部分：业务解释」和「第二部分：产品实施细节」两大板块。不按两大板块组织内容或混淆板块归属视为结构错误。
- **不确定就停，不要自展**：撰写过程中如果对任何业务规则、状态定义、交互逻辑、字段含义、系统关系等内容存在不确定，必须**立即停下来向用户提问确认**，不得自行推测、脑补或展开。宁可多问一句，也不要写出错误内容让用户事后纠正。此规则贯穿 Step 1 到 Step 6 全流程。
- **第三章客户旅程必须完整（三部分缺一不可）**：第三章必须同时包含 3.X.1 可交互原型链接 + 3.X.2 泳道图（`<pre class="mermaid">` 渲染 + 颜色分区）+ 3.X.3 关键环节详解（≥3个 Step 双列布局）。省略其中任何一节、或用其他章节内容替代客户旅程，均视为文档不合格。
- **Mermaid 必须渲染为图形**：所有 Mermaid 流程图（泳道图、系统交互图、资金流图、状态机图）在 HTML 输出中**必须**使用 Mermaid.js CDN 直接渲染为可见的图形，**绝对禁止**输出为 `<pre><code>` 代码块。每次写完包含 Mermaid 的章节后，必须确认所有流程图都以 `<pre class="mermaid">` 形式存在，且 HTML `<head>` 中引入了 Mermaid.js CDN 和 `mermaid.initialize({startOnLoad:true, theme:'base', themeVariables:{primaryColor:'#E8F0FE', primaryTextColor:'#1E293B', primaryBorderColor:'#BFDBFE', lineColor:'#64748B', secondaryColor:'#FFF7ED', tertiaryColor:'#F0FDF4'}})` 配置。
- **客户旅程每一步必须输出 Mockup**：第三章每个 Step **必须**输出左列 Mockup + 右列功能描述的双列布局。左列 Mockup 必须是可渲染的 HTML/CSS 页面组件（卡片/表单/按钮/列表/状态标签等），**绝对禁止**只输出文字描述、纯文本框线、或占位符（如"此处为UI图"）。Mockup 必须严格遵循 Obita Portal 视觉风格规范（品牌蓝色系、组件尺寸、布局结构）。
- **知识回流必须执行**：PRD 交付完成后**必须**执行 Step 6 知识回流流程。即使本次无新增通用知识，也必须显式向用户说明「已检查，本次无新增通用知识需回流」，**不得静默跳过**。
- **可交互原型必须生成**：第三章交付时**必须**同时产出可交互原型 HTML 文件，PRD 中**必须**以链接引用。只输出 PRD 不输出原型视为交付不完整。
- **名词解释表格必须有**：第二章末尾**必须**包含名词解释表格，省略此表格视为结构不完整。
- **改造前后功能对比表必须有**：第五章**必须**包含 5.3 改造前后功能对比表，省略此表格视为结构不完整。
- **Portal 风格必须一致**：Mockup 和可交互原型的所有页面**必须**严格遵循 Obita Portal 视觉规范（品牌蓝 #0072F7、输入框 border-blue-200、卡片 border-blue-100、布局 256px 侧边栏+64px 顶栏、按钮 36px/6px 圆角、输入框 36px/8px 圆角、卡片 12px 圆角、状态色蓝/amber/红/灰、字体 system-ui 14px）。使用通用/默认/非 Portal 风格视为质量不达标。
- **双文件交付**：必须同时产出 PRD 正文 HTML 和可交互原型 HTML 两个文件。只交付单文件视为交付不完整。
- **泳道图必须使用颜色分区**：所有 Mermaid 流程图**必须**使用 `rect rgb(...)` 标注不同阶段色彩分区。输出无颜色分区的纯白流程图视为质量不达标。
- **写前必须读取知识库**：Step 2 输出大纲之前，**必须**先读取以下四个知识库文件，否则不得开始撰写：
  1. `references/business-context.md`——获取业务上下文（产品模块、牌照、账户、渠道、术语表等），确保名词解释引用已有术语、业务规则不与已沉淀知识冲突

  不读取知识库就动笔视为流程违规。

### 禁止事项

- 禁止在 PRD 中输出 API 接口设计（字段定义、请求/响应结构等）
- 禁止输出埋点方案
- 禁止使用推测或编造的内容作为事实，所有业务规则必须基于用户提供的上下文或已确认的决策
- 禁止在不确定时自行展开或假设，必须先提问
- 不同主题的功能需求应单独出文档

## PRD 文档结构模板

生成 PRD 时，必须严格按以下结构组织内容（根据具体功能裁剪章节）。文档分为**两大板块**：

- **第一部分：业务解释**——面向业务方/产品/设计，回答"做什么、为什么、用户怎么用"
- **第二部分：产品实施细节**——面向研发/测试/运营，回答"怎么做、改哪些、异常怎么处理"

```
═══════════════════════════════════════
  第一部分：业务解释
═══════════════════════════════════════

一、背景与目标
    1.1 业务背景           ← 为什么做，解决什么问题
    1.2 需求目标           ← 交付范围和目标

二、核心概念说明
    2.X [核心概念/实体关系]  ← 按需展开各概念
    2.X+1 名词解释          ← 本 PRD 涉及的术语对照表
         （格式：表格，两列：术语名 | 解释。
          先引用 business-context.md 术语表已有条目，
          再补充本 PRD 新增的术语。
          专业术语首次出现时可在正文用括号标注英文）

三、客户旅程
    3.X [功能名]客户旅程
        3.X.1 可交互原型       ← 独立 HTML 文件链接，基于 PRD 客户旅程
                             章节自动生成的可交互页面原型。
                             文件命名：Interactive-[功能简称]-v[版本号].html
                             在 PRD 此处以链接形式引用：
                             「可交互原型：▶ 点击查看 Interactive-xxx-v1.0.html」
        3.X.2 泳道图       ← Mermaid 流程图，按角色分泳道
                             展示客户操作完整链路
        3.X.3 关键环节详解  ← 逐步骤描述，采用页面原型+功能描述
                             双列布局：
                             左列：自动生成的页面 Mockup
                             （HTML/CSS 模拟页面 UI 组件），
                             右列：链路、前置条件、展示内容、
                             数据来源、特殊情况。
                             用户如额外提供 Figma 截图，
                             优先使用截图替换 Mockup）

四、Use Case
    4.X [场景名]用例        ← 用例表格或场景描述
     （包含：用例名、触发条件、参与角色、前置条件、
      操作步骤、预期结果、异常分支）

═══════════════════════════════════════
  第二部分：产品实施细节
═══════════════════════════════════════

五、版本说明与改造范围
    5.1 版本信息            ← 表格：版本/日期/负责人/变更说明
    5.2 涉及改造模块        ← 表格：模块/改造内容
    5.3 改造前后功能对比    ← 对比表格：功能点/改造前/改造后

六、功能详解
    6.X [子功能]业务流程
        6.X.1 系统交互流程  ← Mermaid sequenceDiagram
                             （含颜色分区标注不同阶段）
        6.X.2 资金流        ← Mermaid sequenceDiagram
                             （含颜色分区标注资金流/信息流）
        6.X.3 功能改造点    ← 表格：模块/改造点/说明
        6.X.4 字段定义      ← 表格：字段/是否必填/数据来源
        6.X.5 异常处理      ← 表格：异常场景/处理链路
        6.X.6 状态机        ← 表格：对客状态/系统状态/说明

    （按需包含：风控策略、计费规则、邮件通知、审批功能、

    （状态映射表、表单字段明细、分类枚举、邮件模板等补充材料）
```

## 工作流程

### Step 1: 明确需求范围

在开始编写前，确认以下信息：
- 功能名称和所属业务域（Checkout/充值/提现/审批/账户/风控等）
- 涉及哪些系统模块（Portal、OOMP、Payment、账务、风控、金融网关、会员）
- 是否涉及多牌照（TCSP/MSO）场景
- 是否有参考截图、原型图或现有文档
- **涉及 Portal 页面改造时，是否提供了实际源码路径/代码片段/页面截图？Mockup 和可交互原型需要基于实际页面结构产出，请提供相关代码或截图作为输入。如果未提供，必须主动询问用户获取，不得自行用通用模板代替。**

**信息不足时**：不要猜测或编造，主动向用户确认缺失的关键信息。可以用结构化提问（按模块/字段/流程分组）一次问清楚。**任何不确定的内容——无论是业务规则、状态定义、交互细节还是系统关系——都必须停下来提问，绝不允许自行展开。**

### Step 2: 输出文档大纲

正式撰写前，先输出文档大纲供用户确认：
- 列出所有一级/二级/三级标题
- 标注每个子节计划包含的内容类型（表格/流程图/状态机/字段定义）
- 标注将裁剪的章节及理由
- **第三章「客户旅程」必须拆分到 Step 级别**（如 3.1.3.1 Step1 发起请求、3.1.3.2 Step2 确认提交等），**不得只写一行"三、客户旅程"就带过**。每个 Step 必须标注左列 Mockup 内容和右列描述要点

**目的**：确保整体结构符合预期，避免写完后大改。用户确认大纲后再进入撰写阶段。

### Step 3: 逐章撰写

基于用户提供的功能描述和上下文，按确认后的大纲逐章编写。注意事项：
- 如用户提供具体截图或配置说明，逐条审查并准确理解后再写
- 如功能涉及多个子流程（如充值+提现），每个子流程独立成节
- 表单字段定义必须包含：字段名、是否必填、数据来源/取值逻辑
- 状态流转必须覆盖全生命周期，包含终态（成功/失败/冲正）
- Mermaid 流程图使用 `rect rgb(...)` 标注不同阶段色彩分区
- **第二章核心概念的名词解释**：本章末尾必须包含「名词解释」表格，列出本 PRD 涉及的所有专业术语。先引用 `business-context.md` 术语表已有条目，再补充本 PRD 新增的术语。格式为两列表格：术语名 | 解释。术语在正文首次出现时可括号标注英文。

#### ⚠️ 第三章「客户旅程」撰写规范（红线级别，必须严格遵守）

**撰写顺序**：进入第三章后，必须严格按以下顺序依次输出每个小节，**不得跳过任何一节**：

**第一步：输出 3.X.1 可交互原型链接**
- 必须在第三章开头输出一个蓝色卡片样式链接：「可交互原型：▶ 点击查看 Interactive-[功能简称]-v[版本号].html」
- 同时生成对应的 `Interactive-[功能简称]-v[版本号].html` 文件（详见下方"可交互原型要求"）

**第二步：输出 3.X.2 泳道图**
- 使用 Mermaid sequenceDiagram 按角色分泳道（客户、Portal前端、Portal后端、OOMP、渠道、风控等）
- 展示从用户发起操作到最终完成的完整链路
- 必须使用 `<pre class="mermaid">` 渲染为图形，禁止代码块
- 必须使用 `rect rgb(...)` 标注颜色分区

**第三步：输出 3.X.3 关键环节详解**
- 将完整操作链路拆分为 **≥3 个 Step**（如：Step1 发起请求 → Step2 确认提交 → Step3 处理结果）
- 每个 Step **必须**采用双列布局：
  - **左列**：HTML/CSS 页面 Mockup（卡片/表单/按钮/列表/状态标签等），使用 Obita Portal 品牌色系，填充模拟数据。禁止纯文字、禁止占位符（如"此处为UI图"）
  - **右列**：功能描述表格，包含：链路（API/事件）、前置条件、展示内容（页面显示什么）、数据来源（数据从哪来）、特殊情况（异常/边界）
- **每个 Step 都必须有左列 Mockup + 右列描述，不允许只有文字没有 Mockup**

**可交互原型要求**（与 3.X.1 配套产出）：
- 基于第三章客户旅程内容，生成独立的 `Interactive-[功能简称]-v[版本号].html`
- 模拟完整用户操作路径，支持页面跳转、按钮交互、表单填写、步骤切换
- 严格遵循 Obita Portal 视觉风格（品牌蓝 #0072F7、组件尺寸、布局结构、状态色、字体等，详见 `references/portal-style-guide.md`）

**Mockup 和原型的 Portal 风格要求**：
- 所有 Mockup 和可交互原型的所有页面必须严格遵循 Obita Portal 视觉设计规范（详见 `references/portal-style-guide.md`）
- 核心要素：品牌蓝色系(#0072F7)、输入框 border-blue-200(#BFDBFE)、卡片 border-blue-100(#DBEAFE)、表头背景 #F7FAFF、按钮 36px/6px圆角、输入框 36px/8px圆角、卡片 12px圆角、布局 256px侧边栏+64px顶栏、状态色(蓝/amber/红/灰)、字体 system-ui 14px

**长文档处理**：当文档超过 2000 行时，分批输出，每批完成后暂停等待用户审查再继续。输出过程中在文末标记「继续」以提示用户指示继续。

### Step 4: 质量自检

每章完成后进行自检，全部章节完成后做整体检查。自检清单见下方「质量自检清单」章节。

### Step 5: 交付

#### 输出文件

PRD 交付时产出 **两个文件**：

| 文件 | 说明 |
|------|------|
| `PRD-[功能简称]-v[版本号].html` | PRD 正文文档（独立 HTML，浏览器直接预览） |
| `Interactive-[功能简称]-v[版本号].html` | 可交互原型文件（基于客户旅程生成的可交互 HTML） |

可交互原型在 PRD 第三章「客户旅程」开头以链接形式引用。

#### 样式要求

默认输出为独立 HTML 文件（支持浏览器直接预览），样式要求：
- 清晰的文档标题和版本信息
- 表格使用交替行底色增强可读性
- 流程图使用 Mermaid.js 直接渲染为图形（引入 CDN，用 `<pre class="mermaid">` 标签），不要输出代码块
- 支持打印为 PDF

如用户要求，可同时输出 Word (.docx) 格式。

#### 文件命名规范

```
PRD 文档：PRD-[功能简称]-v[版本号].html
可交互原型：Interactive-[功能简称]-v[版本号].html
```

示例：
- `PRD-法币充值提现-v2.2.html`
- `Interactive-法币充值提现-v2.2.html`
- `PRD-审批规则功能-v1.0.html`
- `Interactive-审批规则功能-v1.0.html`
- `PRD-OOMP法币异常订单-v3.1.html`

命名原则：
- 功能简称用中文，简洁准确（不超过 15 字）
- 版本号与 PRD 文档内「版本信息」表保持一致
- 不使用日期后缀（版本号已包含演进信息）

### Step 6: 知识回流（自动触发）

**触发时机**：Step 5 交付完成后自动执行。PRD 撰写完成或迭代修改结束后的每一轮，都应检查是否有新知识需要沉淀。即使无新增通用知识，也必须向用户显式说明「已检查，本次无新增通用知识需回流」，不得静默跳过。

PRD 交付完成后，检查本次需求中是否有新的**平台级通用知识**需要沉淀到 skill 知识库中。目的：使 `business-context.md` 和 `prd-sample.md` 随着业务积累持续丰富，后续 PRD 撰写时自动受益。

**需回流的知识类型**：

| 类型 | 判断标准 | 更新位置 |
|------|---------|---------|
| 新业务规则 | 本次 PRD 中定义的通用规则（如审批优先级、路由逻辑） | `business-context.md` |
| 新系统模块 | 首次涉及的系统模块及其职责描述 | `business-context.md` → 核心产品模块表 |
| 新状态机 | 首次定义的订单/流程状态流转（对客状态 + 系统状态） | `business-context.md` |
| 新渠道信息 | 首次对接的渠道能力、开户方式、webhook 类型 | `business-context.md` → 渠道信息 |
| 新术语 | 首次出现且后续会复用的业务术语 | `business-context.md` → 术语表 |
| 新写作格式 | 本 PRD 中使用了 prd-sample.md 未覆盖的写作格式 | `prd-sample.md` |

**回流原则**：

- 只回流**平台级通用知识**，不回流单次功能特有的内容（如某个特定页面的交互细节）
- 回流前需**用户确认**，避免将未定稿的临时决策写入知识库
- 更新时保持原有结构不变，在对应章节追加或修改，不重写整份文件
- 如果本次 PRD 无新增通用知识，也必须向用户显式说明「已检查，本次无新增通用知识需回流」，**不得静默跳过**

**操作方式**：

1. 回顾本次 PRD 内容，列出候选回流项
2. 逐项向用户确认：「以下业务规则是否需要沉淀到 skill 知识库？」
3. 用户确认后，更新对应 reference 文件
4. 在回复中告知用户更新了哪些内容

## 迭代修正流程

PRD 编写过程中常需多轮修正，遵循以下流程：

### 用户反馈处理

1. **截图/行号对照**：用户提供截图或引用具体行号时，逐条对比原文与截图差异，定位具体不一致处
2. **精准修改**：只改用户指出的部分，不擅自扩展修改范围
3. **修改标记**：每轮修正后在回复中简要列出改了什么、改了哪里
4. **确认闭环**：修改完成后请用户确认，避免遗留未处理的问题

### 常见修正场景

| 场景 | 处理方式 |
|------|----------|
| 用户说"你理解错了" | 回到原始输入重新理解，不要基于错误理解继续写 |
| 用户追加新信息/新规则 | 将新信息融入对应章节，必要时调整章节结构 |
| 用户要求调整排版格式 | 仅调整格式，不动内容逻辑 |
| 用户指出遗漏 | 补充对应章节，检查是否影响其他关联部分 |
| 截图与输出不一致 | 逐字段对比，修正偏差，输出差异说明 |

### 修正原则

- **不改未提及的内容**：用户只说"修改第三章 3.2"，就不要动第四章
- **不假设用户意图**：遇到模糊反馈，先问清楚再动手
- **保留修改历史**：多轮修正中保持版本号递增，版本信息表记录每次变更
