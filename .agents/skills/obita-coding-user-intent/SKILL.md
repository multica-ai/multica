你正在为一个已有代码库进行需求分析。
不要开始实现、不要编辑文件、不要编写代码，不要提出具体实现方案。
需求澄清阶段默认只基于 docs/system-design、docs/requirements 等需求/系统设计文档判断变更范围；除非用户明确说“读代码确认一下”“看代码确定范围”“根据代码现状判断”等，否则禁止扫描或读取业务代码来做需求范围澄清。

你的任务是：
1. 加载相关项目文档。
2. 澄清用户的修改意图。
3. 识别受影响的技术范围。
4. 产出一份聚焦当前变更的最小需求确认文档。

## docs 系统设计文档目录结构说明
- 优先读取 `docs/README.html` 理解 `docs` 目录结构，再聚焦 `docs/system-design/`
- 如果 `docs/README.html` 不存在，不要报错中断；直接扫描 `docs/system-design/` 的一级与二级目录，并按最小范围原则选择 1-3 个最相关文件

## 核心规则
- 一次只问一个问题。
- 不要直接开始实现。
- 用户已经回答后，不要重复问同一个澄清问题。

## 防重复澄清约束（最小改动）

对于多选澄清（A/B/C/...）：
- 如果用户已经选定一个选项，将其视为已确认选择。
- 除非需求范围发生变化，否则不要再次询问同一组选项。
- 基于该选择继续当前澄清流程，并完成后续需求总结与确认，不要因为用户选择了一个选项就提前结束本技能。

当用户输入是一句简短的后续修改指令时：
- 如果该指令对目标范围进行了纠正、收窄或改写，应以最新用户回答为准，覆盖之前的模糊意图锚点，并重新判定技术范围。
- 如果旧的候选方向与最新回答冲突，应停止沿用旧方向，不要继续围绕旧关键词扩展文档或接口搜索。
- 如果最新回答改变了目标范围，应基于新目标重新执行一次最小范围的关键词提取与文档发现，并重新选择最相关的 1-3 个文档。
- 可以利用当前上下文缩小候选范围，但在仍有歧义时，不要直接锁定目标。
- 如有必要，基于文档依据或一次聚焦确认来最终确定目标。
- 如果用户已明确回答选项或具体对象，则停止重复追问并继续推进。

## 阶段 1 — 项目上下文发现

在提问前，先检查项目文档。默认只做文档发现，不做代码发现。

先扫描以下目录结构：

```text
docs/system-design/
```

如果用户请求看起来与某个具体领域相关，按以下流程进行发现：

1. 从对话上下文中提取 3-8 个意图关键词。
包含模块名、功能名、接口路径片段、页面名、领域术语和用户可见文案。
2. 在 `docs/system-design` 中使用字面量关键词匹配搜索。
除非你明确进行了转义，否则不要把用户输入当作正则表达式。

```bash
# 示例：按字面量搜索每个关键词
rg -n -F -e "agent" -e "task execution" -e "/api/skills" docs/system-design
```

3. 如果 `docs/system-design` 中的结果不足，仅在确认与同一功能明确相关时，扩展到关联文档目录。
允许扩展的目录包括：
- `docs/superpowers/specs/`
- `docs/superpowers/plans/`
- `docs/system-design/frontend/interface/`
- `docs/system-design` 中已发现文件明确引用的其他产品或设计文档
不要仅因为关键词碰巧匹配，就扩展到无关目录。

4. 按以下优先级选择匹配结果：
- 模块路径精确匹配（例如 `docs/system-design/frontend/model-experience/`）
- 设计概览文件（`design.md`、`README.md`、`README.html`）
- 界面或 API 文档
- 被直接引用的补充规格或计划文档

5. 只加载最相关的 1-3 个文件进入上下文。
除非用户明确要求进行大范围文档扫描，否则不要加载整个目录。

7. 文档发现结果只作为内部判断依据，不默认展示给用户；只有当存在阻塞性的缺口、过期文档或冲突文档时，才在需求确认中的“影响面分析”里用一句话说明。

## 代码读取边界

- 默认禁止在需求澄清阶段读取、搜索或总结业务代码、测试代码、配置代码。
- 只有用户明确要求通过代码确认范围时，才允许进行最小范围代码发现。
- 即使允许读取代码，也只用于验证变更范围，不在需求确认输出里展示代码阅读过程、文件清单或实现细节。
- 进入方案设计或执行阶段后，才按后续 skill 流程读取和修改代码。

## 阶段 2 — 技术约束（仅在阶段 1 完成后进行）

使用以下三段模板精确总结需求，禁止输出“已加载文件列表”“项目文档背景”“相关模块清单”等模型阅读过程信息。
只表达和用户当前变更直接相关的内容；任何不适用的部分直接省略，不填写 `N/A`。

### 需求澄清
- 用 2-5 条说明用户要实现的业务变化。
- 必须说明触发条件、目标行为、明确不做的边界。

### 变更范围
- 用 2-6 条列出需要修改的代码/测试/文档范围。
- 只写与本次变更直接相关的模块或入口；不要罗列扫描到但不需要修改的模块。

### 影响面分析
- 用 2-6 条说明 API、数据、权限、外部集成、补偿/幂等、测试的影响。
- 只写存在影响或需要确认的点；无影响的类别不要展开。
- 若存在阻塞性未知点，用“待确认”单独列出；没有阻塞性未知点则不输出该项。

## 阶段 3 — 归纳总结（仅在所有问题回答完成后进行）

1. 使用已收集的需求与项目文档填写每一部分。
2. 只展示“需求澄清 / 变更范围 / 影响面分析”三段内容。
3. 询问：“这样是否准确表达了你的需求？”
4. 在用户确认前继续迭代。

## 升级边界

如果不确定性是局部且低风险的，不要重新升级为宽泛的需求访谈。
只有当用户意图在领域上发生明显变化时，才重新打开更广范围的澄清。
